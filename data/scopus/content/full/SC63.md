#### **Abstract** 

Automated code generation from natural language is a growing challenge in software engineering, enabling the direct translation of requirements into executable code. Machine learning-based automatic code synthesis emerged as a key solution, with transformer-based models playing a pivotal role in enhancing this process. This paper addresses the challenges of generating accurate, functional code efficiently. This paper presents a transformer-based T5 model fine-tuned on the multilingual XLCoST dataset for code generation. The model is evaluated using a novel systematic multi-level evaluation framework. The framework evaluates the model’s across three distinct task complexity levels ranging from beginner to advanced. The evaluation results highlight the model’s versatility and robustness. Furthermore, the model achieves a BLEU score of 30.093, demonstrating competitive performance compared to open-source baselines. The model’s code generation abilities accelerates software development by reducing manual coding efforts, thereby improving workflow efficiency and software quality. These results confirm that transformer-based models are effective tools for automating software development, with promising implications for the future.

### **2.1 Code generation tools, architectures and datasets** 

Code generation has long been a technique in embedded systems and compiler design for problem-solving [14]. The desire for tools that can increase programmers productivity has steadily risen. This field is often viewed as automatic programming and is relied upon for generating problem solutions and text completion [2]. Its has widely spread and regarded as one of the most popular applications of artificial intelligence. It has shown great potential for improving automated tools and aid software development processes [15]. 

Code generation offers numerous benefits. it reduces developers’ effort and time, boosts productivity of teams and individuals, converts natural language into machine code, generates tests for existing code [16], and even evaluates code generation processes [17]. It also checks for syntactic correctness and generates code based on previous models [18]. 

Transformers are deep-learning models that are used to understand natural language and generate human language or code by using different sorts of data that have to do with the generative task [19]. Transformers have become the core of modern NLP due to their ability to learn and comprehend all sorts of text data, and empower tasks such as machine translation, text generation, sentimental analysis, code generation, and code summarization. A lot of pre-trained transformers have enhanced performance on diverse NLP benchmarks and applications, making transformers the state of the art in NLP [19]. Transformers are made to function similarly to the typical NLP machine-learning model pipeline, which consists of data processing, model application, and prediction [19]. 

Transformers compute attention weights among input tokens across multiple feed-forward layers, enabling the extraction of hierarchical features. Each layer refines the representation from the previous ones. Unlike recurrent neural networks (RNNs), transformers process tokens in parallel rather than sequentially. For example, BERT, GPT3, and T5 are state-of-the-art transformers for aiding NLP tasks. BERT stands for Bidirectional Encoder Representations from Transformers, is a transformer-based model that captures bidirectional contextual information from input text by unsupervised pre-training on large textual datasets.

### **2.2 Large Language Models (LLMs) in code generation** 

Large Language Models (LLMs) are a class of a neural network model that is made to process and produce natural human text from large datasets. LLMs, such as GPT [29], are trained on large text corpora of data which allows them to do a variety of tasks with high fluency, including translation, summarization, and question answering. Built on transformer architecture, LLMs demonstrated enhanced capabilities in automated code synthesis, enabling applications such as code completion, bug detection, and documentation generation. 

Recent years have seen rapid advances in code generation driven by LLMs and transformer-based architectures. A number of open-source and commercial models were introduced, including StarCoder [30], InCoder [31], CodeLlama [32], DeepSeek [33], Claude [34], and proprietary models such as Codex [35] and ChatGPT [36]. These models have demonstrated strong performance on a variety of code synthesis and code completion tasks, often surpassing previous baselines in both accuracy and code quality.

### **2.3 Evaluation metrics** 

Many evaluation metrics exit in the literature, with the most widely accepted across most of the text generation works being metric called BLEU [39]. BLEU, short for Bilingual Evaluation Understudy Score, is a metric used in machine learning specifically generative fields to evaluate the quality of the generated text. BLEU scores assess n-gram similarity between generated and reference code. Other evaluation approaches include CodeBLEU, and human evaluations. CodeBLEU incorporates additional code-related features such as syntactic and semantic correctness [40]. 

Another notable metric is CodeScore, an LLM-based code evaluation metric for code generation, proposed in [41]. CodeScore uses generated code concatenated with 



reference code, generated code concatenated with NLP, and generated code concatenated with both for its evaluation. 

### **2.4 Challenges and limitations in existing approaches** 

Automated code generation faces several challenges. One primary challenge is the lack of contextual understanding, where models generate code that is syntactically correct but does not align with user intent. Additionally, handling longrange dependencies in complex programming logic remains an open research problem. Security concerns, such as biased training data and vulnerabilities in generated code, also pose risks, that require further investigation [35]. 

As for the LLMs, most state-of-the-art models require significant computational resources for training and inference, and many are not fully open or accessible due to API or licensing restrictions. This motivates the need for practical, reproducible baselines that can be deployed in resourceconstrained environments and do not rely on proprietary APIs. While models like Codex and ChatGPT achieve higher absolute scores on benchmarks such as BLEU, CodeBLEU, and Exact Match, the proposed approach provides a baseline for the community, with all training and evaluation details available for reproducibility.

### **3.1 Model architecture** 

Transformers improves NLP and code generation by introducing an architecture capable of understanding and processing complex natural language. This architecture improved the ability to understand input and output sequences. Furthermore, pre-training on massive amounts of text enhanced text comprehension. Therefore, choosing the right transformer is an important step for the task at hand (text-to-code generation). 

The design and development of the proposed model were guided by the transformer architecture and its robust functionalities. Using pre-trained transformers is pivotal to working with text-to-code tasks. Furthermore, transformers enables the fine-tuning of pre-trained models on chosen datasets, adapting them to the goal of aiding software engineering process. The fine-tuning process improved the proposed model’s ability to achieve high-performance results in text generation tasks, bridging the gap between text and executable code. 

The selected transformer for the proposed model is T5. T5 stands for Text-To-Text Transfer Transformer, a model that converts all NLP tasks into a text-to-text format. T5 excels at translation and, when used with code generation datasets, simplifies the fine-tuning process [43]. Therefore, T5 was employed as the transformer for the proposed model due to its adaptability and versatility. Availability in various sizes also facilitates efficient resource management; T5 large was chosen for implementation. 

Transformer tokenizers play a crucial role in converting raw text into a numerical format that transformer models can process. Generally, these tokenizers break down words into smaller units to efficiently handle vocabulary size and rare words. This subword segmentation is essential for preserving semantic meaning while accommodating out-ofvocabulary terms. Specifically, the T5 transformer tokenizer was trained on a vast corpus of text to adeptly segment language into consistent and meaningful subword tokens. This approach enables T5 to handle diverse inputs, including text-to-code generation task.

For the proposed model, the chosen text-to-code dataset have three splits: training, validation, and testing for both program and snippet levels. Each entry in the dataset contains two main attributes: text of natural language input and code of target programming code. 

Step 1: Dataset Analysis. The dataset structure was analyzed to understand its attributes and distribution. 

Step 2: Prefix Addition. Prefixes (e.g.,"Generate Python:","Generate Java:") were added to the input text. The prefixes improve the accuracy of the generated code and guide the model to generate code in the specified language. 

Step 3: Dataset Combination. The datasets for Python and Java were combined into two different sets: Python and Java program level dataset, and Python and Java snippet level, including the training, validation, and testing sets. The combined datasets were then shuffled to ensure randomness in the training data distribution. This approach aimed to increase training effectiveness as both levels were trained separately, and addresses issues such as mismatched maximum length across sets. This way, the training progress was more efficient, accurate, and reduced training errors, compared to training both sets at the same time. 

Step 4: Common token addition and Resizing. This step aims to manage and reduce the input token lengths by identifying frequently used phrases or patterns in input text and queries, shortening them, and replacing them with predefined tokens. These additional tokens were added to the tokenizer vocabulary. This makes the overall token length shorter, making the input easier for the model to handle, hence, improving the efficiency of the model. 

Step 5: Tokenization and Length Management. The combined dataset was tokenized using the T5 model pre-trained tokenizer. An important aspect of this step is length management. For program-level tasks, input length was set to 500 and output length to 300. For snippet-level, input length 

was set to 50 and output length to 100. Input and output sequences were checked to ensure they do not exceed model limits. This was essential for ensuring high performance and efficiency. For transformers like T5, constraints exist on the maximum token number it can process, which is 512 for many pre-trained models. Therefor, preference was given to inputs and generated code below this threshold. For that, a check on the length was done to obtain the appropriate tokens length. 

Step 6: Dataset Ready for Training. After all these steps, the preprocessed dataset is ready for use in the training pipeline. This produced two tokenized sets of data: programlevel, and snippet-level. 

A final and conditionally applied step is post-processing based on the generated language. In this step, the generated code is formatted for proper display and to ensure it conforms to the target programming language’s syntax. For example, Python is sensitive to code indention; if indention is not proper, the generated code can result in syntax errors. This step automatically modifies the generated code indentation by parsing the code, identifying blocks such as loops, and applying corrected indentation. 

### **3.3 Model training and fine-tuning** 

This section describes the training process for the model, including hyperparameter configuration, tokenization, and performance metrics. The model was trained on the prepared dataset, and its hyperparameters were fine-tuned to achieve optimal performance. 

The model was trained and evaluated on the XLCoST dataset, which contains parallel data in Python and Java at both the program and snippet levels. The dataset was split into training, validation, and test sets using an 88%/4%/8% ratio as provided by creators. The model was fine-tuned on the following subsets:"Python-program-level","Pythonsnippet-level","Java-program-level", and"Java-snippetlevel". Training and fine-tuning were performed using the selected CodeT5 model via the HuggingFace Trainer API. The process was configured for multi-GPU training, using per-device batch size and gradient accumulation.

Although CrystalBLEU [54] or Exact Match metrics are not reported due to the lack of standardized outputs for all models, their importance for a comprehensive evaluation is recognize. Future work will aim to include these metrics, either by leveraging available outputs or by aligning evaluation with published benchmarks.

## **6 Threats to validity** 

Threats to validity in this research can be categorized as internal, external, and construct validity threats. An internal threat is the availability and quality of training data for the model. Despite using a diverse and versatile dataset, a major obstacle remains the limited availability of natural language specification statements and annotated code. This limitation could introduce biases into the training process, which could have impact the applicability of the model to real-world software development scenarios. 

An external threat concerns computational resources required for model training and optimization, specifically CPU, GPU, and RAM. Training machine learning models, especially those based on transformer architectures, requires significant processing power and memory. The availability of these resources may have limited the scope of the study and the results achieved. 

Additionally, a construct validity threat lies in the selection of evaluation metrics. While metrics like BLEU score offer valuable insights into model performance, they may not fully account for other factors influencing the functionality and quality of the generated code. Furthermore, due to resource and API constraints, direct benchmarking against large proprietary state-of-the-art models such as Codex and ChatGPT was not feasible.
