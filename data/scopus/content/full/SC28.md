#### **1. Introduction** 

Artificial Intelligence (AI) has been increasingly applied in software engineering tasks, which help achieve great advancements in various domains. Among these achievements, Large Language Models (LLMs) have played a crucial role and demonstrated strong capabilities in code generation (Liu et al., 2024b), automated testing (Fakhoury et al., 2024; Siddiq et al., 2024), bug fixing (Kang et al., 2023), and software refactoring (Cordeiro et al., 2024; Khajezade et al., 2024; Liu et al., 2024a). In particular, code clone detection has 

been a popular area of exploration for Large Language Models (LLMs) (Almatrafi et al., 2025; Dou et al., 2023; Zhang and Saber, 2024), and refactoring has also recently received attention (AlOmar et al., 2024c; Batole et al., 2025; Cordeiro et al., 2024; Khajezade et al., 2024; Liu et al., 2024a). Code clones, which copy and reuse pieces of code, are common practices in software development. While it can boost productivity in some cases, too much cloning can make the code harder to maintain and increase the risk of spreading bugs (Hummel et al., 2010). Code refactoring is the practice of improving the internal structure of code without changing its external behavior (Fowler, 2018). The exam-

The rapid developments in AI and LLMs have offered great opportunities to improve this process. As LLMs continue to evolve, their capability to do code clone detection and refactoring has been an interesting area to explore. Recent studies have explored LLMs on code clone detection. For example, Zhang and Saber (Zhang and Saber, 2024) compared GPT-3.5 and GPT-4 for clone detection and found that GPT-4 outperformed GPT-3.5, especially in detecting more complex clones. Similarly, Khajezade et al. (2024) investigated the capability of ChatGPT and fine-tuned models, such as CodeBERT, RoBERTa in detecting mono-lingual and cross-lingual clones. Their results show that ChatGPT performed similarly with fine-tuned models in monolingual clone detection, but it surpassed them in cross-lingual clone detection. Additionally, Moumoula et al. (2024) explored the comparison between LLMs and embedding-based approaches. Their results demonstrated that LLMs can perform well on simple clones, but the performance dropped significantly when dealing with complex cases. While these studies have provided insights into LLMs’ effectiveness in code clone detection, they mainly focused on clone detection rather than clone type classification and refactoring. Moreover, the performances of LLMs on code clone refactoring across different scenarios, such as clones in the same file, clones across different files remain unexplored. 

Traditional tools have been widely used for clone detection (Roy et al., 2009). These tools utilized text-based (Ducasse et al., 1999), token-based (Basit and Jarzabek, 2007) and abstract syntax tree (Baxter et al., 1998). As for refactoring, tools like JDeodorant (Tsantalis et al., 2015; Tsantalis and Chatzigeorgiou, 2011; Tsantalis et al., 2018) and WitchDoctor (Foster et al., 2012) are also well-known among developers. Among different refactoring types, extract method refactoring regularly appears in the top five most frequently used in real-world scenarios (Murphy-Hill et al., 2011; Negara et al., 2013; Tsantalis et al., 2020). Current tools follow software quality metrics (Charalampidou et al., 2016; Maruyama, 2001; Tiwari and Joshi, 2022; Tsantalis and Chatzigeorgiou, 2011; Yang et al., 2009), but their suggestions often do not match real-world Extract Method practices. That is because effective refactoring needs both technical precision and human judgment. Developers draw not only on principles but also on their sense of what fits the code context-something most tools still struggle with. This gap likely contributes to developers’ hesitation to adopt automated refactoring (Pomian et al., 2024). 

On the other hand, LLMs like GPT-4o (Hurst et al., 2024) have demonstrated reasoning abilities to solve complex programming tasks (Wadhwa et al., 2024; Wu et al., 2024; Ye et al., 2025). But their ability to generate compilable and correct refactored code for clones has not been evaluated. This study aims to address these limitations by evaluating GPT-4o’s abilities to detect and refactor clones. We also compared seven contemporary LLMs with traditional tools JDeodorant and AntiCopyPaster to evaluate whether LLMs can demonstrate consistent capabilities. 

In our study, we studied if LLMs can correctly classify the clone type and apply refactoring. Since we used the output of JDeodorant as our ground truth and the refactoring as our baseline, our work primarily targeted Type-1 (Identical except for whitespace, comments, layout), Type-2 (Identical except for variable names, literals, types), and Type-3 (Mostly similar but with some different statements) clones, which are consistent with the types of clones that JDeodorant analyzes. We proposed an approach to analyze four distinct scenarios: (1) whole method clones within the same file, (2) whole method clones across different files, (3) code fragment clones within the same file, and (4) code fragment clones across different files. We applied prompt engineering techniques, including zero-shot, one-shot and few-shot learning to assess how they impact code clone classification accuracy. Furthermore, we manually evaluated the correctness and quality of refactored code based on compilation success, functionality preservation (via test cases) and 

use of lambda expression. We also designed a case study based on JHotDraw project. We tested JDeodorant, AntiCopyPaster and 7 LLMs, including Gemini 2.5 Pro, GPT-3.5, GPT-4o, GPT-5, DeepSeek-R1, Claude Sonnet 3.7 and Claude Opus 4.1. We then evaluated their performances based on clone detection and correct “Extract Method” refactoring. Our dataset can be found in the replication package (Qian and AlOmar, 2025). 

#### **2. Related work** 

In LLM-based code clone detection, Zhang and Saber (Zhang and Saber, 2024) investigated the capability of GPT-3.5 and GPT-4. They used BigCloneBench (human-made) and GPTCloneBench (LLMgenerated) datasets, employed few-shot prompting technique to enhance accuracy. Their results show that GPT-4 outperforms GPT-3.5 in code clone detection, especially in more complex clones. Khajezade et al. (2024) compared ChatGPT (GPT-3.5-turbo) with fine-tuned models, such as CodeBERT, RoBERTa, and GraphCodeBERT for mono-lingual (Java-Java) and cross-lingual (Java-Ruby) clone detection. They found that while ChatGPT achieved similar performances with fine-tuned models in mono-lingual settings, it outperforms them in cross-lingual settings. Due to the use of multiple programming languages, cross-lingual code clone detection has been essential. Moumoula et al. (2024) studied four LLMs, including Falcon-7B, LLAMA2-Chat-7B, StarChat-Beta and GPT-3.5-turbo, with eight prompts, comparing with an embeddingbased approach leveraging OpenAI’s Text-Embedding-Ada-002 model for detecting cross-lingual code clones. Their findings indicate that GPT-3.5-turbo achieved the best result on simple cases, but its performance drops significantly when it comes to complex samples. The embedding-based approach outperforms LLMs on the XLCoST and CodeNet datasets. Nashaat et al. (2025) analyzed transformer-based approaches to code clone detection. They proposed CloneXformer which leverages language-agnostic features and token-based representation. Their results show that CloneXformer outperformed state-of-the-art techniques by 16.88 % F1-score on BigCloneBench. While LLMs have great potential in code clone detection, their performance is poor for Type-4 clones. To overcome this, researchers have proposed finetuning approaches. Inoue and Higo (Inoue and Higo, 2024) executed fine-tuning on LLMs (GPT-3.5-turbo, Llama2-Chat-7B, CodeLlama-7BInstruct) using FEMPDataset. Their results show that fine-tuning greatly improved the precision and overall accuracy. Notably, fine-tuned GPT3.5-turbo outperformed GPT-4o and CodeLlama-7B-Instruct outperformed the base version. Xu et al. (2025) introduced MANTRA, a comprehensive agent-based framework designed to automate method-level refactoring tasks by integrating Context-Aware Retrieval-Augmented Generation (RAG), coordinated multi-agent collaboration, and verbal reinforcement learning. MANTRA achieved 82.8 % success rate in producing code that compiles and passes tests, compared to an 8.7 % success rate by the baseline RawGPT model. 

All of the above-mentioned studies have focused on evaluating the capability of LLMs on code clone detection or code generation and refactoring. Our work is different from these studies as our main purpose is to measure the performance of LLMs on not only the clone type classification, but also the refactoring. We leveraged the results of JDeodorant (Tsantalis et al., 2015), an Eclipse plug-in which can help detect code clones and refactor by method extraction, as our datasets. Although JFreeChart was not originally designed as a benchmark dataset, in our study it serves as a benchmark, since its real-world clone instances and comprehensive test suite provide a consistent and reproducible basis for evaluating clone classification and refactoring across both LLMs and traditional tools. We compare GPT-4o with JDeodorant. We also evaluated different prompting techniques such as zero-shot, one-shot and few-shot learning. For code clones and refactoring, we test four scenarios, whole method clones in the same file, whole method clones in different files, code fragment clones in the same file and code fragment clones in different files. Our goal is to study if the format and different positions of

#### **3. Study design** 

#### _3.1. Research questions_ 

- **RQ1.** _How well can LLMs classify code clones?_ 

Code clone detection and clone type classification can help developers identify redundant code. As LLMs offer a new approach, it is still unclear how well LLMs can distinguish between different clone types (Type-1, Type-2, Type-3). This research question aims to evaluate the performance of LLMs in classifying the types of code clones. 

- **RQ2.** _How well can LLMs refactor code clones?_ 

LLMs have the potential to perform refactoring by understanding code context and structure. This research question investigates the capability of LLMs to correctly refactor clones, as it remains uncertain how effective can LLMs do the refactoring. 

- **RQ3.** _How consistent are LLMs’ clone identification and refactoring capabilities across different models compared to traditional approaches?_ 

LLMs may perform differently when used for clone detection and refactoring, and it is not clear how stable their results are across models. This research question examines whether LLMs consistently yield reliable outcomes or if their performance varies significantly depending on the model. By comparing them with traditional tools, we can see where LLMs are reliable and where they may still fall short. 

#### _3.2. Open source project selection_ 

To select open-source projects for our study, we considered nine projects previously studied by Tsantalis et al. (2015). These projects are Apache Ant 1.7.0, Columba 1.4, EMF 2.4.1, JMeter 2.3.2, JEdit 4.2, JFreeChart 1.0.10, JRuby 1.4.0, Hibernate 3.3.2 and SQuirreL SQL 3.0.3. To fully understand each project and find the best projects suitable for our experiments, we conducted a detailed analysis of each project. We extracted clones from the reports of JDeodorant (Tsantalis et al., 2015) to calculate the number of clones in different projects. Some of the reports contain multiple clones, which we initially mined as separate instances. However, we removed any clones located within the Test folder because the false negatives (clones which are not detected) of clone detection tools occur more frequently in test code than in production code (Van Bladel and Demeyer, 2021). Additionally, we excluded clones that were not located in the source folder. As outlined in our research questions, we aim to assess the capability of LLMs in performing code clone refactoring, test files are required to do the evaluation, so we excluded EMF 2.4.1 and JEdit 4.2 which do not have test files. In projects that include test files, we found that not all cloned methods were covered by tests. To address this, we used GPT-o3-mini-high,<sup>1</sup> which had the highest ELO score for code completion, to generate the missing test methods. Among the remaining projects, we wanted to select the one that would provide the most comprehensive dataset for our experiments. Due to JFreeChart1.0.10’s significantly larger number of code clones, we selected it as the primary dataset for experiments. Table 1 summarizes the number of clones identified across different scenarios. We manually reviewed each clone. Two authors conducted the 

data analysis. One manually inspected the code changes, while the other reviewed the results to reach an agreement. During the manual review, we also came across duplicate clone pairs where two pairs had identical source folders, packages, classes, methods, and start/end lines. We simply skipped these duplicates during later inspections. Since we utilized API to run LLMs, the cost for running multiple projects is much higher than single project. Thus, we selected JFreeChart for our experiment in which its cost was around $1,500. 

For our case study, we selected JHotDraw as our primary project. The reason is that JHotDraw is a mature and widely studied open-source Java framework that has been used in numerous software engineering studies on code clone (Kodhai and Kanmani, 2014; Mel et al., 2012; Chen et al., 2014; Tairas et al., 2006; Lin et al., 2014). We randomly selected 30 methods from the JHotDraw codebase and manually introduced clone instances. The number of selected methods was intentionally limited due to the manual effort required to validate the clone detection and refactoring results across all models. This process involved not only inserting realistic clones, but also thoroughly checking compilation outcomes and test case behaviors for each tool, which made scaling to a larger dataset impractical within the scope of this study. The introduced clone instances included Type-1 and Type-2. However, since Deckard was only capable of detecting Type-2(a) and Type-2(b) clones for Type-2, we limited the comparison among the tools to Type1, Type-2(a), and Type-2(b) clones. After filtering out cases, the total number of methods used in the final comparison was 20. We found that the test files did not fully cover the cloned methods, we utilized GPTo4-mini-high<sup>2</sup> to generate test methods because it achieves remarkable performance in coding tasks (OpenAI, 2024).

However, they lack the advanced reasoning capabilities required for clone type classification and clone refactoring. When providing natural language along with cloned code as input prompts to StarCoder, the model often generates unrelated code instead of indicating the type of provided code fragments as responses. #### _3.4. Traditional tool selection_ 

#### _3.6. Clone type classification_ 

In addition to JDeodorant, in our case study for RQ3, we selected AntiCopyPaster (AlOmar et al., 2022a, 2023, 2024a, 2025), another traditional clone refactoring tool, as a baseline. AntiCopyPaster is an IntelliJ-based plugin designed to automatically detect and refactor code clones in real time. It provides just-in-time “Extract Method” refactoring suggestions immediately after duplicate code is pasted, helping developers eliminate redundancy without interrupting their development workflow. The tool uses both semantic and syntactic code metrics to decide whether a code fragment should be extracted and supports different types of clones, including Type-1 and Type-2. 

We selected AntiCopyPaster as our traditional baseline because it represents a mature, IDE-integrated refactoring tool that combines clone detection and automated method extraction. Unlike LLMs, which rely on text-based reasoning and generation, AntiCopyPaster’s rule-based and PSI-based (Program Structure Interface) approach provides a stable and deterministic comparison point. This allows us to assess how consistently LLMs can achieve human-level clone detection and refactoring quality under a well-established traditional setting. 

#### _3.5. Prompt engineering_ 

Since Tsantalis et al. (2015) primarily focused on Type-1, Type-2, and Type-3 clones, our study also follows this approach, concentrating on these three clone types to ensure a consistent alignment with prior research. Tsantalis et al. (2015) utilized four clone detection tools, including CCFinder (Kamiya et al., 2002), Deckard (Jiang et al., 2007), CloneDR (Baxter et al., 1998) and NiCad (Roy and Cordy, 2008). They found that Deckard tends to detect more Type-2 and Type-3 refactorable clones, and also refactorable clones it found tend to have larger size. Among those clone detection tools, Deckard is the one that mostly reports complete Abstract Syntax Tree (AST) statements as clones. So in our study, we used Deckard clone results. 

Furthermore, we divided our dataset into four scenarios. Specifically, we considered two cases regarding code clone types: 

- Whole method clones - where the entire method is identied as a clone. 

- Code fragment clones - where only a portion of a method is a clone. 

We also considered two cases regarding the position of code clones: 

- Same class clones - clones are within the same file. 

Prompt engineering plays a crucial role in optimizing LLMs. As mentioned in Section 3.3, we applied zero-shot (Palatucci et al., 2009; Wu et al., 2019), one-shot (Fei-Fei et al., 2006) and few-shot learning (Fe-Fei et al., 2003) techniques for clone type classification to evaluate whether different prompting strategies would improve the performance.

Although our primary goal is to study Type-1, Type-2 and Type-3 clones, we still include the definition of Type-4 clones to evaluate whether the LLM can classify types correctly. Next, we include the method information in the prompt, such as method name and its start and end line number. Those details are provided in the previous study (Tsantalis et al., 2015). #### _3.7. Code clones correction_ 

Following the clone type classification step, we study the refactoring further. We store both the prompt and response from the classification phase, to ensure continuity. 

The refactoring prompts for whole method clones are provided in Listing 4. 

Since we use JDeodorant’s results as our ground truth, as shown in JDeodorant (Tsantalis, 2011), approximately 49.6 % of the total refactorings recommended are done by “Extract Method”, we use the same 

approach “Extract Method” to refactoring. “Extract Method” is one of the most important refactorings (AlOmar et al., 2024b). This refactoring was among the first to be formally defined in the field, as it offers a practical way to keep methods short, modular, and free from overlapping responsibilities (Griswold and Notkin, 1993). In addition, “Extract Method” often acts as a stepping stone for carrying out more complex refactorings (Zarras et al., 2015). Both open-source developers (Silva et al., 2016; Tsantalis et al., 2013; Murphy-Hill and Black, 2008; Charalampidou et al., 2018, 2016; Thy et al., 2023; Murphy-Hill et al., 2011; AlOmar et al., 2019) and industry practitioners (van der Leij et al., 2021) regard it as a key operation. Its popularity comes from its versatility: developers apply it for removing duplicated code (Yoshida et al., 2019; Alcocer et al., 2020; Hotta et al., 2010; Higo et al., 2008), extracting reusable functionality (Silva et al., 2016; AlOmar et al., 2022b, 2020), adapting outdated method signatures (Silva et al., 2016), simplifying large or complicated logic (Yang et al., 2009; Morales et al., 2017; Tiwari and Joshi, 2022; Khomh et al., 2012; Palomba et al., 2014, 2018; Oizumi et al., 2020), and improving code testability (Cinnéide et al., 2011; Harman, 2011). Because it supports such a wide range of scenarios, Extract Method is often described as the “Swiss Army knife” of refactoring (Hora and Robbes, 2020). #### _3.9. Statistical analysis_ 

We applied statistical analysis to study the models and tools performance to see if they vary. To rigorously assess whether the observed performance differences between models and tools are statistically significant, we employ McNemar’s test (Baxter et al., 1998; AlOmar et al., 2021), a non-parametric statistical test designed for paired nominal data. McNemar’s test is particularly well-suited for our analysis because it evaluates whether two classification methods (in our case, different models or tools) perform differently on the same set of test cases. The test focuses on cases where the two methods disagree, examining whether one method systematically outperforms the other or if the disagreements are balanced. Unlike unpaired tests that assume independence between samples, McNemar’s test accounts for the paired nature of our data, where each clone instance is evaluated by both methods. The test statistic follows a chi-squared distribution with one degree of freedom, and a significant result ( _𝑝< ._ 05) indicates that the two methods exhibit statistically different performance patterns. 

For each research question, we applied McNemar’s test. For RQ1, we compared the performance of GPT-4o and JDeodorant on code clone classification. For RQ2, we excluded JDeodorant cases with precondition violations and retained only those that could be refactored. We then compared compilation success, test pass rates, and correctness of refactoring. In evaluating test pass rates, we considered only correctly compiled cases and excluded those that failed to compile. For RQ3, we conducted pairwise comparisons among the six models that produced clone detections, excluding GPT-3.5 since it did not detect any clones. Here, we focused solely on correctness of refactorings, as all models were able to detect clones and produce refactorings that compiled and passed test cases.

#### _5.2. Takeaway 2: GPT refactoring despite JDeodorant precondition violations_ 

The refactorings are not applicable for certain code clones due to precondition violations for JDeodorant. JDeodorant’s refactorability analysis (Tsantalis et al., 2015), involves a series of checks to ensure that refactoring preserves the original behavior. These preconditions cover 

various aspects, such as ensuring that control or data dependencies are not broken. When one or more of these preconditions are violated, JDeodorant flags the clone as non-refactorable. However, we found 250 cases, despite these precondition violations, GPT-4o can still generate refactoring suggestions for the same clones, and pass the tests. 

For example, when refactoring clone ID 62-1-2 in JFreeChart, certain precondition violations, initially prevented the refactoring process. However, GPT-4o was able to successfully extract the method (see replication package Qian and AlOmar, 2025). In addition, the extracted method performed correctly and passed all tests. 

#### _5.3. Takeaway 3: Incomplete refactorings by extracting only part of the clone or duplicating logic in extracted methods._ 

As discussed in Section 4, we identified a total of 91 cases where GPT4o did not fully perform the refactoring. Among these incomplete cases, we observed two main issues. The first issue was that GPT-4o failed to extract entire clones; instead, it included only parts of the cloned code in the extracted method, leaving the remaining portions of the clone still in the original method. See example of failing to extract entire clones in the replication package (Qian and AlOmar, 2025), two classes in clone group 309-1-2 contain Type-2 clones, which are similar code fragments with minor differences in identifiers. The two code fragments compute positions x0 and y0, then draw the line based on the orientation situation. However, when GPT-4o performed the refactoring, it only extracted the code responsible for handling the orientation conditions and drawing the lines. 

The second issue occurred when GPT-4o did not properly abstract the common logic shared by two clones. Instead, it placed both clones entirely within the extracted method, using conditional statements to handle the different scenarios. For instance, in the replication package, example of failing to extract entire clones contains two code fragments clone group 1108-1-2 which are Type-2 clones, differing in identifiers. The extracted method generated by GPT-4o extracted more lines than just the clones. Furthermore, it included both clones in the method instead of consolidating the common logic. 

Among those 91 incomplete refactoring cases, we found that JDeodorant did not perform refactoring on 46 instances due to precondition violations. Interestingly, GPT-4o attempted to refactor those clones by shrinking them to smaller parts. While this strategy allows GPT-4o to perform the refactoring, it often led to the incomplete refactoring mentioned above.

_5.4. Takeaway 4: Excessive refactorings that go beyond the clone boundaries_ 

During our manual inspection, we also identified cases where GPT4o applied excessive refactoring, which introduced two issues. One issue is expanded refactorings, where refactoring introduced by GPT-4o included more lines than just the clones. As shown in example of expanded clone in our replication package (Qian and AlOmar, 2025), the two code fragments were Type-3 clones. The second issue is comprehensive refactoring. In these cases, GPT4o did not just refactor the cloned code. For example, in clone group 503-1-2 (see example of comprehensive refactoring in replication package Qian and AlOmar, 2025), similar codes were found in the compareTo methods of two classes, Hour and Minute. #### _5.5. Takeaway 5: All the considered tools and models besides GPT-3.5 can detect the clones_ 

In our case study, we tested nine tools and models for their ability to detect code clones: AntiCopyPaster, JDeodorant, Gemini 2.5 Pro, GPT3.5, GPT-4o, GPT-5, DeepSeek-R1, Claude Sonnet 3.7 and Claude Opus 4.1. Among them, GPT-3.5 was the only one that consistently failed to identify the clones present in the input files. Specifically, in 18 out of 20 cases, as demonstrated in Listing 14, GPT-3.5 produced unrelated code fragments completely, disregarding the provided instructions. For the remaining two cases, it incorrectly reported ‘no clones found’, despite the existence of duplicated code segments. As we can see from Listing 15, although two methods were clearly Type-1 clones, GPT-3.5 still output ‘no clones found’. 

In contrast, the other tools and models, including AntiCopyPaster, JDeodorant, Gemini 2.5 Pro, GPT-4o, GPT-5, DeepSeek-R1, Claude Sonnet 3.7 and Claude Opus 4.1 were able to detect all the clones correctly. This highlights a performance gap between GPT-3.5 and the other more advanced or specialized tools. These results suggest that GPT-3.5 requires further improvement before it can be considered reliable for clone detection tasks. 

#### _5.6. Takeaway 6: JDeodorant and AntiCopyPaster performed more stable refactoring than LLMs_ 

In our case study, we compared the performance of JDeodorant and AntiCopyPaster with seven LLMs: Gemini 2.5 Pro, GPT-3.5, GPT-4o, GPT-5, DeepSeek-R1, Claude Sonnet 3.7 and Claude Opus 4.1. With evaluation of clone refactoring across JHotDraw, the traditional tools JDeodorant and AntiCopyPaster achieved perfect results, with all “Extract Method” refactorings applied correctly, compiling successfully, and passing tests. Among the LLMs, GPT-4o showed strong performance with 14 correct refactorings out of 20, though 6 were incomplete. GPT5 achieved the best overall outcome, with 15 correct and 3 incomplete refactorings, and did not exhibit destructive behaviors such as direct method deletion. Claude Sonnet 3.7 ranked next with 11 correct refactorings, but also produced 6 incomplete cases, along with occasional issues such as direct method removal, call-based substitutions, and fragmented extractions. Claude Opus 4.1 performed 10 correct refactorings and 2 incomplete, but displayed more problematic behaviors, including direct deletions and post-extraction removals. DeepSeek-R1 succeeded in 7 cases, while 9 were incomplete, and it also showed unusual behaviors such as replacing cloned methods with calls to the originals or generating multiple fragmented extractions. Gemini 2.5 Pro-struggled significantly, with only 3 correct refactorings and frequent incomplete refactoring results. It also involved deletion of cloned methods. By con- 

trast, GPT-3.5 failed to perform any refactoring, as it did not reliably detect clones. 

#### **6. Implications** 

#### _6.1. Implications for practitioners_ 

**Guidance for applying LLM-based clone refactoring.** The findings from RQ2 reveal conditions under which LLMs tend to perform more or less reliably in clone refactoring. It shows that LLMs perform more reliably when handling same-file clones, especially Type-1 and Type-2 cases, where duplication is localized and the probability of compilation or testing failures is relatively low. In contrast, cross-file clones present a much higher risk of incomplete elimination and compilation errors. These observations suggest that practitioners may use LLM-based refactoring primarily for same-file clones, where it can significantly accelerate clone removal, but remain cautious in cross-file or highly coupled scenarios, where static or AST-based tools remain preferable. This distinction illustrates when to use what: LLMs are best suited for localized, same-file Type-1 or Type-2 clones, while static or AST-based approaches remain the safer option for cross-file or coupled clones. Even when applying LLM refactoring in favorable settings, outputs should be treated as candidate patches rather than production-ready code. Verifying clone elimination completeness, checking for under-extraction (i.e. incomplete refactoring) or over-extraction (i.e. expanded refactoring, comprehensive refactoring), and validating behavior preservation through tests are essential steps to integrate these methods responsibly into practice. 

#### _6.2. Implications for researchers_ 

**Broadening evaluation criteria for LLMs.** The results from RQ1 and RQ2 indicate that evaluating LLMs on clone tasks cannot be limited to detection accuracy, type classification, compilation, or functional preservation alone. Our experiments show that models sometimes misclassify clone types but still generate correct refactorings, while in other cases they classify correctly but fail to remove all duplication. This reinforces the need to treat classification and refactoring as related yet distinct tasks. Researchers may therefore adopt broader evaluation criteria that include clone elimination completeness, incomplete refactoring, expanded refactoring and comprehensive refactoring rates, and crossmodel consistency, in addition to traditional correctness checks. Such measures better reflect the practical value of clone management and connect directly to developer concerns about maintainability and trust. 

#### _6.3. Implications for tool builders_ 

**Toward hybrid and retrieval-augmented solutions.** Results from RQ3 show that different LLM models exhibit inconsistent behaviors: some achieve higher clone elimination rates but introduce more failures, while others preserve functionality but leave residual duplication. Compared to traditional tools such as JDeodorant or AntiCopyPaster, which demonstrated stability in controlled settings, LLMs offer flexibility but lack consistency. This points toward hybrid solutions in which static or AST-based analyses validate LLM outputs to ensure both completeness and correctness before changes are accepted. Tool builders can focus on supporting tools with dashboards that report metrics such as duplicate reduction, test pass/fail outcomes, and extraction coverage, so developers can make informed decisions about acceptance. A promising avenue is retrieval-augmented generation (RAG), which can ground LLM outputs in project-specific histories of accepted refactorings, coding conventions, and past clone removals. Evidence from the Improta et al. (2025) highlights the importance of curating high-quality retrieval corpora, since noisy examples directly increase the likelihood of poor generations. MANTRA (Xu et al., 2025) further demonstrates that combining

Tool builders can draw on these lessons by coupling retrieval with safeguards and repair pipelines to deliver consistent and trustworthy clone refactoring support. 

#### _6.4. Implications for educators_ 

**Teaching the use of AI for clone refactoring.** Our study also carries implications for software engineering educators. The results from RQ2 show that LLMs succeed more in same-file refactorings but struggle across files, providing a natural context for teaching students about the strengths and limitations of AI-assisted development. Classroom activities can ask students to apply LLMs to both same-file and cross-file clones, then critically evaluate the outputs using the same criteria applied in our study: completeness of clone elimination, compilation and testing outcomes, and readability of the extracted methods. Such exercises can train students to recognize when LLMs provide reliable support and when caution or alternative approaches are required. Framing LLMs as assistive partners rather than infallible tools equips future developers with a balanced understanding of how to integrate AI responsibly into software maintenance tasks. 

#### **7. Threats to validity** 

**Internal Validity.** Several factors could influence our findings. First, we used JDeodorant as the ground truth for evaluating clone classification. However, although JDeodorant is widely used, it relies on external clone detection tools such as Deckard for detecting clones, which introduces certain limitations. Specifically, the quality and usefulness of refactoring largely depend on the clones provided by Deckard, which do not always represent the most optimal set of code duplications. During our manual inspection, we encountered issues such as duplicated clone groups or closely similar groups differing by just one or two lines. To mitigate this limitation, we skipped duplicated clone groups when we manually reviewed the refactoring. Also, the effectiveness of LLMs heavily depends on prompt engineering. Different prompts could lead to different results. To mitigate this, we experimented with zero-shot, oneshot and few-shot learning. Since LLMs can produce slightly different responses each time, especially across different runs, we set the temperature to 0 in our experiments to ensure the outputs were as consistent as possible. 

**External Validity.** Because JFreeChart provided the largest amount of data among available projects when analyzed using JDeodorant, our study was based on JFreeChart, which is a Java-based open-source project. This may cause our results not generalize to other programming languages or other open-source projects. And JFreeChart project may have characteristics such as consistent coding style, mature architecture, or limited use of advanced language features that make it more amenable to automated refactoring. As such, the quality of results observed with GPT-4o may not directly generalize to other codebases. In projects with more complex or less regular code, the effectiveness of refactoring tools could be significantly lower. To complement this, we conducted an additional case study on the JHotDraw project, where we compared multiple LLMs (Gemini 2.5 Pro, GPT-3.5, GPT-4o, GPT5, DeepSeek-R1, Claude Sonnet 3.7 and Claude Opus 4.1) alongside JDeodorant. However, the sample size of this case study was relatively small, with only 20 evaluated instances. Future studies are needed to evaluate the performance across more diverse projects. Another limitation of this study is the reliance on GPT-4o, a proprietary and black-box model whose future availability and behavior are not guaranteed. As OpenAI may modify or deprecate the model or its API, full replication of our results may not be possible. This threat should be considered when interpreting the generalizability of our findings. Also we evaluate GPT-4o, which cannot represent the capabilities of other LLMs. While 

previous studies suggested that GPT models perform well in clone detection (Khajezade et al., 2024; Moumoula et al., 2024; Zhang and Saber, 2024), its performance in refactoring may vary from other LLMs. 

#### **8. Conclusion** 

In this study, we systematically evaluated GPT-4o’s ability to classify and refactor code clones. We assessed clone classification accuracy, refactoring correctness, compilation success rates, and the use of lambda expressions. Our findings show that GPT-4o performs well in classifying Type-1 clones but struggles with distinguishing Type-2 and Type-3 clones, often misclassifying them. When refactoring clones across different files, GPT-4o encountered significant challenges. These results highlight both the potential and limitations of LLMs in code clone classification and refactoring. While GPT-4o can effectively assist in clone classification and refactoring, it still requires improvements in clone classification accuracy and handling cross-file dependencies. Our study provides valuable insights for researchers seeking to integrate LLMs into software maintenance workflows. To further explore how consistent LLMs are in clone identification and refactoring compared to traditional approaches, we conducted a complementary case study on the JHotDraw project. This study compared several LLMs, such as Gemini 2.5 Pro, GPT-3.5, GPT-4o, GPT-5, DeepSeek-R1, Claude Sonnet 3.7 and Claude Opus 4.1, against JDeodorant and AntiCopyPaster. Although limited in scale with 20 evaluated instances, the case study revealed clear differences in performance across models, with traditional tools providing stable results while LLMs showed varying levels of effectiveness. These findings emphasize that the capabilities of LLMs are not yet consistent across models, which highlights the importance of broader and more systematic evaluations in future work.
