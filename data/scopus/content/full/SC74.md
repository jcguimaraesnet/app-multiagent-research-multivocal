##### **1. Introduction** 

Large Language Models for code (code LLMs) have demonstrated remarkable performance on a variety of automated programming tasks. Automating such tasks is inherently complex, requiring the model to understand numerous concepts within the underlying code. However, despite this effectiveness on various code-related tasks, a fundamental question remains: _How deeply do these code LLMs truly understand the code they interact with?_ Current evaluations usually assess the usefulness (e.g., _𝑝𝑎𝑠𝑠_ @ _𝑘_ metric) of programs generated by these models based on the passed tests for the given tasks in several benchmarks such as HumanEval [1], MBPP [2], and CodeContests [3]. While these taskdriven evaluations measure end-to-end performance, they fail to reveal the code LLMs’ capabilities in understanding code semantics, which could be crucial for ensuring the reliability of code LLMs. 

Prior studies [4–6] have begun to explore the capabilities of LLMs in code understanding. However, a deeper understanding of the LLMs’ capabilities to capture semantic aspects, including control dependence 

and data dependence, remains elusive. Wan et al. [4] prioritize syntactic correctness over semantic understanding. Evaluation methods might assess whether generated code compiles without truly grasping its intended functionality. Hooda et al. [5] evaluate LLMs’ capability in understanding code semantics through a single task, which is code completion. This restricts our understanding of how broadly code LLMs can understand code. Ma et al. [6] evaluate the code LLMs by testing their ability to reconstruct explicit code syntax and semantic structures, such as predicting nodes and edges in Abstract Syntax Tree (AST) or identifying control and data dependencies. It is unclear how LLMs arrive at their outputs, making it difficult to assess their reasoning process and identify potential biases in syntactic and semantic understanding. 

In this paper, we propose Empica, a novel framework for **systematically and empirically evaluating code LLMs’ capabilities in understanding essential code semantic aspects** . Our approach involves analyzing the _sensitivity_ and _robustness_ of code LLMs to changes in code semantics. Within this framework, we focus on two fundamental semantic relations in the programming analysis: _control dependence_ and

We apply our framework to six critical software engineering tasks with varying levels of semantic complexity: code summarization, method name prediction, output prediction, bug detection, code completion, and code translation. We evaluate the models for each task without fine-tuning or additional training data to minimize potential biases. Ideally, models must recognize semantic changes introduced by non-preserving transformations and adjust their outputs accordingly. Our work incorporates eight state-of-the-art (SOTA) code LLMs: DeepSeek [10], Code Llama [11], MagicCoder [12], CodeQwen [13], OpenCoder [14], StarCoder [15], GPT-3.5 [16], and GPT-4o [16]. ##### **2. Background and related work** 

##### _2.1. Large language models for code_ 

Large Language Models for Code (code LLMs) are becoming popular in the Software Engineering (SE) industry. Multiple code LLMs, such as Github Copilot<sup>1</sup> , AlphaCode [18], and GPT-3.5 [16], have been employed to speed up the software development process. These models play a significantly important role in the automation of multiple SE tasks such as code completion [19], program repair [20], or test generation [21], etc. As the sizes of these models and the amount of training data have increased, code LLMs have exhibited impressive performance in these tasks [19–21]. 

Existing LLMs for code are constructed based on the Transformer [22] (encoder–decoder or decoder-only). These models are trained on extensive, unlabeled corpora of code-related data, enabling them to generate code in various programming languages. The training process typically involves the causal language modeling objective, where the model learns to predict the next token in a sequence. The success of LLMs in the SE task can be attributed to three main factors: Large Size, Premium Data, and Expert Tuning [23]. 

Recent advancements in code LLMs demonstrate a trend toward larger model sizes, accompanied by superior performance [24,25]. These studies show that increasing the number of model parameters can significantly enhance model capabilities. For instance, early models such as CodeT5 [26] had limited success due to their relatively small size. In contrast, modern models like Codex, Code Llama, and DeepSeek achieve remarkable performance on various benchmarks with their massive number of parameters. 

Additionally, high-quality code corpora enable LLMs to learn different programming languages and coding paradigms effectively. To ensure the quality of the training corpus, it is common for LLMs to perform extensive data preprocessing on the large amount of data collected. One common strategy is removing likely auto-generated or unfinished code files. Additionally, specific rules are employed to filter out uncommon code files. These rules include the repository star rating, file size, line length, and alphanumeric rate. 

Expert tuning is also essential for optimizing model performance. Common practices include using the Adam optimizer or its variants and further training from the models pre-trained on text. Key hyperparameters such as learning rate, batch size, window size, warmup steps, gradient accumulation steps, and sampling temperature require expert tuning. Specialized tokenizers, such as Byte-level Byte-Pair-Encoding and SentencePiece, are applied to code corpora to handle programming syntax effectively. 

##### _2.2. Evaluating code LLMs_ 

Several benchmarks have been established to evaluate the capabilities of code LLMs in various SE tasks. For example, Chen et al. [1] introduced _HumanEval_ , a popular benchmark designed to assess the functional correctness of the generated code in multiple programming languages. Austin et al. [2] proposed _MBPP_ dataset, which contains solvable problems for entry-level programmers. This dataset is also widely used to evaluate the program synthesis of code LLMs [27–29]. Another manually crafted benchmark _ClassEval_ [30] focuses on evaluating LLMs’ performance on class-level code generation. Experimental results on these benchmarks have demonstrated the remarkable performance of code LLMs across various SE tasks, significantly accelerating the software development process. 

In addition, many approaches [5,25,31] proposed to further thoroughly test code LLMs in code generation and code understanding tasks. Liu et al. [31] conducted comprehensive experiments on eight models using five benchmarks to examine the capabilities and limitations of these models. In another research, Lie et al. [25] evaluates whether the code generated by code LLMs is really correct by introducing more sophisticated test cases for the problems in the benchmark _HumanEval_ . Their experimental results show that the performance of code LLMs at _pass@k_ for the problems in _HumanEval_ significantly decreased when additional test cases were included. Hooda et al. [5] assessed the code completion capabilities of code LLMs by perturbing concept predicates such as control or data flow in the inputs. Their evaluations of ten LLMs reveal that these perturbations frequently led to incorrect code completion by the models. 

Furthermore, the abilities of code LLMs in understanding different aspects of the programs have also been investigated in the literature [4,6,32–34]. A common approach involves probing tasks, which investigate what models can capture by requiring them to predict

Specifically, we introduce controlled semantic-preserving and semantic non-preserving transformations into input code and observe outputs to evaluate the models’ robustness and sensitivity. By examining whether a model produces consistent outputs for semantically equivalent code and distinguishing outputs for semantically different code, we target to evaluate a deeper and task-oriented understanding of code semantics. This evaluation methodology allows us to go beyond structural reconstruction and focus on the functional and semantic consistency of code LLMs in SE tasks. Our framework can complement the existing work to provide a more holistic assessment of code LLMs’ capabilities, enabling a thorough evaluation of both explicit and implicit facets of semantic comprehension.

##### _4.1. Semantic-aware code transformations_ 

This work aims to evaluate code LLMs’ capability in code semantic understanding in the two fundamental code semantic relations: _control dependence_ and _data dependence_ . Fig. 3 illustrates the overview of our framework, Empica, containing four main components: _(1) Program Generation, (2) Program Transformation, (3) Code Understanding, and (4) Robustness & Sensitivity Evaluation_ . 

First, for a code LLM _M_ under evaluation, Empica employs _M_ to generate code. Then, Empica directly uses the generated code to evaluate the capabilities of _M_ in understanding code semantics. This is reasonable because evaluating _M_ in understanding arbitrary code could hinder the capability of _M_ . Indeed, _M_ could misunderstand an entirely new code snippet, especially if the code significantly deviates from the patterns and distributions seen during training (i.e., _outof-distribution_ ) [37]. Therefore, Empica conducts the analysis on the code generated by the model itself for a more impartial evaluation of the model’s capability and to mitigate the potential risk of capability hindering due to the unseen data. 

Next, Empica systematically introduces controlled transformations into the input code and observes how the model reacts in _M_ ’s output for a specific code understanding task. Empica applies a variety of transformations designed to target both control and data dependencies in the input code snippets. We employ both _semantic preserving (SP) transformations_ and _semantic non-preserving (SNP) transformations_ to modify the code snippets. Specifically, given a code snippet, _SP transformations_ create variants of the code while maintaining the original functionality of the snippet in its variants. In contrast, _SNP transformations_ produce variants whose functionality differs from the original ones. 

Ideally, if _M_ is capable of understanding code semantics, _M_ ’ predictions must be _robust to the SP transformations_ and _sensitive to the SNP ones_ . For a specific task, _M_ must produce the equivalent responses to the original input code and its variants created by the _SP_ transformations. In contrast, _M_ ’s outputs for the original input and its variants modified by the _SNP_ transformations are expected to be different. 

For thorough evaluation, Empica selects six representative SE tasks that heavily require code understanding, including _code summarization_ , _method/function name prediction_ , _output prediction_ , _bug detection_ , _code completion_ , and _code translation_ . Given a code snippet _𝑐_ ∈ , a _transformation operator 𝑡_ :  →  is a function that maps _𝑐_ from and to a space  of all possible code snippets in the given programming language. Let assume that we have an oracle function _𝛾_ :  → , which maps from the space of code snippets  to the specification space  which represents the code behaviors. A code transformation _𝑡_ can be categorized into two groups based on its impact on _𝛾_ ( _𝑡_ ( _𝑐_ )), _semantic preserving_ ( _SP_ ) and _semantic non-preserving_ ( _SNP_ ) transformations. 

**Definition 1** ( _Semantic Preserving (SP) Transformation_ ) **.** Given a code snippet _𝑐_ ∈ , a transformation _𝑡𝑝_ is a semantic preserving transformation if the modifications introduced by applying _𝑡𝑝_ on _𝑐_ do not affect its specified behaviors, ∀ _𝑐_ ∈  _, 𝛾_ ( _𝑐_ ) ≡ _𝛾_ ( _𝑡𝑝_ ( _𝑐_ )). 

**Definition 2** ( _Semantic Non-preserving (SNP) Transformation_ ) **.** Given a code snippet _𝑐_ ∈ , a transformation _𝑡𝑛_ is a semantic non-preserving transformation if the modifications introduced by applying _𝑡𝑛_ on _𝑐_ change its specified behaviors, ∀ _𝑐_ ∈  _, 𝛾_ ( _𝑐_ ) ≢ _𝛾_ ( _𝑡𝑛_ ( _𝑐_ )). 

To systematically investigate the capability of code LLMs in understanding control and data dependencies, we curate the transformations that target these relationships. In this work, we select eight representative operators [38–40], including both _SP_ and _SNP_ transformations. Table 1 introduces the details of the selected transformation operators and the corresponding examples. 

There are several reasons for choosing these eight operators. First, these transformations are _directly_ relevant to control and data dependencies, which form the main focus of this study. Second, the selected operators are commonly used in practice and broadly applicable across programming languages and domains, making the evaluation more representative and less biased toward specific syntactic constructs or contexts. Third, these operators cover a wide range of code structures, such as loops, conditionals, variables, parameters, and arithmetic operations, ensuring the evaluation comprehensively addresses key programming concepts. By focusing on these eight operators, we establish a targeted and efficient evaluation framework that balances coverage, relevance, and applicability.

Given a code snippet _𝑐_ and a transformation operator _𝑡_ , the transformed code _𝑐_<sup>′</sup> , i.e., _𝑐_<sup>′</sup> = _𝑡_ ( _𝑐_ ), can be generated by applying _𝑡_ on a _single_ or _multiple_ appropriate positions in _𝑐_ . In Empica, _𝑡_ is applied on _all_ transformable positions in _𝑐_ . In practice, single transformations often involve minor changes that might not sufficiently challenge the model. For instance, a code LLM might produce consistent responses for the original input _𝑐_ and the transformed input _𝑐_<sup>′</sup> simply because it fails to detect a small change and cannot differentiate the two inputs rather than due to an actual understanding of the code’s semantics. By applying transformations to all transformable positions, the scope of perturbations is maximized, compelling the code LLM to demonstrate its capacity to maintain output consistency for _SP_ transformations and detect logical inconsistencies for _SNP_ transformations across a broader range of changes. Moreover, we empirically evaluate the impact of transformation sizes on the robustness and sensitivity of code LLMs in Section 6.1.7.

##### _4.2.1. Code summarization_ 

_Code summarization_ ( _T𝑆_ ) is the task of concisely describing a code snippet’s overall functionality, purpose, and behavior in a natural language [43]. ##### _4.2.4. Bug detection_ 

_Bug detection_ ( _T𝐵_ ) is often formulated as a binary classification problem, where the goal is to determine whether a given code snippet _𝑐_ is correct or incorrect [48]. An ideal code LLM comprehending the code semantic should yield consistent predictions for _𝑐_ and its _SP_ transformed variants. Since _SP_ transformations retain the semantic equivalence of the code, _𝑐_ and its _SP_ transformed versions should always receive the same classification either correct or incorrect. 

Conversely, the code LLM should exhibit sensitivity to _SNP_ transformations by producing different predictions for _𝑐_ and its _SNP_ transformed variants _𝑐_<sup>′</sup> = _𝑡𝑛_ ( _𝑐_ ). In particular, if the model predicts _𝑐 to be correct_ , it should classify _𝑐_<sup>′</sup> as incorrect, as the _SNP_ transformation _𝑡𝑛_ modifies _𝑐_ ’s behaviors in a way that violates its intended semantics. Note that this property does not necessarily hold if the original code _𝑐 is already predicted as incorrect_ . While _𝑡𝑛_ changes behaviors of _𝑐_ , it cannot guarantee a correction, and both _𝑐_ and _𝑐_<sup>′</sup> may remain incorrect. 

To evaluate the code understanding capabilities of a code LLM _M_ in this _T𝐵_ task, we assess two aspects. First, we evaluate whether _M_ is _robust_ to _SP_ transformations by producing consistent predictions _𝑜_ and _𝑜_<sup>′</sup> for _𝑐_ and its SP transformed variant _𝑐_<sup>′</sup> = _𝑡𝑝_ ( _𝑐_ ), either _𝑐_ is predicted as correct or not. ##### _4.2.5. Code completion_ 

Given a partially written code snippet, _code completion_ ( _T𝐶_ ) predicts and suggests the next segment(s) of code to complete the given snippet based on the provided context [49]. Code completion can be performed at various levels of granularity, such as token, line, method, or file. In this work, we follow the similar method-level completion setting proposed in the related research [5], to evaluate the _𝑅𝑜𝑏𝑢𝑠𝑡𝑛𝑒𝑠𝑠_ and _𝑆𝑒𝑛𝑠𝑖𝑡𝑖𝑣𝑖𝑡𝑦_ of code LLMs in this _T𝐶_ task. 

Specifically, the original code snippet _𝑐_ and its transformed variant _𝑐_<sup>′</sup> are cut at the same location to create partial code _𝑝𝑐_ and _𝑝𝑐_ ′ , which form the inputs for the _T𝐶_ task. Then, a code LLM _M_ is employed to complete the partial code snippets _𝑝𝑐_ and _𝑝𝑐_ ′ . If _𝑐_ and _𝑐_<sup>′</sup> are semantically equivalent ( _𝑐_<sup>′</sup> = _𝑡𝑝_ ( _𝑐_ )), the completions generated by _M_ for _𝑝𝑐_ and _𝑝𝑐_ ′ should be similar, as the underlying semantic context remain unchanged. Meanwhile, if _𝑐_ and _𝑐_<sup>′</sup> are semantically non-equivalent ( _𝑐_<sup>′</sup> = _𝑡𝑛_ ( _𝑐_ )), the completions suggested for _𝑝𝑐_ and _𝑝𝑐_ ′ should be different, reflecting the changes in semantic intents and behaviors introduced by the _SNP_ transformations. 

To measure the similarity between the code completions generated by _M_ for _𝑝𝑐_ and _𝑝𝑐_ ′ , we employ BLEU [50], which is a widely used metric in completion-related tasks [51,52]. Let _𝑜_ and _𝑜_<sup>′</sup> be completions suggested by _M_ for _𝑝𝑐_ and _𝑝𝑐_ ′ , respectively. BLEU evaluates the overlap between _𝑜_ and _𝑜_<sup>′</sup> , considering both precision and _𝑛_ -gram continuity. The higher BLEU score, the more similar _𝑜_ and _𝑜_<sup>′</sup> . Therefore, the similarity function in measuring _𝑅𝑜𝑏𝑢𝑠𝑡𝑛𝑒𝑠𝑠_ and _𝑆𝑒𝑛𝑠𝑖𝑡𝑖𝑣𝑖𝑡𝑦_ of _M_ is expressed by _𝑠𝑖𝑚_ ( _𝑜, 𝑜_<sup>′</sup> ) = _𝐵𝐿𝐸𝑈_ ( _𝑜, 𝑜_<sup>′</sup> ). 

##### _4.2.6. Code translation_ 

The _code translation_ task ( _T𝑇_ ) involves converting a code snippet written in one programming language into an equivalent snippet in another programming language [53]. This task requires the translated code not only to adhere to the syntax and conventions of the target language but also to preserve the original semantics and functionality. To evaluate the _𝑅𝑜𝑏𝑢𝑠𝑡𝑛𝑒𝑠𝑠_ and _𝑆𝑒𝑛𝑠𝑖𝑡𝑖𝑣𝑖𝑡𝑦_ of code LLM _M_ in this task, Empica employs _M_ to translate both the original code snippet _𝑐_ and its transformed variant _𝑐_<sup>′</sup> into the same target language. If _𝑐_ and _𝑐_<sup>′</sup> are semantically equivalent ( _𝑐_<sup>′</sup> = _𝑡𝑝_ ( _𝑐_ )), the translations produced by _M_ should also maintain semantic equivalence. Conversely, if _𝑐_ and _𝑐_<sup>′</sup> are semantically non-equivalent ( _𝑐_<sup>′</sup> = _𝑡𝑛_ ( _𝑐_ )), their corresponding translations should diverge, reflecting the differences in their semantics. 

Similar to the code completion task, the similarity between the generated translations is evaluated using the widely adopted metric, BLEU. This setup allows us to evaluate whether _M_ is robust enough to produce consistent translations for _SP_ transformations while being sensitive enough to generate distinct outputs for _SNP_ transformations. Furthermore, this task also provides valuable insights into the ability of code LLM _M_ to understand and accurately convey cross-language semantics. 

##### _4.3. Prompt design_ 

In this work, Empica utilizes prompting to evaluate code LLMs with the code understanding tasks. The design of the prompt can significantly impact the performance of the models. Thus, we aim to design simple prompts to mitigate the risk that the models could perform poorly because of misunderstanding the prompts. To design appropriate prompts for each task, we evaluated and optimized the prompts via some trial queries. Note that each LLM could have different prompt formats; we follow their original papers and the provided templates on _HuggingFace_ to adapt for prompt design. Generally, the prompt template is defined as follows: 

‘‘Given the following code snippet [CODE]. [TASK DESCRIPTION].’’ 

where [CODE] specifies the input code and [TASK DESCRIPTION] specifies the query for the task. For example, for code summarization, the [TASK DESCRIPTION] is ‘‘Please summarize the given code snippet’’. The detailed prompts for the other tasks are shown on our website [17]. 

##### **5. Evaluation design** 

To evaluate code LLMs’ capability in understanding code semantics, we seek to answer the following questions: 

- **RQ1:** **_Robustness and Sensitivity Analysis_ .** How robust are the state-of-the-art code LLMs [10–16] to the _SP_ transformations in code understanding tasks? How sensitive are code LLMs to _SNP_ transformations in these tasks? 

- **RQ2:** **_Model Size Analysis_ .** Does code LLMs’ size affect their capability to understand code semantics? 

- **RQ3:** **_Correctness Correlation Analysis_ .** Is there any correlation between the code semantic understandability of code LLMs and the correctness of their generated code? 

- **RQ4:** **_Semantic Inference Analysis_ .** Can the code LLMs correctly capture the code semantics and directly infer the relations, including control and data dependence?

##### _6.5. Threats to validity_ 

As for any empirical study, there are several threats to the validity of our results and conclusions. The first potential threat lies in the design of the prompts. To mitigate this threat, we carefully follow the prompt templates of the studied models to design and optimize the prompts for each code understanding task. Second, there is a certain level of randomness of the studied code LLMs in the generation procedure, which could affect the models’ performance. To reduce this threat, we set the temperate value to 0 to minimize the randomness. Another threat could lie in the selection code LLMs. To mitigate this threat, we chose diverse and representative models, including both open-source and commercial ones. We also conducted experiments on different model sizes. In addition, one threat could come from the evaluation procedure, i.e., different performance metrics may exhibit different biases. To mitigate this threat, for each code understanding task, we measure the similarity of the models’ outputs for the original code and the transformed code at different levels. For example, for code summarization, the similarity is measured at both semantic and lexical levels. Another potential threat to the validity of our evaluation arises from our reliance on a static analysis tool, as the reference for assessing the accuracy of code LLMs in inferring control and data dependencies. In practice, static analysis tools may produce false positives and false negatives, which could impact the accuracy of our evaluation. To mitigate this threat and ensure the reliability of our findings, we leverage Joern, a tool that has been extensively validated and adopted in prior research [57–59]. While the accuracy of Joern can influence our evaluation, Joern is a well-established tool for program analysis, and it provides a consistent and systematic baseline, which is valuable for capturing general trends in the performance of code LLMs. 

##### **7. Conclusion** 

In this paper, we introduce Empica, a novel framework designed to systematically and empirically evaluate the code semantic understanding of code LLMs. Specifically, Empica systematically introduces controlled modifications into the input code and examines the models’ responses. For a given code snippet _𝑐_ , if a code LLM understands the semantics of _𝑐_ , the model should produce consistent and robust outputs for _𝑐_ and its semantic equivalent variants while generating different responses for _𝑐_ and its semantic non-equivalent variants. Our experimental results on eight SOTA code LLMs and six representative code understanding tasks reveal that the _𝑅𝑜𝑏𝑢𝑠𝑡𝑛𝑒𝑠𝑠_ and _𝑆𝑒𝑛𝑠𝑖𝑡𝑖𝑣𝑖𝑡𝑦_ of code LLMs to code transformations vary significantly across tasks and transformation operators. In addition, code LLMs exhibit better robustness to the semantic preserving transformations than their sensitivity to the semantic non-preserving transformations. These results highlight a need to enhance the model’s capabilities of understanding code semantics, especially sensitivity.
