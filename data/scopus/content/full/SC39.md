# Intent-LLM: A Framework for Automated Network Configuration Through Code Generation 

Claudio Provvedi , Lorenzo Seidenari , Benedetta Picano , _Member, IEEE_ , and Romano Fantacci , _Life Fellow, IEEE_ 

**_Abstract_ —Intent-based Networking (IBN) is designed to streamline the complexities of network configuration and management. Instead of focusing on the granular implementation details, IBN defines high-level objectives that describe the target state and behavior of the network. This paper proposes a Large Language Model-based approach to translate the declarative intent of users into Python code. The approach designed and implemented is a novel framework with the objective of optimizing the network performance, producing operational Python code for solving instances of routing and resource allocation problems, and it is able to modify network topology, also considering energy efficiency. The framework developed is inspired by ViperGPT, a software tool where vision-and-language models are split into subroutines to produce results for computer vision queries. More specifically, ViperGPT demonstrates how an LLM can generate and orchestrate Python code to call a predefined set of domain modules through an API, dynamically composing them into executable programs, a paradigm that we adapt to networking by defining an analogous API for intent translation. Validation of the proposed approach is provided, considering both the syntactical correctness and network effectiveness of the code produced. Finally, the human feedback is provided to test the strategies generated.** 

**_Index Terms_ —Intent-based-networks, large language models, generative artificial intelligence.**

Unlike other studies involving the application of LLMs to IBN, we draw inspiration from the modular paradigm utilized in computer vision, as presented in [14]. 

Authors in [14] propose ViperGPT, a framework in which a code-generation model composes a set of pre-existing vision modules to solve complex visual queries. Its scope is strictly limited to computer vision and relies on a predefined API of visual operations. Our work differs in both purpose and architectural design. While the high-level idea of coordinating modular components through generated code is conceptually related, the context of Intent-Based Networking introduces a completely different set of abstractions. Here, the objective is to translate declarative intents into runnable operations for the main classes of network management problems, e. g, routing, resource allocation, topology management, which requires domain-specific APIs that have no correspondence in vision tasks. For this reason, the framework developed in this paper includes a fully new set of APIs designed explicitly for networking. These APIs define the operational space through which the LLM can act on the network. To the best of our knowledge, no prior work had introduced a modular, APIdriven architecture for intent translation in networking through LLMs. The proposed design therefore extends the intuition behind code-based modular reasoning into a new domain, rethinking it from the ground up to suit the requirements and semantics of network management.

the whole IBN process that goes from the intent fulfillment phase to intent assurance phase. 

Utterances analysis inside sentences given as network intents is not this much popular as subject in the existing literature, but there is a Manias et al. work [26], in which authors, while addressing the problem of semantic routing in LLMassisted intent-based 5G core network, talk extensively about the prompt generation task. They explored various prompt transformation methods and analyzed utterances number and variety for many types of intent, in relation to performances of prompts with and without model’s hallucinations. In the end they suggest Retrieval Augmented Generation (RAG) as a potential solution to still improve performances. 

Compared with existing LLM-based IBN solutions, which typically rely on direct text-to-configuration generation or rule extraction, the proposed approach introduces a fundamentally different technical pipeline. Current works use LLMs either to classify intents, derive policies, or produce semi-structured configuration templates; however, these strategies operate in a monolithic manner and do not provide an executable reasoning layer. In contrast, our framework adopts a modular architecture inspired by ViperGPT, where the LLM is not asked to output the final configuration, but rather to compose executable Python programs by calling domain-specific API functions. This shift from text generation to code synthesis introduces a set of capabilities that are absent in existing IBN literature: (i) a clear and verifiable operational action space, defined by networking primitives such as routing, resource allocation, and topology management; (ii) architectural flexibility, since new capabilities can be added by extending the API; and (iii) improved interpretability, as the generated program explicitly exposes the sequence of network operations selected by the model. To the best of our knowledge, no prior IBN-LLM research employs a program-composition mechanism analogous to that used in computer vision, making our design a novel and domain-tailored reinterpretation of modular LLM reasoning for network management. 

## III. AN LLM-BASED FRAMEWORK FOR IBN 

## _A. LLM Fundamentals_ 

LLMs originated from the systematic scaling of Pre-trained Language Models (PLMs). The transition from millions to billions of parameters revealed unanticipated emergent properties, signifying a qualitative leap in model capabilities as a function of increased parameterization. The well known advantages of LLMs are the following. 

- In-Context Learning: the ICL ability consists in generating the expected output, without requiring any additional training or gradient update, by simply infer it form a known context. This is the most known emergent ability, first discovered in the GPT-3 model by OpenAI [27]. 

- Instruction Following: by showing them instructions formatted via natural language descriptions, LLMs perform well on unseen tasks described in the same form of instructions [27]. With this ability the model can solve new tasks, without requiring explicit examples, showing a greatly improved generalization capability.

- Step-by-Step Reasoning: with a chain of prompts strategy, LLMs can solve tasks, that involves intermixed steps to derive the final answer. These models are said, to have gained this ability due to their heavy training in programming code [27]. 

In addition to generating high-quality natural language text, LLMs also show strong abilities to generate formal language. As a result of pre-training on specific corpora, these skills are especially regarding computer programs (i.e. code) that satisfy some conditions. Moreover, this ability, called “code synthesis”, unlike those regarding natural language, is easier to test and evaluate, by executing generated code with the corresponding language compilers and interpreters. 

## _B. Methodological Perspective_ 

This work adopts a system-oriented and simulation-based research methodology, aimed at investigating the feasibility and effectiveness of LLM-driven intent translation in networking environments through controlled experimentation. From a methodological viewpoint, the proposed framework is structured around three main stages. First, an operational action space is defined through a modular Python-based API, which exposes a constrained set of high-level networking primitives (e.g., routing, resource allocation, topology management, and energy optimization). This step establishes the functional boundaries within which the LLM is allowed to operate. Second, a structured prompting strategy is designed to provide the model with the necessary contextual information, including API documentation, scenario descriptions, and natural-language intents. This enables the LLM to perform intent-to-program translation by composing executable 

code through the available primitives. Third, the generated programs are executed within a controlled simulation environment, where their syntactic validity and operational impact are systematically evaluated. A key methodological principle underlying this design is the explicit separation between system specification and system usage. The API definition determines the admissible operational space and remains stable across experiments, while network scenarios and intents are varied to assess generalization and robustness. This separation enables reproducible experimentation and supports systematic analysis of the framework’s behavior under heterogeneous conditions. The evaluation protocol is designed to reflect the main phases of the intent fulfillment process. Syntactical correctness assesses compliance with the exposed API, human comparison evaluates alignment with expert implementations, and network effectiveness measures post-deployment compliance with the intended objectives. Together, these metrics provide complementary perspectives on the functional, interpretative, and operational quality of the generated solutions. Overall, this methodological framework enables a principled investigation of LLM-based intent translation as an executable and controllable process, combining modular system design with empirical validation in a simulated setting.

The API has to be built around a network environment, which will represent the cornerstone of our LLM-based approach. Following this, we decided to proceed with our own handcrafted network simulator, and since there is no reference general-purpose language for network functionalities, we have resorted here to the Python language. According to speculations, the reasons to support this choice can be: (i) diffusion, as many other works dealing with LLMs are written in Python, (ii) pre-training, as LLMs are very familiar with Python, considering that they are trained on the huge amount of code written in this language available on the internet, (iii) clearness, as we said before, we needed an easily understandable and well-known language to build up our API-based approach. 

The chosen framework for event simulation in Python is SimPy. In reference to the API documentation, we decided to follow the Docstrings Python code documentation standard as in [14], and proceeded to add short descriptions, obviously written in natural language. First of all our LLM needs to provide a description of the purpose of a class or a function. Specifically, for classes, we described their attributes alongside with the respective types; while for methods we described their input parameters and returned outputs, as well as the types. Finally, when relevant, we added a concise example where the class or method can be seen in use.

Finally, it is common in computer-vision program-synthesis frameworks to provide the LLM with a small set of illustrative examples that demonstrate how the API should be used. These examples take the form of pairs, where a naturallanguage query (an intent, in our case) is coupled with a minimal Python snippet implementing the corresponding logic through the available API functions. This technique, known as _query–code pairing_ , serves as an in-context demonstration of the expected reasoning pattern: by observing how a query maps to executable code, the model learns how to generalize this mapping to new intents without requiring task-specific fine-tuning [14]. An excerpt of the API documentation and representative query–code pairs are shown in Figures 2 and 3. 

An important tip to keep in mind while documenting APIs to work with LLMs, is that the focus has to be on specifying only some crucial points and do not give in input to LLM an extensive documentation of our API. Abstracting away from details is fundamental in this phase for three reasons that are: (i) the limited size of the LLM context window, which could make it infeasible to include a whole extensive implementation of the API (plus the prompts), (ii) the eliciting of hidden abilities via a strong training of the generalization capability, (iii) the achieving of maximum independence between the code generated and any further changes made to the API implementation. 

To avoid possible misunderstandings, it is important to distinguish between how the framework is _designed_ and how it is _used_ at inference time. From a design perspective, the documentation of classes, methods and API functions pertains to the definition of the operation space that the LLM is allowed to invoke. Developers specify only the signatures and short natural-language descriptions (docstrings) of each primitive (e.g., routing operators, topology modifiers, resourcemanagement functions). This phase is performed only when a new capability is added to the system and is therefore infrequent and independent of any particular network scenario. 

At inference time, the model receives a context that is the concatenation of: (i) this compact API specification, (ii) a Python snippet describing the current network scenario (topology, parameters, traffic model), and (iii) the naturallanguage intent. The API part plays the role of a stable “instruction set”, while the scenario code and the intent text can be changed arbitrarily from one experiment to another. Crucially, end-users do not need to re-document or modify

the API when they change the network settings: they simply provide a different scenario snippet and a new intent, and the LLM generates Python code by composing the already available API functions. 

This design has two practical consequences. First, from the user viewpoint, there is no additional effort once the API and an initial topology are defined: obtaining new configurations or optimization strategies only requires updating the scenario description and the intent. Second, the approach remains general and easily extensible without any fine-tuning of the LLM: adding a new functionality consists in introducing a new primitive in the API (with its signature and short description), which is then immediately usable by the model through incontext learning. 

In terms of performance, the impact of this mechanism is limited to the prompt length. The API documentation is concise and static, contributing a nearly constant number of tokens, while only the scenario description grows with network size. In our experiments, all considered scenarios fit comfortably within the model’s context window, and the syntactical correctness and network effectiveness metrics remained stable across increasingly large topologies. As a result, the documentation and API-specification phase does not introduce a significant runtime overhead nor does it hinder the scalability of the framework. 

A crucial aspect of the proposed architecture is that the operational boundaries of the framework are entirely defined by the API exposed to the LLM. Each module provides only a limited set of high-level primitives, such as routing operators, resource-allocation functions, topology modifiers, or energy-related utilities, while all low-level mechanisms of the simulator (event management, protocol logic, internal state updates, and cross-module interactions) remain inaccessible to the model. This design enforces a strict separation between modules: they can interact only through the explicit API surface, and no implicit state sharing or direct manipulation of internal data structures is allowed. As a result, the LLM can compose network actions only within this predefined action space, ensuring controlled behavior, preventing unintended side effects, and making the translation from intent to executable code predictable, verifiable, and easy to extend by adding new API functions. 

## _D. Model and Prompting Implementation_ 

The other main stakeholder in our approach is the model. Among the rich variety of available LLM, we had to choose the best one for our purpose. To do so, we took into consideration many models, progressively reducing the pool according to their characteristics: open-source or not, multimodal or not, local execution and installation or remote execution, and so on. Another good pick from our pool was one of the GPT series models by OpenAI. These models are closed-source, but they are extremely capable, even multimodal, and they come with an API for remote execution. So our final decision was to work with GPT 4o Mini, based also on the code generation performance measured in [28]. of [14]). With its chat completion style, the prompting of this model is based on a list of messages composing a conversation. Each message of this list, comes with a content obviously, but more importantly with a role, representing the role of the message author. The available roles we are interested in are: system and user. A system message can be used to prime the model with the desired behaviors and personalities. It is here that we are going to specify what is the job we need to be done, how we are expecting that job to be done, and what is the invariant context in which the model is going to operate. A user message is the base message, and is used to ask the question or pass the query (in our case an intent), to which a certain user needs an answer. 

The messages we give in input, be they system or user messages, are actually the prompts we pass to the model, and the way we submit those to the model, defines the prompting strategy. Usually, in visual-question-answering tasks, prompting consists of giving in a zero-shot way, the entire API preceded or succeeded by a description of the task we have to do [14]. But we can not proceed like this, since we do not have to deal only with a straight classical API (i.e., a set of methods to be exposed to give functionalities) as we have also the classes of our network simulator. Here, despite the fact that we can still call it API in its entirety, we are facing a non canonical API. So we have to design a brand new prompting strategy, because classes and methods must be treated differently. Another prompting technique, well know in the field of LLMs for Autonomous Network is a four-way strategy, based on Role, Instructions, Context and Intent [24]. We decided to merge the best of each discussed strategies and to opt for a zero-shot 4-way technique, for prompting GPT with four different messages, the first three in the form of system messages, and the fourth as a user message.

The first message is carrying the classes and the indications about the personality we need GPT to cast in. To do so, we preface the classes API documentation with the following sentence: “ _You are an expert Python programmer operating in a telecommunication network scenario represented by the following classes_ ”. In this way, we prepare the model to deal with Python classes representing a certain scenario. The second message carries the methods and the indications about the task GPT will face and how we need the job to be done. In this case we preface the methods API documentation with the following sentence: “ _Write simple and concise Python code using the functions below, that could be executed to provide an answer to a given intent in a given scenario_ ”. With this we tell the model we want him to draw from the given set of functions, and produce python code, that best answer our request, in the form of an intent. The third message is about the specific scenario, basically an example of a simulation run in our network simulator, starting from which we need to apply our intent. This network simulator example code is preceded by the sentence “ _The following is the specific scenario you are facing_ ”. With the addition of this third prompt we are adding the flexibility of making GPT operate, not only in fixed static cases, but also in different use cases, that can be changed every time it is needed. And finally we can prompt GPT with the fourth message, that is going to be a user message, like this “ _Intent: ’Find a path in graph from node1 to node2 with guaranteed optimality for end-to-end delay.’_ “, in which we tell our model the intent we need to apply. We have to set it to zero because, aiming to obtain executable code in output, the more the answer is deterministic, the better for us [14].

## _C. Network Effectiveness_ 

The other key activity of IBNs is intent assurance. However, in standard IBN architectures, assurance refers to the continuous verification of compliance throughout the entire lifecycle of the intent. In our setting, we do not implement continuous monitoring; instead, we introduce a one-shot post-deployment metric, which we call ‘Network Effectiveness’. This metric allows us to evaluate whether the network state resulting from the execution of the LLM-generated code satisfies the semantic goal of the intent immediately after deployment. Although Network Effectiveness does not model the full assurance loop, it represents a practical proxy to assess whether the translated code leads to an intent-compliant network state at execution time. By simply launching the simulation of this code inside our network simulator, we can monitor and track the results after executing the intent, and in so doing we can determine if the semantic goal was met. ## V. LIMITATIONS AND CONCLUSION 

This paper explored the application of LLM to the IBN context, promoting a modular approach common in visualquestion-answering learning tasks. In so doing, we developed a modular framework devoted to translate high-level user intent, expressed in natural language, into python code routines. The goal is the correct generation of code, both functionally and syntactically, instrumental to network management optimization tasks. In particular, the problems of routing, resource allocation, and network topology considering performance constraints were considered. Performance metrics include syntactical correctness, network effectiveness, and human validation. Future works will investigate performance of the proposed framework in more complex and general optimization network problems, also in comparison with competing LLM models for code generation [28]. Although the scenarios considered in this work remain well within the context window of the model, future extensions involving larger topologies will require assessing the framework under long-context conditions. We identify this as a promising direction for further investigation. The proposed framework inherently depends on the set of primitives exposed through the API, which defines the admissible operational space of the LLM. Requests that exceed this space may lead the model to generate unsupported implementations, potentially affecting execution reliability. Moreover, as a consequence of the LLM-based nature of the system, performance remains sensitive to the quality and specificity of the prompting context, with underspecified or ambiguous intents potentially degrading translation accuracy and network effectiveness. Finally, while the current study focuses on one-shot post-deployment validation, future work will investigate closed-loop assurance mechanisms to continuously adapt configurations under evolving network conditions and long-term dynamics.
