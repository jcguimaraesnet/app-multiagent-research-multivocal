##### **1. Introduction** 

Automated code translation, a crucial aspect of software engineering, has streamlined the tedious process of converting code from one language to another, thereby enhancing efficiency and scalability with minimal effort. It facilitates seamless integration of diverse projects using different languages, reducing manual effort and the potential for human errors. As stated by Echeverria et al. [6], automated code translation has been valuable for modernizing legacy systems since it has lowered costs by minimizing reliance on multilingual developers and has fostered collaboration in open-source communities. It has also supported the development of cross-platform technologies, enabling the flexible migration of functionality between different platforms. Automating code translation thus aids in the maintainability and scalability of large and complex projects. Traditional translation tools rely on established syntax rules to convert code, which provides speed and predictability for 

straightforward tasks but has been less effective with nuanced contexts. In contrast, methods that incorporate Artificial Intelligence (AI) use contextual understanding capabilities of Large Language Models (LLMs) for more accurate and adaptable translations.

[13] (which employs a Mixture-of-Experts approach developed by Dai et al. [16]). All models were deployed using 4-bit quantization to significantly reduce memory

This study leverages Rosetta Code by Rosetta Code [17], a comprehensive coding chrestomathy website, to construct a robust translation dataset. The sampled dataset covers 17 everyday programming tasks with implementations across seven programming languages: C, C++, Golang, Java, JavaScript, Python, and Rust. This approach generated 42 translation pairs per task, serving as reference translations and covering diverse programming domains such as bitwise operations, integer operations, string manipulations, and file operations. This resulted in a total of 714 translations for each LLM. 

LLM performance was evaluated through multiple evaluation metrics. Primary metrics, which included CodeBLEU by Ren et al. [18], chrF by Popović [19], METEOR by Banerjee and Lavie [20], and BLEU by Papineni et al. [21], measured translation accuracy. Additionally, supplementary metrics were developed to evaluate code extractability and syntactic correctness. The unbiased Pass@k metric introduced by Chen et al. [22] was also calculated for evaluating the functional correctness of code generated by LLMs. Advanced techniques like Feature Ablation from Merrick [24] and Shapley Value Sampling from Mitchell et al. [25] were used to understand the LLM’s interpretative capabilities. The experimental protocol also incorporates an error correction mechanism. When an LLM generates code with errors, the error feedback is provided within the same conversation, instructing the model to explain and rectify the issue. This approach allows assessment of the LLMs’ capacity for selfimprovement through iterative prompting. Furthermore, the reasoning behind these corrections is analyzed to identify common error patterns across different programming languages. 

Various prompts were developed to address different types of LLMs, programming languages, and experimental stages. The regular expression-based code extraction methodology demonstrated a 99.4 % success rate across 10,710 translations evaluated from multiple LLMs. The LLM-generated translations, formatted using language-specific code formatters, compile successfully or demonstrate syntactical correctness in approximately 62.75 % of instances with DeepSeek-Coder-V2 and YiCoder achieving syntactically correct code generation in more than 80 % of the translation pairs.

##### **2. Background and related work** 

##### _2.1. Automated code translation_ 

Code translation has been a fundamental task in software engineering. It involves transforming source code from one programming language to another while preserving the original functionality. It is essential for modernizing legacy systems, cross-platform compatibility, and enhancing software performance. Automated code translation reduces maintenance efforts, improves reliability, and boosts non-functional properties like security and performance. It has enabled seamless integration between diverse projects, reduced reliance on multilingual developers, and fostered open-source collaboration. 

Traditionally, code has been translated using rule-based source-tosource compilers, which rely on hard-coded rules and program analysis to convert code between languages. However, these tools have been costly to develop and have been limited to specific language pairs, often failing to produce code that aligns with idiomatic expressions of the target language. In contrast, LLMs have presented a promising alternative for code translation, offering the potential to understand and generate

##### _2.2. LLMs in software engineering_ 

LLMs have been utilized in software engineering for code generation, code development, software testing, quality assurance, requirements engineering, code translation, and program repair (debugging and bug detection), with prompt engineering proving crucial for enhancing their accuracy and controlling their output. LLMs have required techniques such as iterative prompting, feedback, and program analysis to address some of their limitations in software engineering tasks. Code LLMs have been explicitly trained on code data, which renders them more effective at code generation and translation compared to general LLMs. On the other hand, general LLMs have been trained on a broader range of text and code data and thus have demonstrated greater proficiency with natural language instructions but exhibit less robustness in core coding tasks. Both categories of models benefit from the implementation of prompt engineering techniques aimed at improving code quality. 

The field of automated code generation has witnessed significant breakthroughs with the introduction of interactive and self-improving methodologies. Fakhoury et al. [60] proposed TiCoder, an interactive workflow that uses LLM-generated tests to clarify user intent and improve code generation accuracy by 45.97 % through guided intent clarification. Adnan et al. [61] developed automated debugging strategies that enable LLMs to identify and fix code flaws using intelligent debugging mechanisms rather than relying on preemptive test case generation. Complementing these developments, Zhang et al. [62] introduced a sophisticated approach that combines backtracking algorithms with program analysis techniques. This methodology allows models to revise and refine generated code iteratively, resulting in more robust and accurate code output that better meets developer specifications. 

The automation of software testing has experienced substantial innovation through the application of large language models, with researchers developing comprehensive frameworks that address multiple aspects of the testing life cycle. Sherifi et al. [63] presented an agent-oriented framework that represents a holistic approach to test automation. This framework integrates LLMs to generate unit tests, visualize call graphs, and automate test execution and reporting with high test coverage, significantly reducing the manual effort required in traditional testing workflows. 

Security considerations have become increasingly important in software development, leading to innovative applications of LLMs in vulnerability detection and security analysis. FuzzCoder by Yang et al. [64] proposed using fine-tuned large language models to guide fuzzing test mutations by formulating the entire process as sequence-to-sequence modeling, where LLMs learn to predict optimal mutation locations and strategies in byte sequences to trigger program crashes and vulnerabilities. The approach demonstrates significant improvements in crash detection across various file formats compared to traditional heuristic fuzzing methods like AFL, showing that LLMs can enhance automated vulnerability discovery in software testing. Similarly, LProtector, proposed by Sheng et al. [65], demonstrates an automated vulnerability detection system that leverages LLMs to identify security vulnerabilities in code through pattern recognition and static analysis. This approach represents a significant advancement in proactive security measures, enabling organizations to identify potential security issues earlier in the development process. 

The output format of code LLMs is not always consistent and can include natural language text, comments, or code in unexpected formats, which may require post-processing to extract the code. This variability in output has affected the reliability of evaluation metrics and the practical use of the generated code. Research from the likes of Abukhalaf et al. [26], Jiang et al. [27], White et al. [28], Zhuo et al. [29] shows that LLMs have also struggled with understanding the overall task and have acted like “next-token predictors”, often requiring adequate context via prompt crafting. 

##### _2.3. Evaluation of LLMs in code generation scenarios_ 

The evaluation of LLMs for code generation involves several key metrics that assess their performance beyond merely confirming the functional correctness of the generated code. Functional correctness, which ensures that the produced code generates the expected output for given inputs, has often been measured using test suites. This measurement is done with metrics such as Pass@k, which evaluates whether at least one of the top k code samples passes all tests, and Computational Accuracy (CA) for code translation, which assesses whether a transformed function yields the same output as its source counterpart. Efficiency is evaluated by measuring the runtime performance of the generated code, utilizing metrics such as average normalized runtime, and considering how well the generated code performs in comparison to that produced by other users on platforms such as LeetCode. Adherence to coding standards is evaluated to determine if the generated code adheres to best practices, thereby ensuring its readability and maintainability. Furthermore, code quality metrics—such as validity, maintainability, and the absence of code or security “smells” – are deemed essential for ensuring the overall quality of the generated code. In the context of code translation, translation accuracy metrics, including CodeBLEU, chrF, METEOR, and BLEU, have measured the quality of the translation. However, these text-based metrics have been found to be misleading, indicating that it is preferable to assess the functionality of the code through execution. 

##### _2.4. Interpretability and explainability of LLMs_ 

Interpretability in LLMs is essential for understanding their output generation and decision making, which helps to build trust, debug, and identify biases. Kokhlikyan et al. [30] highlighted that interpretability involves methods like attribution algorithms, categorized into primary, neuron, and layer attributions, which highlight the importance of input features. These methods can be gradient-based (e.g., Saliency, Integrated Gradients) or perturbation-based (e.g., Feature Ablation, Shapley Value Sampling). The Captum library supports these interpretability methods, offering implementations for various PyTorch models. Miglani et al. [31] showed that it allows flexibility in defining attribution features and target outputs (most likely decoded token sequences or a user-specified sequences), benefiting the evaluation of specific outcomes. Wang et al. [32] showed that explanations generated by LLMs alongside code provide insights into the model’s reasoning process, which is crucial for understanding how the LLM arrived at a particular solution or identified a bug. Palacio et al. [33] showed that this interpretability allows users to assess the reliability and accuracy of the code, going beyond simple functional correctness to evaluate the logic and approach used. 

##### _2.5. Related works_ 

Non-LLM code translation techniques use rule-based source-tosource compilers, like C2Rust by Immunant [14] and CxGo by Transpile [15], which convert code by analyzing it into an abstract syntax tree (AST) before translating it to the target language’s AST. Other methods include statistical learning approaches like those presented by Nguyen et al. [34,35], and neural networks that employ attention mechanisms in the work done by Chen et al. [36]. As Roziere et al. [37] pointed out, LLM-based methods utilize large language models trained on extensive code datasets and require prompt engineering to get the best results. LLMs can produce inconsistent output, require post-processing, and struggle with complex dependencies and API mappings, potentially introducing bugs. 

A study of 1700 code samples in five programming languages, by Pan et al. [38], identified 15 categories of translation bugs, primarily due to syntax and semantic challenges. These issues often stem from language differences, incorrect data types, missing library imports, and API mismatches. Manual analysis is often needed to identify these bugs, which can be time-consuming. The study found that LLMs struggle with real-world projects due to complex dependencies and language-specific

The BESTER study by Song et al. [39] explored an iterative process for improving code generated by LLMs. Initially, the LLM creates code from a natural language description, which is then tested. If the code fails, the LLM uses self-reflection to identify errors, review its output, and utilize test results to guide revisions. ##### **3. Methodology** 

##### _3.1. Large language models_ 

Many LLMs are openly available for code generation tasks on the HuggingFace platform by Wolf et al. [42]. Each of these LLMs has different architectures, parameters, context lengths, capabilities, and performance. Specifically for this study, instruction-tuned versions of the code generation LLMs are used since these models are fine-tuned to follow prompted instructions. Since LLMs can have a varying number of parameters, and the resources required for working with LLMs scale with the number of parameters in the model, only LLMs with less than 10 billion active parameters were chosen. The list of selected LLMs, along with their details, is summarized in Table 1. The LLMs were selected based on three principles: LLM should be finetuned for code generation, its weights and architecture should be open-source, and the number of parameters should be below 10 billion. Since LLMs require many resources, efficient deployment frameworks like llama.cpp by Gerganov [43] and Ollama by Morgan and Chiang [44] are important in ensuring LLMs can be run efficiently on any hardware and operating system. Hence, an additional constraint was put in place to ensure that the selected LLMs are not only compatible with HuggingFace but also with llama.cpp and Ollama frameworks. This ensures that these LLMs can be easily deployed and used in not only research applications but also in the software engineering industry. 

The LLMs selected for this study can be grouped into two distinct categories based on the architecture – LLaMa and its derivatives, and other architectures like Qwen2 and Gemma. Categorizing by 

architecture reveals shared characteristics and potential performance similarities derived from the same foundational model. A significant portion of the models, including Yi-Coder (both sizes), Granite Code (both sizes), DeepSeek Coder, Magicoder by Wei et al. [45], and Code LLaMa by Rozière et al. [46] utilize the “LLaMa” architecture. Models like CodeQwen1.5 by Q. Team [47], Qwen2.5-Coder by Q Team [48] (both sizes) utilize the Qwen2 architecture. Models like StableCode by Pinnaparaju et al. [49], DolphinCoder by Hartford [50], CodeGemma by Zhao et al. [51] and CodeGeeX4 by y Hartford (2024), CodeGemma by Zhao et al. (2024) and CodeGeeX4 by Zheng et al. [52] have their own distinct architectures. DeepSeek-Coder-V2 is different from other models selected in the study since it uses a Mixture of Experts (MoE) architecture called DeepSeekMoE, introduced by Shazeer et al. [53], which means that the total number of model weights that are actively used during decoding is significantly smaller than the total number of parameters in the LLM. 

##### _3.2. Quantization_ 

Quantization is a technique used to compress LLMs and make them more efficient during execution. Instead of using very precise numbers (like 32-bit floating point numbers) to represent the weights and activations of LLM, quantization uses lower precision numbers like 8-bit integers as shown by Dettmers et al. [54] in LLM.int8(). This reduces the memory footprint of the model, making it faster to load and run, and reducing the power consumption. Liu et al. [55] introduced 4-bit quantization, which makes it possible to run powerful LLMs on devices with limited resources. All the models used in this study were quantized to 4-bit integer precision using llama.cpp with the Q4_K_M quantization preset and deployed using Ollama. Q4_K_M is a specific quantization format within GGML specification by Gerganov et al. [56] that uses 4 bits per weight, and the 128-weight block is a common unit for processing. This strategy preserves critical layer precision while achieving substantial compression ratios, typically reducing model size by approximately 3.3x compared to 16-bit precision models. However, quantization can lead to a slight decrease in model accuracy due to the loss of information from reduced precision of the weights, which is often an acceptable trade-off to ensure that larger high-performance LLMs can be run on consumer-grade hardware. Empirical evaluation on the WikiText-2 dataset by Chen et al. [57] demonstrated that 4-bit quantization using Q4_K_M resulted in perplexity increases of 8–15 % compared to 16-bit models, while maintaining 85–92 % of original performance on downstream tasks such as MMLU (accuracy dropping from approximately 65 % to 55 % for 7B parameter models). Kim et al. [58] found that complex reasoning tasks requiring multi-step chain-ofthought showed greater sensitivity to quantization, with performance degradation of 10–20 %, compared to single-token prediction tasks which exhibited minimal impact. ##### _3.3. Programming languages_ 

To evaluate the capabilities of the Large Language Models, each LLM was asked to execute each code conversion task in the following languages: C, C++, Golang, Java, JavaScript, Python, and Rust. These languages were chosen to represent a range of programming paradigms, levels of abstraction, and technological trends. The languages can be broadly categorized into two groups – System Programming Languages and Application-Focused Languages. System Programming Languages (C, C++, Golang, Rust) prioritize performance and low-level system access, while Application-Focused Languages (Java, JavaScript, Python) are typically used for higher-level applications. A summary of these programming languages is provided in Table 2. 

These languages represent diverse challenges from an LLM performance perspective. They encompass varying syntax complexity, from C’s minimalist approach to Python’s readability and Rust’s complex type system, thus testing an LLM’s adaptability. They also represent different programming paradigms, like procedural (C), object-oriented (Java), and functional (Rust), along with varying levels of abstraction, from low-level system languages to high-level scripting languages. Fayyad et al. [69] highlighted that these languages also reflect key technological trends, such as the increasing focus on memory safety and performance, the growth of cloud-native and distributed computing, the expansion of web and cross-platform technologies, and the rising importance of data science and machine learning ecosystems. 

##### _3.4. Code translations_ 

Since there are seven languages (C, C++, Golang, Java, JavaScript, Python, and Rust) against which the LLMs are evaluated, 42 valid combinations of code translations are possible for each task selected from the Rosetta Code dataset. The LLM is asked to convert the code from a source to a target language along with a task description to ensure proper clarity on its code conversion task. This process tests the LLM’s understanding of diverse syntax, data structures, memory management models (where applicable), and standard programming paradigms when implementing the same algorithm or concept. 

##### _3.5. Dataset_ 

Rosetta Code by Mol [70] is a programming chrestomathy website that collects solutions to a variety of tasks in hundreds of different programming languages. The website presents a task, for example, “Implement the Bubble Sort algorithm”, and invites programmers to contribute solutions in their chosen language. The Rosetta Code dataset provides a valuable resource for evaluating code generation LLMs for several reasons. First, its diverse range of over 745 tasks, spanning simple exercises to complex algorithmic problems and data structure implementations, allows for comprehensive evaluation of LLM capabilities across various computational problems and programming paradigms. Second, the inclusion of solutions in 379 programming languages 

enables assessment of how well LLMs handle the nuances and idioms specific to each language. Furthermore, the presence of multiple solutions for each task in different languages provides reference implementations. These reference solutions, developed and refined by a community of programmers, ensure a high standard of quality and correctness, facilitating robust evaluation of the correctness and quality of LLM-generated code. Finally, while Rosetta Code solutions are generally smaller than real-world applications, they often represent core functionalities crucial to larger projects. Nanz and Furia [71] stated that this focus on core functionality makes Rosetta Code a suitable benchmark in terms of quality, performance, and reliability of these essential components, and by proxy, for evaluating LLMs.

##### _3.8. Translation metrics_ 

To evaluate the code translations quantitatively, the study uses three metrics, Code Extraction Rate (CER), Code Correctness Rate (CCR) and Pass@K. CER represents how successfully a piece of code can be extracted from the response generated by LLM for a particular translation task. CCR captures the “correctness” of code extracted from the LLM and reflects the reliability of the LLM. CCR does so by checking whether the generated code compiles successfully (C, C++, Go, Java, Rust) or has correct syntax (Python, JavaScript), depending on the programming language. 

Pass@K is a critical evaluation metric for quantitatively assessing the functional correctness of Large Language Models in code generation tasks, where K represents the number of code samples generated per programming problem. The metric measures the probability that at least one of the top K generated code samples for a given problem successfully passes all associated unit tests, fundamentally differing from traditional text-similarity metrics by focusing on executable correctness rather than syntactic resemblance to reference solutions. Originally introduced in the evaluation of OpenAI’s Codex model by Chen et al. [22], Pass@K demonstrated that repeated sampling significantly improves problem-solving capability, with Codex achieving 28.8 % success on Pass@1 but 70.2 % success on Pass@100 for the HumanEval benchmark. The metric addresses limitations of binary assessment approaches by enabling evaluation across multiple attempts, though recent research by Yeo et al. [23] has highlighted the need for more granular accuracy measures beyond the all-or-nothing nature of traditional Pass@K implementations.

Common natural language translation metrics include BLEU by Papineni et al. Hence, for this study, metrics like CodeBLEU, chrF, BLEU-METEOR, ROUGE and WER are used to evaluate the translations produced by the LLMs for all the tasks and translation combinations. 

##### _3.9. Interpretability metrics_ 

An LLM’s output is fundamentally a probability distribution over its token vocabulary at each step. This process can be summarized as, 

1. For each next token that the LLM predicts, the model calculates probabilities for every token in its vocabulary (which can be 10 s or 100 s of thousands of tokens). 

2. For example, if predicting the next word after “The cat sat on the ___”, it might output: “‘ mat: 0.35 floor: 0.25 chair: 0.20 table: 0.15 [other tokens]: 0.05 

   - ”’ 

3. When generating text, the model typically samples from this distribution (rather than always picking the highest probability token) to create more natural and varied outputs. The sampling can be adjusted via temperature and other parameters. 

4. This process repeats for each new output token, with the previous tokens (both from the input prompt and previously predicted tokens) providing context that influences the next probability distribution. 

Thus, changing any token in the input prompt will change the probability distribution of the output token. So, given a sequence of output 

tokens, the LLM outputs will provide a probability distribution over the entire LLM token vocabulary for each output token. 

Perturbation techniques to interpret LLMs use the fact that the output token probability distribution is conditioned on the input tokens and compute the change in the output token probability distribution as the tokens in the input are systematically augmented. A token or a sequence of tokens is removed from the input, and the probability distribution of each token in the output is compared to the corresponding probability distribution of the token in the original output, and this change in probability is quantified as an interpretability score using various methods. This results in an interpretability score being assigned to each token or sequence of tokens in the input that is perturbed, and each token in the original sequence of output tokens. 

Feature ablation is one way of calculating this interpretability score by taking the difference in the log probabilities at each token position of the original output. Similarly, Shapley values also calculate this interpretability score by considering other factors like efficiency, symmetry, and linearity along with the difference in the log probabilities at each token position of the original output. 

##### _3.9.1. Feature ablation_ 

As stated by Merrick [24], feature ablation is a technique used in machine learning to understand the importance of different features in a model. In the case of Large Language Models specialized for Code generation, a feature can be considered as a set of tokens in the input. Each line of the input source code is considered a feature for the LLM in code conversion experiments. It involves removing or “ablating” features one at a time and observing the impact on the model’s performance, and by comparing the performance with and without a particular feature, it provides an idea of how important that feature is for the model’s predictions. Performance in the code conversion experiments is the log probability of the output tokens generated by the LLM for the input in question. 

##### _3.9.2. Shapley value sampling_ 

Shapley Values are a concept from game theory that has been adopted in machine learning to explain the contributions of individual features to a model’s prediction. They offer a way to understand how much each feature contributes, positively or negatively, to the final output, providing valuable insights into model behavior. Shapley Value Sampling by Mitchell et al. [79] involves calculating Shapley Values for a subset of possible feature combinations instead of exhaustively evaluating all of them. This significantly reduces computational costs while still providing a good approximation of feature contributions. Like Feature Ablation, each line of the input source code is considered as a feature for the LLM, and the Shapley Values calculated reflect the importance of that line with respect to the output tokens. 

##### _3.10. LLM explainability_ 

Even though LLMs tend to be black boxes in the sense that it is difficult to accurately estimate why and how the LLMs generated the response, interpretability metrics like Feature Ablation and Shapley Value Sampling provide an insight into the importance of input features with respect to the LLM output. Another way to obtain insights into how an instruction-tuned LLM thinks is to ask follow-up prompts that instruct the LLM to explain its prior response. In the case of this study, when the LLM is provided with a compiler error message or syntax check issue for the translated code it generated, it is often helpful if the LLM can explain the mistake it made and then fix it. This improves the reliability of an LLM, where it can not only solve problems in zero-shot, but also improve upon its mistakes when prompted with the errors in the code.

##### **6. RQ2 - what do the findings from the explanations reveal about the reliability and accuracy of the code conversion task accomplished by the foundation models?** 

##### _6.1. Investigation strategy_ 

This part of the study follows a two-step process for each translation task from Rosetta Code. First, the LLM attempts to translate 

code from one programming language to another, whose process and results have been discussed in Sections 4.4 and 4.5. Then, after receiving compiler error and syntax check error feedback, the LLM gets a chance to explain and correct its mistakes. This closely follows the Chain-of-Thought reasoning process to ensure the LLMs don’t hallucinate, since the study prompts the LLM to first explain the error and then fix it. 

This also mirrors the real-world process of code development in software engineering, where programmers often iterate based on compiler and syntax checker tool feedback. If the LLM made a mistake in the code it generated, then it is prompted again using the prompt template shared in Section 3.7. This prompt is tailored to provide the LLM with the compiler error or the syntax checker error and asks the LLM to explain the error and then fix it. Rather than just testing one-shot translation capability, this experiment examines how LLMs handle errors and improve their

_Neurocomputing 656 (2025) 131461_ 

_V.S. Pendyala and N.B. Thakur_ 

output. The feedback loop incorporated as part of this study provides interesting insights into the reliability and accuracy of the LLM. 

- It tests the LLM’s understanding of both the source and target languages. To fix an error, the model must understand not just what the original code does, but also why its translation failed to achieve the same result. 

- It evaluates the LLM’s ability to reason about code at a meta level. The LLM must understand compiler error messages (which can be quite cryptic) and relate them back to both the original code’s intent and its attempted translation. 

- It simulates a realistic development scenario where initial code often needs refinement based on compiler feedback. 

Since the LLMs in this study are tuned to follow instructions, they are provided with the entire context as a conversation. This means that the feedback/follow-up prompt contains the following sections. 

1. A system prompt 

2. A user-prompt – The original translation prompt which contains the task description, the source language, the target language, and the source code. 

3. An assistant prompt – The original response generated by the LLM, which includes the code the LLM generated. 

4. A user-prompt – If the code had an error, then this contains the error message along with instructions to explain and fix the error. 

The code generated by the LLM in response is then extracted, formatted, and checked for correctness using the same methods as described in Sections 4.2–4.5. The key metrics to assess here are the Code Correctness Rate (CCR) and Pass@1, since they summarize how often the LLM is generating syntactically correct code and how reliably can that code be executed without runtime errors. The expectation here is that the LLM should be able to parse the compiler or syntax checker errors and fix the errors. This should reflect directly in the increased CCR as well as Pass@1 and thus reflect the reliability and accuracy of the LLMs. 

The explanations and reasoning generated by the LLM in the process of writing the code were analyzed with respect to the target programming language. To do so, all the errors belonging to a particular language were combined and then tokenized into individual sentences. These sentences would contain the error description and would repeat across multiple examples. These sentences were then ranked using TF-IDF and helped in isolating the most important errors from the reasoning generated by the LLMs.

##### **7. Discussion** 

The first key finding from this study uncovers scale patterns in the Rosetta Code dataset. This study reveals that the Code Correctness Rate (CCR), a measure of how many translated code samples generated by the LLMs were syntactically correct and compiled correctly, scales with the number of parameters in the LLM, with larger models performing better on this measure compared to smaller models. A similar phenomenon was also observed in Pass@1 metric, which measures the functional

correctness of the generated code against a set of known test cases. This agrees with the results from Code Llama research by Rozière et al. [66], which also observed the same phenomenon on the HumanEval and MBPP datasets. The same phenomenon was also observed in the study by Macedo et al. [83] that evaluated the effect of output format on LLMs in code translations. In the study, this effect was observed on the Computational Efficiency (CA) metric for the WizardCoder family of models, while this effect was noted across all the models considered in the study. This can be explained by the availability of more parameters for the LLM to learn from a greater number of coding samples, resulting in a better CCR. 

Contrary to expectations, the second finding revealed that translation metrics like CodeBLEU, chrF, METEOR, BLEU, ROUGE-1, and WER are ineffective when it comes to evaluating machine translations on the same set of translation tasks and language pairs. This is in contrast to the results obtained on natural language translation tasks from the study by Huang and Liu [84]. The difference is expected, as a code translation dataset is being used in conjunction with LLMs fine-tuned for code generation. Instead, it was demonstrated that these metrics are more suitable for distinguishing between different translation tasks and are much more representative of the quality and quantity of reference translations available for each task. 

The next findings showed that the interpretability of source code lines with respect to the translated code decreased with increasing line number. This effect can be attributed to the fact that the original prompt contained the task description as well, so the LLM relies even less on the later source code lines to translate the code. The initial source code lines, along with the task description, provide enough context and signal for the LLM to make accurate translations even without the remaining lines. This phenomenon was observed across all the translation pairs and both Feature Ablation and Shapley Value Sampling methods. Based on prior research, this is the first time such interpretability studies have been conducted on coding LLMs for translation tasks. 

It was observed that the interaction pattern between source code lines and translated code lines is sparse and shows strong interpretability scores when they both represent the same programming concept, such as loops, expressions, and logical flow. This is expected behavior and closely resembles human translation efforts, since human translators will only change the section of translated code when the equivalent section is changed in the source code. It was also observed that the interpretability of translated code lines decreases as the line number increases with respect to the source code lines. The latter translated code lines depend less on the source code lines. This can be explained by the context window of LLMs, where the latter tokens from translated code would attend more to the previously generated tokens than the source code tokens, as the source code tokens move out of the context window. 

Based on the findings from RQ1, it is proposed that line-to-line interpretability heatmaps can serve as a valuable tool in guiding the output code towards a desired outcome. If the translated code contains logical errors, it is possible to analyze how the source code lines influence the code generated by the LLM and then adjust the source code lines with high interpretability scores to achieve a correct output. Consider the scenario where the source code is heavily commented. Since LLMs can understand natural language as well, it is possible that the translated code would attend more to the comments in a heavily commented source code. In such scenarios, the source code comments that have interpretability with the affected line could also be tweaked to direct the translated code towards a desired outcome, along with traditional prompt engineering techniques. 

The findings from RQ2 show that LLMs can not only translate code but also comprehend and explain their own errors and then correct those errors when presented with compiler or syntax checker feedback. A 10 % improvement on average was achieved across all the LLMs on the CCR metric when feedback from the compiler or syntax checker was provided to the LLM. An 8 % improvement on average was also achieved across 

all the LLMs on Pass@1 metric as well after the follow-up prompt. This result agrees with the observations from the BESTER study by Ref. [85], where they also saw improvement in the Pass@K metric for HumanEval and MBPP datasets when the LLM was systematically prompted with compiler feedback to output correct code. The study also revealed a similar phenomenon, though it differs from that study with the use of the Rosetta Code dataset for translation tasks across multiple open-source Code Generation LLMs. 

It was observed that the improvement in CCR and Pass@1 was dependent on the target programming language. JavaScript is often underrepresented in Machine Translation studies, and it was noted that code translated to JavaScript had a CCR greater than 0.90 even without feedback, increasing to 0.94 when feedback was provided by the Node compiler. Python, on the other hand had the highest Pass@1 score before any follow-up prompt across all the LLMs and tasks. This increase was the highest for Rust, which had the lowest CCR of 0.4 and Pass@1 of 0.3 before feedback, but improved massively to CCR of 0.61 and Pass@1 of 0.46 after Rust compiler feedback was given to the LLM in follow-up prompts. The BESTER study by Song et al. [85], which implemented a similar setup, didn’t provide any details about the sensitivity of this phenomenon on the target programming language. This suggests a form of systematic error correction capabilities in LLMs that enhances reliability in two ways. 

- It demonstrates that LLMs can leverage concrete feedback like compiler output to improve their responses, much like human programmers do when debugging code. 

- The consistency of improvement across multiple LLMs, programming languages, and code types indicates that this isn’t a random occurrence but rather a reliable pattern of behavior. 

The observation that statically-typed languages like Rust exhibit more substantial improvements through feedback loops, while dynamically-typed languages such as JavaScript achieve higher initial correctness rates, can be attributed to several factors inherent to their respective paradigms and how LLMs interact with them. Statically-typed languages enforce rigorous type checking at compile-time. Even minor type mismatches, incorrect function signatures, or misuse of data structures result in immediate compilation errors. When an LLM generates code for these languages, a single type error can cascade into multiple compiler errors, making the initial raw correctness rate lower. 

However, the explicit and specific nature of these compiler errors provides very clear, unambiguous feedback signals to the LLM. This precise error information allows the LLM to learn and correct fundamental structural and type-related issues more effectively during feedback sessions, leading to more significant jumps in correctness. The LLM essentially gets highly informative hints on exactly where and how it deviated from the language’s strict rules. 

On the other hand, dynamically-typed languages defer type checking to runtime. Syntactic validity often doesn’t guarantee functional correctness, as type errors or attribute errors may only manifest during execution. This leniency means an LLM can generate syntactically valid, but semantically incorrect, code that appears correct initially but fails later. Consequently, their raw correctness rates tend to be higher for simple syntactic checks. However, the feedback from runtime errors in these languages can be less precise regarding the root cause of the issue, especially for complex logical errors or subtle type interactions that are only discovered during execution. The feedback loop might thus offer less direct guidance for fundamental structural corrections, leading to smaller relative improvements. 

Statically-typed languages often require more explicit declarations such as variable types, function return types, and error handling mechanisms. This explicitness provides more surface area for the LLM to learn the correct patterns and relationships. When an LLM gets it wrong, the error messages pinpoint these explicit elements. Dynamically-typed languages rely more on inference and convention, leading to more concise 

18 

_Neurocomputing 656 (2025) 131461_

code. This brevity can mean fewer explicit signals for the LLM to latch onto for self-correction when errors occur. 

The improvement through feedback can also be explained based on error propagation and debugging complexity. In statically-typed languages, errors often stop the compilation process early, preventing deeper, more complex runtime issues. Fixing one compilation error can resolve several downstream issues. In dynamically-typed languages, an error might only appear deep within a program’s execution flow, far from its original cause, making feedback less effective in pinpointing the core issue for an LLM to correct iteratively. 

This differential impact of feedback highlights that while LLMs are proficient at generating syntactically plausible code for dynamic languages, the structured and explicit nature of static type systems, coupled with their precise compile-time error feedback, provides a more effective learning signal for LLMs, enabling more substantial and systematic improvements through iterative feedback mechanisms. This suggests that for applications requiring high reliability and performance, guiding LLMs with environments that provide strong, immediate, and explicit feedback, such as those found in statically-typed languages, can be particularly beneficial. 

As part of RQ2, it was observed that LLMs can clearly parse the output from code compilers and syntax checker tools, and they can describe those errors in natural language. The last finding suggests LLMs can accurately identify language-specific errors and describe a fix for them in natural language. This is significant because it shows that LLMs have a degree of metacognition – they can analyze their own output, understand what went wrong, articulate their reasoning process, and fix the errors. This transparency in decision-making is crucial for explainability, as it helps in understanding how the LLM arrives at its solutions rather than treating it as a black box. 

Rosetta Code tasks, though not full applications, effectively assess an LLM’s proficiency in handling fundamental linguistic, algorithmic, and translational challenges in real-world software development. Even if the tasks are smaller, the types of errors LLMs make are representative of the challenges developers face in more complex scenarios. Understanding these patterns in a controlled environment is crucial before scaling up. In real applications, developers constantly translate user stories, design documents, and architectural specifications into code. While our prompts are concise, they capture the essence of defining a problem and expecting a functional solution. 

Many Rosetta Code tasks are foundational components of virtually every real-world application, from backend services to data processing pipelines and user interfaces. The ability of an LLM to correctly generate or translate these core components directly reflects its utility in building larger systems. The insights gained from the controlled experiments detailed in this study are directly transferable to understanding LLM behavior in more complex, practical coding scenarios. 

##### **8. Limitations** 

The most important limitations of this study were the selection of only seven programming languages, 15 LLMs, and the Rosetta Code dataset. The programming languages were shortlisted based on their popularity in Agarwal et al. [86], ultimately resulting in a selection of C, C++, Go, Java, JavaScript, Python, and Rust, as this combination provided a representative mix of procedural versus object-oriented languages, scripting versus compiled languages, and systems versus application programming languages. The selection of 15 LLMs was guided by various factors. Initially, the LLMs that were available on both the Hugging Face and Ollama platforms were identified. Subsequently, models that had been fine-tuned for code generation and optimized for adherence to instructions were selected. These models were further filtered to ensure they would run on most consumer-grade devices (1–10 billion active parameters). 

The dataset, Rosetta Code, though not as widely used as CodeNet, contained implementations for many programming tasks in a much 

wider variety of languages. From the entire dataset, only seventeen tasks were selected to represent the core concepts of programming languages, specifically bitwise, integer, string, and file operations. The dataset was specifically chosen because it allowed for direct, homogeneous comparisons of LLM performance across different languages and task types for the same underlying problem. This control is often difficult to maintain in more unconstrained, large-scale datasets. 

It also allowed for focused interpretability analysis. The structured nature of Rosetta Code tasks facilitated a more precise application of interpretability techniques, enabling us to pinpoint the influence of specific prompt elements without the overwhelming noise present in highly varied real-world code. Constructing and rigorously evaluating the output of LLMs on significantly larger, more diverse datasets, especially while also conducting in-depth interpretability analysis, would necessitate a substantially larger research effort, exceeding the current project scope and computational resources. Similarly, the experimental setup was focused on LLMs that can be deployed readily on consumer laptops, so that experiments are more accessible and easily reproducible. Hence, the focus is on 4-bit quantization and a limit on the number of parameters. Using larger models would require more resources. The exploration of larger, more complex datasets and larger models is therefore a direction for future work. 

Another limitation in any study of LLMs has been their nondeterministic behavior. To mitigate this, the LLMs were queried using a constant seed of 42, which is a commonly used seed whose origins were popularized by Adams [5]. The study adhered to the chat templates provided for each LLM to ensure clarity in the adherence to instructional prompts. Additionally, a limitation of this study was the lack of test cases to support each translation task, which could have aided in analyzing the performance of code generated by LLMs at runtime. However, this was not a concern since the primary aim of this research was to evaluate the interpretability and explainability of code generation LLMs. 

The metrics used for this study, such as BLEU and METEOR, primarily assess syntactic and stylistic quality and not direct functional correctness. Functional correctness evaluation frameworks can be more challenging to standardize across diverse languages and tasks. Metrics like BLEU and METEOR were selected to provide quantitative measures of code similarity and structure adherence, which are crucial precursors to functional correctness. While not direct measures of functional correctness, a high score in these metrics implies a greater likelihood of correct syntax and structure, making the code more amenable to debugging and potential functional validation. 

Therefore, integration of automated functional tests is the next step for a more comprehensive evaluation in future studies. This becomes particularly pertinent as dedicated testing frameworks for LLMgenerated code mature across diverse programming languages and problem types. 

##### **9. Conclusion** 

The use of Large Language Models (LLMs) in software engineering for code generation and translation has become increasingly common; however, it remains critical to ensure both their reliability and the interpretability of the code they generate. This study reveals that larger models tend to achieve higher Code Correctness Rates (CCR) on the Rosetta Code dataset, indicating that the ability to generate syntactically correct code scales with the size of the models. This study also suggests that the effectiveness of translation metrics like CodeBLEU, chrF, METEOR, and BLEU is not effective in differentiating between the quality of code generated by various LLMs. Therefore, the application of interpretability methods such as Feature Ablation and Shapley Value Sampling is crucial for gaining insights into how LLMs process code in translation tasks. 

Interpretability methods discussed in this study show that the initial source code lines, along with the task description, exert the most significant influence on the translated code. These initial lines provide

sufficient context for the LLMs to generate accurate translations, with later lines exhibiting less impact. Furthermore, the interpretability of translated code lines relative to the source code decreases with the line number, indicating that later lines in the translated code are conditioned more on earlier translated lines than on the source code lines. These interpretability techniques are useful for understanding how the model generates code and can potentially be used to guide the LLM towards a desired result. 

Moreover, when LLM generated code produces errors on execution, LLMs can parse error feedback from compilers and checkers and articulate the errors in natural language. Analysis of these explanations and corrections provided by the LLM uncovers common language-specific errors, such as missing header files in C or absent semicolons in Java. Prompt engineering also plays a critical role in obtaining usable code from LLMs. Techniques such as iterative prompting improve the quality of code generated by LLMs. By providing additional context in the follow-up prompts, including error messages from the compiler, this study demonstrates that LLMs can often correct their errors and produce enhanced code within the same conversation context. The consistency of improvement in the follow-up prompts across multiple LLMs, programming languages, and tasks from the Rosetta Code dataset suggests that this is a reliable behavioral pattern, thereby supplementing prior research in this area. 

Further work is still required in the direction of interpretability and reliability of LLMs in Code Translation tasks. Rosetta Code is a vast repository of coding tasks that have been implemented in multiple programming languages, of which only a subset of tasks and languages is included in this study. Methods like feature ablation and Shapley value sampling are only some of the ways in which interpretability can be measured, and further work is required to extrapolate the results from this study across other interpretability methods.
