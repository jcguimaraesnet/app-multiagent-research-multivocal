##### **1 . Motivation and significance** 

Large language models (LLMs) have become a key component of contemporary software engineering, particularly in tasks such as code generation, refactoring, and program analysis [1–3]. Their effectiveness, however, strongly depends on access to accurate and up-to-date infor­ mation about the software project under development. Even advanced models trained on large-scale code corpora cannot reliably perform pro­ gramming tasks without precise knowledge of the constantly evolving project codebase. 

The limited access to up-to-date information has driven the develop­ ment of the Retrieval-Augmented Generation (RAG) architecture, whose goal is to enrich queries with external, task-specific context prior to in­ ference [4,5]. By supplementing parametric knowledge with retrieved information, RAG aims to improve accuracy and relevance.

in a vector database. At query time, the user input is embedded and compared to stored vectors using similarity measures such as cosine similarity. The most relevant fragments are retrieved and appended to the prompt, enabling the LLM to generate responses grounded in external context (Fig. 1). Despite their popularity, traditional, naive RAG approaches exhibit significant limitations in programming tasks. Embedding-based retrieval is primarily optimized for natural language and does not adequately capture the semantics of source code [6–10]. Common chunking strategies may split semantically coherent program structures, obscuring relationships such as inheritance hierarchies, or control flow. As a result, probabilistic retrieval mechanisms introduce uncertainty into context construction, which is particularly problematic for deterministic and structurally constrained artifacts like source code [11,12]. 

##### _Enhanced RAG architecture_ 

To address these issues, a variety of Enhanced RAG architec­ tures have been proposed [3,5,13]. These approaches extend the basic retrieval–generation pipeline with techniques such as improved ranking, context compression, tighter coupling between retriever and genera­ tor, or syntax-aware chunking based on Abstract Syntax Trees [14]. Examples include systems such as REPLUG [15], Atlas [16], CRAG [17], and R<sup>2</sup> AG [18]. While these methods often improve relevance and coherence, they continue to rely on vector similarity search, without incorporating autonomous planning or iterative flow. 



**Fig. 2.** Conceptual overview of Agentic RAG with an agent-driven retrieval loop. 

##### _Agentic RAG architecture_ 

Agentic RAG represents a further evolution of the RAG architecture. In Agentic RAG, retrieval is embedded within an iterative, goal-driven control loop managed by an autonomous agent [19–23]. Instead of a sin­ gle retrieval step, the agent performs explicit planning, adaptive query reformulation, and multi-step reasoning across heterogeneous knowl­ edge sources (Fig. 2). It may selectively invoke retrieval operations, external tools, and execution feedback [24]. In software engineering contexts, this enables complex workflows involving repositories, doc­ umentation, tests, and APIs, supporting tasks such as repository-level code generation, debugging, refactoring, and security analysis. Although Agentic RAG increases computational and architectural complexity, it offers greater adaptability and reasoning depth for non-trivial program­ ming tasks [2,25].

##### _The RAGdeterm approach_ 

Replacing naive RAG with Enhanced or Agentic RAG generally im­ proves contextual relevance but comes with the cost of increased system complexity. Moreover, these approaches still depend on probabilistic re­ trieval, which can limit reliability, transparency, and reproducibility, properties that are particularly important in software engineering. 

RAGdeterm, presented in this paper, introduces an alternative solu­ tion that addresses these limitations while retaining the simplicity of a single-step retrieval pipeline. Instead of embedding-based similarity search, RAGdeterm employs deterministic prompt enrichment grounded in an explicit relational representation of code. Code elements and their relationships are modeled in a relational database, enabling predictable and reproducible retrieval of all relevant context elements (Fig. 3). This allows for the precise construction of queries before sending them to the LLM.

RAGdeterm represents an approach to retrieval-augmented code generation, combining established software engineering principles, such as relational modeling and deterministic querying, with AI-assisted pro­ gramming workflows. This approach suggests a promising direction for future research on hybrid systems that integrate classical code analysis techniques with LLM-based generation [9,26].

By replacing embedding-based retrieval with explicit relational modeling, RAGdeterm aligns conceptually with traditional program analysis and model-based engineering approaches, while remaining op­ erationally compatible with modern LLM-based generators. As such, RAGdeterm may be considered a viable alternative to enhanced RAG systems in domains where reliability, transparency, and repeatability are critical.

RAGdeterm is a solution composed of a set of interrelated compo­ nents, including projects that implement core functionalities as well as supporting applications. Its primary goal is to provide a comprehensive mechanism for enriching prompts with the contextual information necessary for the correct execution of programming tasks.

##### _RAGdeterm database_ 

Information about code is stored in a relational PostgreSQL database (Fig. 5). All types, primarily classes, but also interfaces, enums, and nested classes, are stored in the _klazz_ table, which also contains the source code of each type. The remaining tables are used to record relationships between the types. The _inheritance_ table captures inheri­ tance relationships. Structural relationships, understood as connections established through field types, are stored in the _structure_ table. The _cooperation_ table records associations resulting from class method pa­ rameters and return types. Practical experience has shown that these four tables are sufficient to retrieve the context of a class, understood as its surrounding environment defined by the set of all types related to it.

##### _2.2 . Software functionalities_ 

The objective of RAGdeterm is to enrich prompts prior to their sub­ mission to an LLM. Specifically, this enhancement involves supplement­ ing prompts with the current source code, or with information reflecting the current state of the source code, in all situations where such context is required. In most cases, this process entails the generation of new code, for which access to the up-to-date codebase constitutes a necessary in­ put to ensure correct operation output. The mechanism presented here relies on the definition of a dedicated query language, with expressions embedded directly within prompts and evaluated during the prompt pre­ processing phase. Each embedded query is subsequently transformed into a value obtained by invoking the corresponding function with a given set of parameters. ##### _Populating the database with information about code_ 

The operating procedure of the _loader-java_ module (Fig. 6) is iden­ tical regardless of whether it is invoked from the command line or used as a Java library. The process is supervised by the _Loader_ class. The content of the specified JAR file is loaded and inspected ( _LoaderJar.exec()_ ), and all contained types (classes, interfaces, etc.) are iterated over ( _LoadKlazzDepot.loadJar()_ ). The information about them is then saved in the appropriate data structure in memory. Subsequently, for each class (type), a sequence of operations is performed to identify relationships that are recorded in the _inheritance_ , _structure_ , and _coop­ eration_ tables. The main type information is stored in the _klazz_ table ( _SaveKlazzDepot.save()_ ), where the source code may also be optionally persisted. ##### **3 . Illustrative examples** 

Below are several examples illustrating the functionalities of RAGdeterm. All of them were implemented using the sample libraries available within the RAGdeterm project as shown in Fig. 4, in the section labeled _sample_ . 

A detailed description and the complete results of the conducted experiments are provided in the online documentation accompanying the project. It should be emphasized that the evaluation focused on the 

quality of the generated context used to enrich prompts, rather than on the quality of the final responses produced by the LLM system. The eval­ uation of specific solutions was based on comparing the code returned by RAG with the code representing the correct context, which is under­ stood as a complete set of types associated with a specific class. The metrics used for evaluation are _precision_ , _recall_ , and _F1 score_ . In addi­ tion, the performance of code analysis and loading operations into the RAGdeterm database, as well as response times, including under high system load, were measured. 

##### _Person-lib_ 

The _person-lib_ library contains a set of Java classes shown in Fig. 9. It is assumed that these are domain classes used in the backend part of the application. The task is to create, for each of these classes, a corre­ sponding Data Transfer Object (DTO) class in Java and its counterpart in GraphQL. This operation is performed from within the _ai-client-determ_ application. The query is shown in Fig. 10. 

The query employs the _StruCoopContext_ function, which constructs the complete context of the class _pl.org.opi.person.PersonSection_ , that is, all classes associated with it. The function returns the source code of these classes ( _SOURCE_CODE_ option). This code is then injected into the prompt before it is sent to the LLM. 

The _person-lib_ classes should be loaded into the RAGdeterm database; one possible approach is to use the _loader-java_ tool executed from the command line (Listing 3). 

##### _Vehicle-lib_ 

The _vehicle-lib_ library comprises a set of classes defining a family of vehicles ( _AirVehicle_ , _WaterVehicle_ , and _LandVehicle_ ). Within this family, various types of land vehicles are distinguished, including cars, body types, as well as car components and subsystems. The model is character­ ized by numerous inheritance and structural relationships. An example task is to identify which classes, fields, and methods have not yet been documented. The operation is to be performed in the context of the _Hatchback_ class. For this purpose, the prompt shown in Fig. 11 was prepared. 

The correct result of the _StruCoopContext_ function consists of 21 classes. The high complexity of the relationships between them made it possible to compare the RAGdeterm approach (which is 100% effective) with a solution based on traditional RAG. The conclusions drawn from this comparison are discussed in the description of the _ai-client-naive_ application.

##### _RAGdeterm performance_ 

In RAG solutions, which are used to enrich prompts sent to LLM models, in addition to the quality of the prompt itself, which is crucial, performance is also an important parameter. This is confirmed by the example of _agentic-llm_ , where response times of around 1500 seconds seem to rule out the practical application of such a solution. 

Therefore, RAGdeterm performance tests were conducted. Performance issues are important in two aspects of RAGdeterm operation. The first is loading project code into a relational database. In order for the database to contain up-to-date data, this operation should be performed frequently, preferably after each change in the project (after each commit). The second aspect is the enrichment of prompts itself, i.e., the execution time of embedded RAGdeterm functions. 

Code loading performance experiments were conducted using test libraries. The loading test was also performed on the Java JDK. 

The loading times of test libraries show that the RAGdeterm database can be updated after each change in the code, and any delays for medium-sized projects can be a few or several seconds (Table 6).

##### **4 . Impact** 

When there is a need to employ LLMs to support software develop­ ment tasks that require access to constantly evolving code, a natural solution is the use of RAG. However, traditional RAG does not perform well in this role. Improved results can be achieved by solutions classified as Enhanced RAG or Agentic RAG, yet their implementation is difficult and complex [19,27]. In such cases, RAGdeterm, which is effective and easy to use, may be an alternative. 

RAGdeterm is a RAG-class solution in which classical, imperative programming techniques are used to enrich queries with the required context in the form of relevant source code elements. The process is de­ terministic and efficient. Owing to the maturity of relational database technologies, the limits on the number of projects loaded into the RAGdeterm database are very high, and the volume of stored data does not cause a decrease in performance. Furthermore, the parallel execution of multiple queries does not pose a problem (test results are provided in the online documentation accompanying the project). The main lim­ itation of RAGdeterm is that it is a solution dedicated to software

development tasks and operates on object-oriented language code. At present, the Java language is supported, which follows from the tech­ nologies used within the organization where RAGdeterm was developed. Work is currently underway to add support for TypeScript and C#. It is worth noting, however, that according to [1], as many as 57% of the sur­ veyed use cases of code generation techniques concerned Java, whereas C# accounted for only 3%. The second limitation concerns the embed­ ded query language. Functions defined within this language must be syntactically correct. In practice, however, complete prompts are not written manually; instead, they are based on templates that, together with the embedded functions, are constructed automatically as part of a broader processing pipeline. It should also be noted that system main­ tenance, administrative activities, and supervision of database content constitute an additional operational effort and entail further costs. 

The described technique is employed at the National Information Processing Institute (OPI PIB) in Warsaw as a core component of the automated generation of selected software artifacts. It operates on code drawn from multiple cooperating projects and, through RAGdeterm, enables the construction of accurate cross-project contexts more effec­ tively than conventional developer assistants. This capability supports batch-oriented solutions, such as software factories, that extend beyond the scope of individual projects. The RAGdeterm relational database can be viewed as an indexed corporate monorepo, allowing efficient retrieval of relevant code fragments without exhaustive codebase traver­ sal. Embedded functions are straightforward to implement and extend, and the database can be readily synchronized with code changes, en­ suring that queries consistently return up-to-date results (loading and update speed test results can be found in the online project documenta­ tion). 

##### **5 . Conclusions** 

RAGdeterm is a tool intended to support LLMs in tasks that re­ quire precise and up-to-date access to source code. Beyond this specific application, however, the underlying concept of integrating large lan­ guage models with solutions based on a simple, well-defined processing pipeline and classical technologies, such as relational databases, rep­ resents a broader research direction worthy of systematic investigation. The design of hybrid systems that combine the generative capabilities of LLMs with deterministic, mature data management mechanisms enables the effective enrichment of queries with current, externally maintained data sources [28]. 

From a scientific and engineering perspective, these hybrid archi­ tectures offer an opportunity to explore alternative retrieval-augmented generation patterns that emphasize predictability, performance, and ease of implementation. By relying on simple pipelines and wellunderstood technologies, they lower the barrier to adoption while remaining robust and scalable in industrial settings. Consequently, sys­ tems of this class can serve as a practical and complementary extension to Enhanced RAG and Agentic RAG approaches, providing competitive effectiveness while preserving simplicity, reproducibility, and ease of use.
