##### **1. Introduction** 

Technological advances have reshaped the entire Software Engineering (SE) status quo in recent years, lowering the digital literacy and expertise barrier for people and enabling them to transform their ideas into functional applications. Technologies such as Model-driven Engineering and low-code platforms [1] attempt to include more nonexperts in the application development life cycle. They offer user empowerment in terms of a more intuitive SE medium, resulting in a more rapid development process and boosting application productivity. At the same time, Large Language Models (LLMs) have disrupted the entire Human–Computer Interface domain, simplifying how users interact with systems in various cases, even for SE tasks [2,3]. 

LLMs are highly successful models that can assist users via natural, contextual, multi-turn conversations, automating repetitive and highly complex procedures. In the SE domain, developers increasingly adopt a practice often referred to as _vibe coding_ , a concept introduced by Andrej Karpathy in February 2025 through X (former Twitter), where software is developed by interacting with LLMs using natural language prompts instead of writing code manually. This AI-driven approach enables a variety of tasks, such as code generation, code summarization, vulnerability detection, and repair [3,4], thereby streamlining parts of the development process. Therefore, the focus shifts from traditional coding to proper interaction with an LLM. This is largely possible because LLMs are highly familiar with General-purpose programming

Languages (GPLs) like Python and JavaScript, having been trained on massive corpora of code from public repositories. However, despite their capabilities, LLM-based software development has notable limitations. Their code generation ability, one of the most important SE tasks, is still error-prone [4], and they often produce incorrect or suboptimal code and cannot self-reflect, validate, or repair their output, limiting their role to that of assistants rather than autonomous developers. Furthermore, LLMs cannot yet support the entire system development life cycle and create complex, end-to-end, production-ready systems and applications. Instead, they should be used in the context of larger AI [5] or SE [6] systems. Another important limitation is that LLM knowledge is limited due to its outdated training set [4]. As a result, they cannot automatically include any new framework updates and keep up with new versions and features without retraining, hence, developers must manually embed any new functionality that requires new framework knowledge. Still, LLMs are ideal as _copilots_ , i.e., tools that recommend code snippets while developers remain in control, reviewing, editing, and integrating the output themselves. Although LLMs provide an intuitive and conversational interface to code, they fail to produce complete and consistently correct software systems independently. Moreover, effective use still requires significant understanding of both the application domain and the underlying technologies. 

On the other hand, a different, better-established SE approach follows low-code principles and the Model-driven Engineering paradigm, which uses _meta-models_ , these being high-level representations of domain-specific concepts and their relationships. These low-code platforms have been around for several years, hiding software complexity and offering an abstract and more intuitive medium for the entire application life cycle development. They usually employ Domainspecific Languages (DSLs) with which users can describe the desired software outcomes, by creating a _model_ following the DSL syntax and terminology. Thus, their focus shifts from software development traits and framework-specific implementation to application design and modeling, requiring significantly less domain-specific and technical knowledge and expertise. 

In this context, several tasks such as model validation, testing, and code generation are automated, ensuring accuracy and consistency while minimizing time-consuming debugging and iteration. However, despite these strengths, DSL-based development comes with a critical limitation: a steep learning curve. Each DSL introduces its own syntax, semantics and characteristics, which are often unintuitive for newcomers. This complexity poses a significant barrier to adoption, especially for non-technical users or so-called citizen developers, who may lack the specialized knowledge required to navigate DSL grammar and tools effectively [7–10]. Especially when the domain is broad and complex, DSLs often become difficult to use. As a result, DSLbased platforms tend to be accessible primarily to programming or DSL experts, rather than the broader audience that low-code development aims to empower. 

We argue that, as we speak, both LLM-based and DSL-based approaches, for their own reasons, fail to reduce the domain and programming expertise required to involve non-experts in the software development process. However, by strategically combining their strengths, we can surpass these limitations and create a more intuitive and accessible software development paradigm. Particularly, LLMs excel as conversational interfaces, and given a set of precise instructions, a subdomain called _prompt engineering_ , they can be guided to operate within specific contexts, such as a given DSL. Their strong understanding of programming concepts allows them to learn domain-specific patterns quickly, enabling them to generate accurate DSL models more efficiently and precisely than many human developers. At the same time, low-code platforms provide robust DSL model validation mechanisms that ensure syntactic and semantic correctness of a produced model, minimizing the debugging time. Moreover, valid models can be automatically transformed into fully functional software applications following the pre-defined transformation rules and processes. By integrating LLMs as 

natural language interfaces for DSL modeling, we harness the flexibility and accessibility of LLMs combined with the efficiency of DSL-based low-code platforms. This fusion enables developers to build valid software models through simple, guided conversations, lowering the entry barrier while preserving reliability and structure. 

In this study, we present **_DSL Agent_** , an LLM-based conversational agent designed to support vibe coding, i.e., the practice of generating software through natural language interactions, within the context of DSL-based low-code development. Embedded into Locsys,<sup>1</sup> a lowcode SE platform, the DSL Agent enables developers to generate the desired DSL models through conversation, reducing the time and effort required and eliminating the need for deep familiarity with the DSL syntax. The agent is adaptable and can support any textual DSL via prompt engineering and in-context learning, using an automated pipeline that provides relevant information such as DSL concepts, grammar, and examples on the fly. It communicates with the Locsys platform, which comprises the rest of the low-code functionality, i.e., model transformations, model validation and generation, and application deployment, offering a complete LLM-based SE interface for application development and deployment. 

The proposed Agent is empirically validated in a large-scale 2-day workshop where more than 130 junior developers without any prior experience in the low-code domain or the DSLs’ application domains participated. The participants experimented with two textual DSLs of different complexity, namely _CodinTxt_ , a DSL for dashboard creation, and _dFlow_ [11], a DSL for developing task-based virtual assistants. The data collected shows that the _DSL Agent_ can generate DSL models more accurately, consistently, and efficiently than non-LLM-based lowcode application development. Particularly, it is far more efficient in producing valid DSL models than inexperienced developers, increasing overall productivity, regardless of DSL complexity, while requiring minimal domain knowledge or expertise. Finally, the _DSL Agent_ improves usability and user experience, resulting in a simpler and more friendly interface for low-code application development. 

The main research questions that we answer through our work are 

the following: 

- **RQ-1** Can we teach an LLM to create valid and correct DSL models? 

- **RQ-2** Can we accelerate the development of DSL-based applications? 

- **RQ-3** Can we offer a simpler and more intuitive interface for DSL model development? 

- **RQ-4** Do users need minimum domain knowledge to utilize our approach?

##### _2.1. DSL-driven and low-code development_ 

Domain-specific Languages (DSLs) and low-code development platforms represent two significant trends in modern software engineering

that enhance productivity and facilitate the development process. DSLs are tailored programming languages designed to address specific problems within a particular domain, while low-code platforms enable users to create applications with minimal coding effort. 

DSLs provide tailored notations and constructs that align closely with specific application domains, thereby improving productivity and facilitating better communication between developers and domain experts. The evolution of DSLs has been significantly influenced by the need to manage the complexity of modern software systems, as traditional General-purpose Languages (GPLs) often fall short in addressing domain-specific requirements [12]. The integration of DSLs with MDE practices has led to the development of tools that automate the generation of code and documentation, thereby streamlining the software development life cycle [13,14]. For instance, tools like CRUDyLeaf enable the rapid generation of REST APIs from entity CRUD operations, aligning with contemporary DevOps practices that emphasize on quick and reliable software delivery [15]. 

Usability remains a critical focus in the development of DSLs, as the effectiveness of these languages is heavily dependent on their ability to be understood and utilized by their intended users. Recent studies have proposed guidelines for improving user experience in industrial DSLs, emphasizing the importance of intuitive design and accessibility [16, 17]. Additionally, evaluating DSLs through frameworks that assess their usability has become a common practice, enabling developers to refine their languages based on user feedback and performance metrics [18]. The ongoing research into the interoperability of DSLs, particularly in the context of the Internet of Things (IoT), highlights the necessity for DSLs to work seamlessly across diverse platforms and technologies [12,19]. 

On the other hand, low-code development platforms (LCDPs) leverage MDE methodologies, treating models as first-class entities in the software development life cycle. This approach enhances productivity by allowing developers to create applications using high-level DSLs, visual programming interfaces, and pre-built components, significantly reducing the time required for traditional coding [20,21]. For instance, LCDPs enable users to design applications through drag-and-drop interfaces, which democratizes software development by empowering non-technical users, often referred to as ‘‘citizen developers’’, to participate actively in the application life cycle [22,23]. This shift not only accelerates development timelines, but also aligns with the growing demand for agile and responsive software solutions in various industries [24,25]. 

The comparative analysis of low-code frameworks reveals that they provide substantial advantages over traditional coding methods, particularly in terms of functionality and performance. For example, platforms like Salesforce and OutSystems have been noted for their robust capabilities in integrating with various data sources and services, thereby enhancing the overall application development experience [26]. Furthermore, using DSLs within these platforms allows for tailored solutions that cater specifically to the needs of particular domains, thus improving the relevance and effectiveness of the applications developed. 

However, despite the advantages, adopting low-code and DSLdriven approaches is challenging. Issues such as scalability, integration complexity, language extensibility, and the need for effective governance frameworks have been highlighted in recent studies [27]. Moreover, while low-code platforms aim to simplify the development process, there remains a risk of creating applications that lack the robustness and maintainability of traditionally coded solutions [28]. Additionally, DSLs introduce their unique syntax and characteristics that might confuse or frustrate developers who must first familiarize themselves with new language-specific terminology for each new DSL they want to exploit [7–9]. As organizations increasingly rely on these platforms, it is crucial to address these challenges to fully harness their potential in driving digital transformation. 

In summary, the proper integration of DLS-based development concepts within the framework of model-driven engineering presents a promising avenue for enhancing software development efficiency. By enabling a broader range of users to engage in application development, these platforms not only streamline processes but also foster innovation. However, ongoing research and practical insights are necessary to overcome the inherent challenges and ensure that the applications produced meet the required standards of quality and performance. 

##### _2.2. LLM-driven software development_ 

Large Language Models (LLMs) are currently employed in several SE tasks, as discussed in various surveys [3,4,6]. Code generation and program repair are the most common use cases, assisting in the development and maintenance processes, increasing development productivity and user satisfaction [29]. 

However, LLMs for SE have their inherent flaws, challenges, and limitations [4]. First, they frequently generate erroneous code segments or hallucinate, presenting seemingly correct answers that are actually wrong [6,30]. Some solutions include few-shot approaches where LLMs are provided with examples to improve the generation accuracy or finetuning for even better results [31]. Additionally, LLM-generated code may not follow the best coding practices, which can result in higher code complexity or security vulnerabilities [32–34]. Another important limitation is that LLMs do not have adaptive learning capabilities, since their knowledge relies on the data they have been trained on, making them unable to incorporate software updates, new features, or frameworks automatically. The latter happens frequently as open-source or domain-specific frameworks and systems are constantly released and updated. 

Another drawback is that LLMs tend to struggle when faced with complex problems or multiple conditions and requirements [4]. Many attempts to develop complex and autonomous systems follow an agentbased approach. LLM-based agents are autonomous systems that can reason, plan, and execute actions in a provided context. In the SE domain, several such systems are attempting to automate the entire SE and software development life cycle, from requirements elicitation to software development and testing [35–37], even though they cannot yet surpass the relevant LLM limitations and challenges [37]. Particularly, they often hallucinate and lack external knowledge retrieval connections, among others, resulting in inefficient independent operation and production of functional software applications. This makes the agentic approach redundant [38] as it introduces a rather complex architecture, without solving the primary issue of valid code generation. 

Consequently, there is a need to ensure the validity and soundness of the generated code while also enabling the integration of new features and functionality into the system’s knowledge. These can be achieved by separating the LLM-generated outcome from the endsystem code. Particularly, intermediate Domain-specific Languages can be employed [39], so that developers can describe their requests to LLMs, which can produce the DSL-based models, while the low-code platform will lead the validation of the models and the generation of the application code. This approach follows the suggestion of several other studies, mentioning that LLMs can be highly efficient if embedded in a larger SE system. 

##### _2.3. Conversational interfaces for DSL-based software development_ 

Developing Natural Language (NL) interfaces for describing DSL models is not new. Before LLMs were introduced, a few studies attempted to create such interfaces that used a set of rule-based, predefined interactions embedded in a chatbot, allowing users to describe, for example, domain-specific models [7,8] or smart contracts [9,40]. Users reported such interfaces as useful, even though the NL understanding and conversation capabilities were limited [9]. This was due

to the underlying technology and the pre-defined set of interactions, which either required all mandatory information in one user prompt using specific expression patterns [7], restricting users to describe one concept at a time [9], or producing several errors [9] with reporting or feedback [8]. 

Large Language Models can tackle most of these issues and enhance the overall performance and interaction. They have been employed in several cases, for example, for generating DSL-based diagrams [41], graphical games [42], and educational chemistry experiments through robotic movements [43]. The latter reported two aspects that are also important in our pipeline: (a) to produce outcomes consistent with the DSL rules, the LLM prompts must include extensive information about the DSL, and (b) enabling the interaction between the DSL model generator module, in this case, the LLM, and the model validator, or verifier, can also improve the overall performance. 

In another study, a DSL for interacting with Microsoft Office applications named ODSL (Office-DSL) was introduced [44]. ODSL is a high-level DSL with syntax similar to Python, leveraging APIs to perform actions within Office applications. The system employs an LLM and in-context learning, incorporating key information in the LLM prompt, such as the DSL description, syntax, and relevant examples, to generate the DSL-based program based on user input. This in-context learning is conducted in two phases. First, the user request is analyzed and mapped to a set of relevant ODSL entities, which are then used to filter the prompt aspects and retain only the necessary ones. Then, an entity-aware semantic search is conducted to retrieve examples containing those entities. This two-step approach is very promising and improves the pass rate of the generated code scripts by approximately 10%–15%. However, a significant limitation is the simplicity of ODSL. It generates only a few lines of code, unlike many more complex DSLs – especially external ones – that involve greater conceptual complexity and do not typically follow a Python-like syntax. Moreover, there is no interaction between the LLM module and the validator module, meaning there is no feedback loop to detect and automatically correct errors, which limits the system’s robustness in handling complex or erroneous cases. Additionally, the system lacks real-world evaluation and usability testing, as its validation was based on 197 manually created requests reflecting the supported functionality. Finally, it cannot be considered a fully functional interactive tool, as it is not integrated with an interactive system where users can refine their requests and conversationally develop DSL models. 

On the contrary, Netz et al. [45] presented an interactive system that allows domain experts to describe data structure models that are then utilized to create web applications automatically. Even though the presented DSLs are quite simple and resemble typical models of objectoriented programming, their methodology follows a human-in-the-loop approach, where the domain expert self-validates the LLM response and then requests the application generation, an approach also included in the Locsys platform. 

Finally, a different approach is grammar prompting [46]. This technique empowers LLMs with external knowledge and domain-specific constraints using the Backus–Naur Form (BNF) and improves the generated program accuracy across various DSLs. Through this method, LLMs learn from and reason with rules expressed by the provided grammar. However, this technique presents challenges, especially when the LLM must grasp the full DSL grammar with only a few examples, limiting its effectiveness in some cases. The generated code segments are often small, and the approach does not show significant improvements with already familiar languages, such as SQL or regex. Furthermore, as in the ODSL study, this method lacks extensive evaluation and integration with an interactive system, where users could receive feedback, refine their requests, or interact conversationally to develop their DSL models. 

##### _2.4. Proposed novelty_ 

This paper proposes _DSL Agent_ , a novel LLM-based conversational interface integrated within a low-code platform for DSL-based development. While LLMs have been applied in GPLs and, in limited cases, in simple DSLs, no prior work focuses on complex external DSLs, where their unique syntax and semantics require deeper contextual understanding. Our approach addresses this gap by showcasing how the DSL Agent generates DSL models with greater consistency and accuracy than citizen developers, even in highly complex domains, by efficiently leveraging prompt engineering and in-context learning. 

The agent significantly reduces the expertise required for DSL-based development by enabling users to describe their application processes and requirements in natural language, which are then processed and transformed into valid DSL models. Unlike traditional DSL approaches that demand specialized knowledge, the DSL Agent adapts to different DSLs with minimal reconfiguration, providing a scalable solution for a wide range of application domains and use cases. 

Additionally, the efficiency of the proposed approach is empirically evaluated at a large scale involving more than 130 participants, a considerably larger sample than most studies. This scale offers stronger evidence of the agent’s effectiveness in improving both development time and model accuracy compared to conventional DSL workflows. Based on the assessment performed in the context of this work, we highlight the potential of LLMs to transform DSL-based development, providing a more intuitive, adaptable, and efficient software development interface. 

##### **3. The proposed DSL Agent** 

In this section, the _DSL Agent_ is described in detail. Particularly, its internal components are presented, then the implemented conversation and agent flow process and interactions between its components are discussed, followed by the employed prompt engineering pipeline and the different prompts employed, both for DSL model requests and error fixing. 

##### _3.1. The DSL Agent internal components_ 

The proposed _DSL Agent_ consists of several key components, illustrated in Fig. 1. Core component of the agent is an LLM, which, combined with prompt engineering and in-context learning techniques, enables the efficient generation of DSL models that follow the language’s syntax and terminology. To ensure accuracy and robustness, the agent incorporates DSL-specific information, such as the DSL grammar and its constraints, alongside a validation mechanism that checks the syntactic correctness of the generated models. In addition to syntactic checks, the high-level DSL Agent design also includes a semantic validation component. This module is intended to assess the alignment between a user request and the generated DSL model, ensuring that the produced solution is not only well-formed but also contextually relevant. However, semantic validation is not essential for the DSL Agent’s core operation and, given its demanding nature, represents a complete study on its own. For this reason, it has not been implemented within the scope of this work. An error-fixing process further refines the output, correcting any inconsistencies or errors detected in a generated model. Although these components work closely with DSL-specific data and logic, the overall design remains modular and DSL-agnostic, allowing the agent to be adapted or extended to other DSLs as needed. All functionality is exposed via a REST API endpoint, facilitating integration with external systems and user interfaces. ##### _3.2. Conversation and agent flow_ 

The internal structure of the _DSL Agent_ enables users to interact with a low-code development environment using natural language. From the user’s perspective, the process begins after logging into the platform, selecting a particular DSL, and creating an empty model. The user can then select the _DSL Agent_ and describe the intended functionality of the DSL model through a message in natural language. 

The DSL Agent is embedded into the platform, which provides the user interface and manages all DSL-related operations, such as model validation. The platform sends the user message, along with other relevant parameters, to the agent’s REST API, which then triggers a multi-step processing pipeline. Internally, the agent constructs prompts using DSL-specific knowledge stored in a vector database, communicates with the employed LLM, and stores the full conversation context in a memory module for continuity in subsequent interactions. The exact interactions among these components, i.e., platform, API, vector database, LLM, and memory, are illustrated in the block diagram in Fig. 2 and further detailed in the sequence diagram in Fig. 3. 

When the platform sends a request to the agent’s API, it includes parameters such as the model ID and a reset flag. The agent uses this information to determine whether the user is initiating a new conversation and, if so, it creates a new system prompt; otherwise, it retrieves the existing conversation from memory. LLMs handle three types of prompts: system, user, and assistant prompts. The system prompt provides high-level instruction and context, informing the model about the DSL, its expected behavior, and the goals of the conversation, as described in Section 3.3. The user prompt consists of the user’s message, optionally augmented with additional instructions. Assistant prompts are the LLM’s previous responses. 

After constructing the system prompt or retrieving the previous conversation from memory, the LLM is requested to generate a new DSL model, and its response is returned to the platform. The platform then checks the generated model and, if valid, presents it to the user. In the case of an invalid model, there is an internal iterative validation error-fixing loop process leveraging the LLM to attempt to 

solve any conceptual or syntactic errors. As discussed in Section 3.4, several error-related information and instructions are sent to the LLM, requesting it to solve the error. For a better user experience, and to avoid infinite loops, the error-fixing loop terminates either (a) after three iterations, or (b) if the same validation error is produced in two sequential LLM calls. In this case, we acknowledge that the LLM has failed to fix the error. Finally, whether the model is valid or not, the generated output is returned to the user interface. 

The DSL Agent’s API endpoint that handles user interaction (from the platform UI) comprises the following parameters: 

- _language_ to distinguish the exact DSL the conversation is about and to retrieve relevant information from the local database 

- _model_id_ of the currently developing model 

- _first_prompt_ a boolean parameter used to reset older conversations of the particular model 

- _currentPrompt_ to include the exact user message 

- _current_model_ which consists of the (potentially) already existent DSL model in the platform 

The users can initiate the model generation by writing a small part of it and then prompting the agent. Similarly, they might alter a generated DSL model and then wish to continue their conversation with the agent. In those cases, the agent must incorporate this new version as its current model and any LLM generations after that must be made based on this. Hence, the platform sends the existing model to the API, which compares it with its internally stored version for changes and decides to use or ignore it. Finally, there are three parameters regarding the validation error fixing loop, namely (a) _validation_error_ that contains the error message produced from the DSL validator, (b) _error_line_ which is the line number where the error is, and (c) _erroneous_line_ , the text content of this exact line.

Another important aspect is the relevance of the examples, as LLM performance is sensitive to prompt examples [31]. Hence, the provided examples must contain all these concepts required to create the requested model and also be as relevant as possible. This is accomplished in an intermediate two-step process, namely a concept prediction and a semantic search process. 

First, the LLM is used as a zero-shot multi-class classifier to identify the necessary DSL concepts based on a user request. It is prompted with both the user input and specific instructions, which include the DSL’s name, a brief description of the DSL, a list of its concepts with explanations, and the mandatory concepts. The LLM then predicts the relevant concepts needed to build the requested model, which are subsequently used as filtering parameters in the following semantic search process. 

The semantic search process is implemented using the vector database, which can leverage embeddings of a sentence transformer [47], a small Language Model ideal for sentence-level tasks. These LMs convert each input text into a multi-dimensional vector known as _embedding_ that captures its semantic meaning. The search process compares the user’s input (transformed into embeddings) against precomputed embeddings representing DSL examples in a vector database.

As discussed earlier, an error-fixing loop is initiated when the LLM produces a syntactically invalid model. Then, the platform sends errorrelated information, specifically, the validation error message produced from the DSL validator, the number, and the text of the model line where the error happened. When these parameters are filled and an active conversation exists in the memory, the API process prompts the LLM to solve the aforementioned error. Several empirical tests were performed to determine what information must be included in this LLM prompt alongside the error-fixing instructions. Apart from an instructions-only prompt, attempts were made to include the exact validation error message (which was found to confuse the model), some parts, or even all the DSL documentation as written in its repository, but with poor results. At the same time, we empirically witnessed the agent make several mistakes, most of which fit into specific error categories. Hence, instead of prompting the LLM with DSL documentation, we documented and added the error categories in the initial system prompt. Then, we directly request the model to fix the error using the following user prompt: 

_‘‘There was a validation error in line {error_line}: {erroneous_line}. Consider the common mistakes’’_ .

##### _4.1. Locsys platform_ 

Locsys is a web platform that hosts heterogeneous DSLs in type (graphical/textual) and domain, allowing for the creation and management of DSL models. The novelty of Locsys is that it provides a common interface for all supported DSLs, offering standardized operators that can be applied in all models, namely validation, import, transformation, generation, and deployment. 

The _validation operator_ applies to all DSLs and is one of the most fundamental functions offered. When the validation operator is applied to a model, its correctness is checked with respect to the metamodel of the specific language, but also to its syntax. Successfully validating

_N. Malamas et al._ 

_Journal of Computer Languages 85 (2025) 101367_ 



**Fig. 4.** DSL Agent user interface.

The _import operator_ concerns the integration of a model of a DSL into another model of the same or another DSL. Integration is often necessary since some models (hence some DSLs) must necessarily include information from a previous model to make sense or perform validation. The _transformation operator_ refers to the second basic operation of the MDE methodology, in which the model in question is transformed into another model of another, related DSL. How models are transformed varies, most frequently using transformation rules that specify how elements of one model are transformed into elements of another. The _code generation_ operator converts a model into executable code, which may consist of one or more files. This operator is a version of the transformation operator since a transformation is also performed, not

##### _4.3. Language models and frameworks_ 

As discussed in the previous section, the DSL Agent integrates several software modules and models to deliver its functionality. This section outlines the components and models selected to implement the agent in the Locsys platform. The agent’s REST API is implemented using the FastAPI framework,<sup>4</sup> chosen for its efficiency and ease of integration with modern Python-based backends. For semantic retrieval, a Haystack instance<sup>5</sup> is deployed to manage a vector database containing all the DSL-related information and examples. To support conversational memory and short-term context persistence, a local instance of Redis<sup>6</sup> is employed. In terms of models, the agent leverages Claude-3 Sonnet [52], accessed via the AWS API, as the primary large language model (LLM), while _all-MiniLM-L6-v2_ , a lightweight sentence transformer, is used to compute dense vector embeddings of DSL examples for efficient similarity-based DSL example retrieval within the vector database.

far presented is that the complexity of the grammar has a significant impact on the accuracy of the LLM in constructing syntactically valid and reasonable models of Domain-specific Languages. In this study, we measured complexity solely in relation to the metamodel of each DSL, conducting experiments on two DSL cases: CodingTxt and dFlow, the latter of which exhibited twice the complexity of the former. Nevertheless, the usability and performance results obtained from the present study are encouraging and can be regarded as a crucial foundation for subsequent investigations into LLM-driven development, where more comprehensive complexity metrics of the DSL grammars will be taken into account. It can be proposed that by reducing the complexity of the DSL (analogous to GPL complexity), enhanced accuracy outcomes for the LLMs can be achieved while developing in DSLs. 

Another set of potential threats arises from the comparative evaluation design, which is a common challenge in similar studies. Particularly, it is inherently difficult to fully minimize the influence of factors, such as task similarity, transfer learning, and variations in motivation or fatigue across the two workshop days. To mitigate these risks, tasks were carefully designed to be of similar complexity but not identical, avoiding duplication that could lead to any possible bias. Using the same tasks on both days would have likely advantaged the second day due to prior exposure, thus artificially reducing task completion time and usability scores in favor of the DSL Agent. While knowledge transfer between the two sessions is possible, we argue it is rather minimal, since most participants were non-experts who could require more exposure than a few hours to fully familiarize themselves with the involved concepts to an extent that could skew the second day’s results. Furthermore, the two-day structure was also intentionally chosen to reduce cognitive overload and fatigue. Despite these limitations, the large sample size, task design, and consistent performance improvements support the conclusion that the DSL Agent improves usability and productivity. 

##### **6. Conclusions and future work** 

In this study, we present the DSL Agent, a conversational interface for creating DSL models, integrated in a low-code platform named Locsys. The agent offers a simpler and more intuitive interface that accelerates development time by producing DSL models in less time than traditional DSL development approaches and with fewer validation errors, without requiring extensive domain knowledge or expertise, as reported from a 2-day workshop. 

The results collected are very promising and validate such an interface’s significance and usefulness in the domain of DSL-based development and MDE. However, there is still room for improvement. Firstly, we are planning to improve the agent’s performance by diminishing the minor (syntax) errors generated. This could require more extensive prompt engineering and in-context learning or DSL refinement. Regarding the former, we will analyze the significance and influence of each information included in the prompts to determine the most optimal set. This can also allow us to integrate new DSLs into the agent’s API as the required information will be well-described and documented, a process that can later be automated. 

Furthermore, we are working on creating an LLM-based semantic model validator, which is included in the high-level DSL Agent internal structure. While generated models could correctly follow the DSL syntax, there is no guarantee that they contain the user-requested functionality and logic. Thus, such a mechanism that evaluates whether the produced DSL model solves the requested problem would be more than welcome, especially in cases involving further software development automation. 

Regarding the DSL refinement, we are planning on leveraging LLMs to improve the existing DSL syntax and terminology. This can lead to DSL meta-models whose models can more easily be comprehended and generated by the agent. Furthermore, we will also explore the creation 

of new DSLs using LLMs as the development tool. Similarly, this could lead to more LLM-friendly meta-models and less complex syntax. 

Finally, our study will also focus on extending the agent to a multiDSL one that can simultaneously handle all Locsys DSLs. This is a huge leap forward from a DSL-specific agent to a generic low-code agent. Other agentic studies have shown the LLM capabilities given a welldesigned prompting architecture, which makes us hopeful that most programming interfaces, particularly the DSL-based ones, will embed conversational capabilities in the near future.
