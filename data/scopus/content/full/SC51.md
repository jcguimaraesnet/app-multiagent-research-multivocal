## ABSTRACT 

Generative AI (GenAI) is currently being utilized in many tasks to improve their quality. Because GenAI tools are highly qualified in text-based applications, they have the potential to automate tasks across the software engineering lifecycle. In this study, we empirically investigate ChatGPT’s ability to perform code refactoring tasks. Considering five widely employed refactoring scenarios, we propose testing scenarios upon which we derive 200 test cases for five Java open-source applications. These test cases are applied using ChatGPT and NetBeans, and the results are compared and evaluated. To enable ChatGPT to perform refactoring, we follow prompt engineering approaches to design effective prompts. The results show that ChatGPT has generated refactored pieces of code, which have been successfully compiled for 88% of the test cases. However, it has correctly performed the intended refactoring for only 29% of the cases, compared to NetBeans, which has achieved a 67% correctness rate. These findings indicate that although ChatGPT has some potential to contribute to code refactoring tasks, it is not yet ready to be used as a fully automated refactoring tool for large-scale real-world applications. Its outputs still require human oversight to improve the refactored code’s correctness. 

## 1 | Introduction 

Generative AI (GenAI) has recently been considered one of the most popular areas for real-world applications in general and for artificial intelligence research in particular. The increasing focus on GenAI applications is a typical result of the high capabilities that these applications are capable of. Such capabilities include creating human-like new content in video, audio, images, or simply text [1–3]. GenAI applications are based on training their models on vast amounts of data records that are nowadays available with the exponential growth of internet applications. 

Software engineering is one computing area that has good potential in including GenAI in the different tasks of its life cycle [4–8]. 

Tasks can include code-related tasks such as code generation, reusability, and completion [9–12], code explanation [5, 13, 14], and code refactoring [4, 5, 15–24], or code unrelated tasks [5, 25, 26] such as requirement engineering, software evaluation, and testing [27–30]. 

In this article, we only consider code refactoring for software development. Software code refactoring is concerned with the process of improving the design of a fully functioning objectoriented code without affecting the expected outputs of the code in order to improve its quality measures [31]. Such quality measures include cohesion, coupling, size, maintainability, comprehension, and reusability [32, 33]. Refactoring aims to improve

In object-oriented programing (OOP), refactoring is applied to classes, methods, and their relationships to improve design quality and reinforce principles such as encapsulation, modularity, cohesion, and low coupling. Code refactoring has been considered earlier in combination with machine learning techniques to perform refactoring, detect refactoring, or test its effectiveness [34–39]. 

Existing popular code development tools such as NetBeans, Eclipse, and IntelliJ IDEA have code refactoring options that are typically used to assist software developers in refactoring. However, these tools suffer from some limitations such as only limited refactoring scenarios are implemented in these tools, each tool is specific for certain programing languages, some of the tools are commercial, and they usually require training the users to be able to use them. We, therefore, explore the idea of using GenAI for code refactoring to overcome the limitations of the existing tools, as GenAI models are well suited for text-based applications. Through prompt engineering, refactoring intent, and explicitly specified constraints, we explore the idea of the ability of GenAI to perform controlled code transformations. 

In our study, we specifically choose ChatGPT [40, 41] as the GenAI tool according to its popularity, effectiveness, and its ease of use, and NetBeans [42] as a common existing code refactoring tool. To ensure a fair and controlled comparison between ChatGPT and NetBeans, it is important to clarify the scope of the refactoring process evaluated in this study. While refactoring research often emphasizes the identification of refactoring opportunities, such as detecting code smells or recommending suitable refactoring types, our work specifically focuses on the execution of refactoring operations. In particular, we evaluate the ability of ChatGPT to correctly apply predefined refactoring operations once the refactoring opportunity and target are known. This design choice enables a controlled and fair comparison with NetBeans, which similarly requires explicit user selection of refactoring actions and does not autonomously identify refactoring opportunities. By isolating the execution phase, the study assesses the correctness, consistency, and reliability of ChatGPT as a refactoring tool. 

In addition, while refactoring is traditionally associated with improvements in software quality and maintainability, this work explicitly limits its scope to structural transformations. The goal is not to measure quality improvements or operational effects, but to evaluate the ability of GenAI to follow refactoring prompts and correctly apply the specified refactoring methods. 

We base our study experiments on five open-source existing Java applications with different sizes of line of code (LOC). Java is a widely used object-oriented language with mature and welldefined refactoring support, as well as extensive prior use in refactoring research [16–20, 34–39]. We consider five common refactoring scenarios namely, Rename method, Move method, Extract method, Extract superclass, and Pushdown field/method. These scenarios and their specifications are adopted from Fowler’s refactoring catalog [31]. The choice of Java, as a mature language with well-defined refactoring support, ensures that these refactoring scenarios can be consistently applied and validated. 

We extract a number of test cases from each application code to represent different test scenarios for each refactoring scenario. The 

total number of test cases on which we base our experiments is 200 cases. We also explore prompt engineering techniques to derive a general prompt to use for our experiments. It is important to note that our prompt can be also generalized for applying any refactoring scenario using GenAI tools. We, then, perform the refactoring scenarios on the 200 test cases through NetBeans and ChatGPT and analyze their results. Results show that although ChatGPT seems to work well for some cases, it fails many others since it is not designed to process the code based on an abstract syntax tree (AST), which results in missing many coding details. However, ChatGPT seems to be promising if it is explicitly well-trained on coding and refactoring detailed instructions. 

The major contributions of this article are as follows: 

1. To introduce test scenarios for five popular refactoring scenarios, including rename method, move method, extract method, extract superclass, and pushdown field/method. 

2. To generate test cases, based on the introduced test scenarios, for five open-source Java applications. 

3. To perform an empirical study on refactoring Java code by comparing the outputs of ChatGPT as a GenAI tool with the outputs of NetBeans as a popular existing tool. 

4. To derive a general prompt for software refactoring using GenAI tools. 

These contributions are evaluated through an empirical study guided by the following research questions. 

1. RQ1: Can ChatGPT correctly perform predefined refactoring tasks using structured prompts? 

2. RQ2: Does the performance of ChatGPT vary across different refactoring scenarios? 

3. RQ3: What key limitations does ChatGPT exhibit in prompt-based refactoring? 

These research questions are addressed through an empirical evaluation of prompt-based refactoring on real-world Java applications, combining quantitative analysis across refactoring scenarios with qualitative analysis of observed behaviors and limitations. 

The rest of the article is organized as follows: Section 2 presents the background and related work. Section 3 states the research methodology that we use for our study, which describes the refactoring test case selection as well as the applied prompt engineering. Section 4 explains the experimental settings, including the testing environment and evaluation criteria. Section 5 discusses the results of the experiments. Section 6 states a detailed discussion of the results. Section 7 lists the empirical study threats to validity. Finally, Section 8 concludes our work and states future work recommendations.

## 2.2 | Related Work 

A number of work in the literature consider the topic of using GenAI tools for software refactoring. The work by [4, 15, 17–20] use LLMs to search for refactoring potential in codes, that is, code smells. They do not consider directing the LLM for specific refactoring opportunities. Each of Zhang et al. [15] and Pomian et al. [18] consider only one well-known refactoring scenario, namely, move method and extract method, respectively. Whereas the others do not really focus on known refactoring scenarios, instead, they only define the refactoring as applying code improvement in general, such as improving the code appearance, reducing the number of lines, changing the fields types, and so on. The datasets that are used for their refactoring tests vary in sizes. However, full projects size are not considered as they focus on only the piece of code to be refactored. DePalma et al. [19] state that their Java files are of only one class each and contain 20–50 lines, while the others only mention the number of test cases. 

Xu et al. [16] propose MANTRA (Multi-AgeNT Code RefAactoring) that is based on a Refactoring Oracle Dataset by Tsantalis et al. [46]. They consider three well-known refactoring scenarios, namely, extract method, move method, and inline method, along with three related compound refactoring activities, namely, extract and move method, move and inline method, and move and rename method. They show that MANTRA results are similar to human refactorings in terms of readability and reusability, but only when providing a specific piece of code to refactor. 

Shirafuji et al. [21] and Poldrack et al. [22] base their study on Python code. Shirafuji et al. [21] train the LLM with few-shot examples extracted from introductory programing problems. The purpose of training is to enable the LLM to suggest less complex codes. Poldrack et al. [22] mainly demonstrate the ability of ChatGPT to refactor its own generated Python code, requiring only a little experience with prompt engineering. They select their Python code examples from GitHub, considering only codes with a restricted number of lines, which leads to their claim that since the codes have been heavily filtered, the results are probably underestimated. It is worth mentioning that the article does not provide any details about the refactoring methods that they use. 

White et al. [23] target software refactoring from the perspective of prompt engineering. They formulate test cases of prompt patterns for refactoring a newly generated code such that it follows the refactoring instructions while generating the code, or refactor an existing human-written code by giving example patterns of the requested code results. In both cases, they prompt ChatGPT to refactor the code using user/application-specific required patterns. They do not use the popular refactoring scenarios in the literature. 

Both work by Ebert and Louridas [24] and Biswas [5] suggest software refactoring as a potentially promising application of GenAI tools in software development and code optimization. They do not provide any experiments or results; rather, they study the tools’ capabilities and conclude with their potential use across the software development cycle stages.

## 3 | Research Methodology 

## 3.1 | Refactoring Test Cases 

We base our experiments on five Java applications that are opensource to support reproducibility and vary in sizes and domains to enhance generalizability. Details of the applications are shown in Table 2. 

We consider five refactoring scenarios, namely, Rename method, Move method, Extract method, Extract superclass, and Push down field/method based on their popularity in the literature [32–38], as well as their availability in the NetBeans tool that we use for our testing. These scenarios have different complexities in terms of involved tasks and classes. For example, the extract method refactoring usually involves changes in one class, whereas the move method refactoring may involve changes in many classes and packages. 

For each of the considered refactoring scenarios, we define several acceptance-based test scenarios. These test scenarios are considered to test whether the NetBeans refactoring tool and ChatGPT can correctly perform the refactoring scenario taking into consideration various refactored code possibilities. The test scenarios are provided in Table 3. 

For each of the five selected applications and for each of the listed test scenarios, we manually search for a piece of code on which the test scenario is applicable. That is, for each application, we select one test case per each test scenario, except for Cases 6 and 7 of the rename method refactoring, where we select two cases each. The total number of cases adds up to 200, which cannot be fully presented in the article due to space constraints. The complete set of test cases is, however, available upon request. 

We then apply NetBeans and ChatGPT to perform refactoring for each of the test cases and manually compare the actual outputs with the expected ones based on the refactoring scenario description. Because our study primarily focuses on evaluating the capability of GenAI to perform refactoring, we do not assess the code quality before and after the refactoring is applied. ## 3.2 | Applied Prompt Engineering 

In this section, we present our method of deriving the prompts that are to be used for refactoring experiments using ChatGPT. 

We base our design on the catalogs of the previous work by White et al. [23, 43], which has been explained in Section 2.1.1. It is important to note that prompts have initially been empirically developed, then enhanced using the prompts’ patterns and decided after a number of experiments that have been implemented independently of our test cases. 

We have first explored the Persona pattern from the Output Customization category as shown in Table 1. Thispattern assignsa clear role to ChatGPT in order for it to complete the required task. The Persona pattern requests to explicitly state two major things, namely, the role that ChatGPT is required to fulfill and the expected desired output of the given task. In our case, we request ChatGPT to act as an expert in Java software development as follows: 

### As an expert in Java software development, use [REFACTORING SCENARIO NAME] refactoring technique to refactor the attached Java code.

This pattern allows the user to state a sequence of steps that leads to the expected end result, possibly with known information or constraints. ## 4 | Experimental Settings 

We perform the refactoring for extracted pieces of code through NetBeans and ChatGPT, then manually analyze their results to check if the refactoring has been properly done or not. The test cases are based on the scenarios that are listed in Table 3. It is important to note that for the ChatGPT to perform the refactoring, we upload the whole project and ask ChatGPT to provide the updated code as a downloadable whole project as well. This is done to ease the process for the user such that the user is not required to know the specific files that are requested to be modified. The following subsections explain the testing environment of our experiments, and the evaluation criteria by which we evaluate the results. 

refactoring outcomes were assessed by two expert evaluators, with additional evaluators involved in cases of disagreement, resulting in up to four reviewers per decision. Furthermore, only a single execution of ChatGPT has been performed for each test case to ensure fairness and consistency in comparisons with NetBeans, which produces deterministic refactoring outcomes. For each of the two tools, the evaluated case is tagged with one of the following: 

- Pass: where the refactoring has been successfully performed, matching the expected result for this case. 

- Incorrect: where the refactoring has been incorrectly performed, as it does not match the expected result for this case. 

- Fail: where the refactoring has failed to be performed for this case (i.e., tool is unable to do the refactoring at all). 

- Not Applicable: where the case itself has not been found in the Java application. 

We also add two more tags for only ChatGPT results, namely, compiled and not compiled, to test whether the ChatGPT generated code after refactoring can be successfully compiled or not, respectively. Performance measures are documented in relation to both the refactoring scenarios and the evaluated applications. 

## 4.1 | Testing Environment 

We conduct the refactoring experiments using Apache NetBeans IDE v19 [42] and ChatGPT-4o [40]. NetBeans has been selected because it is a fully open source and free featured Java IDE that supports all refactoring operations evaluated in this study, enabling a consistent baseline across all scenarios. In contrast, lighter environments such as VS Code do not natively support some complex structural refactorings (e.g., Extract Superclass and Pushdown), which are central to our evaluation. While we acknowledge that some practical limitations exist across IDEbased refactoring tools, comparing or generalizing results across different IDEs is not the focus of this study. Our selection of NetBeans is driven by functional coverage and reproducibility rather than popularity. 

ChatGPT has been accessed via the web interface and prompted interactively using both single-turn text prompts, with the default temperature (0.7). All experiments have been conducted between February 2024 and January 2025. 

As any IDE, Apache NetBeans supports refactoring based on AST analysis, enabling semantic checks, safe refactoring constraints, and consistent updates across multiple files. In contrast, ChatGPT does not operate on an AST. Instead, refactoring is performed through natural-language prompts, and lacks built-in semantic validation. In this study, NetBeans is used as a reference for established refactoring operations, while ChatGPT is evaluated solely on its ability to follow refactoring instructions and generate structurally correct code transformations. The compilers that are used for our tests are Ant v1.10.14 [52] and Maven v3.9.5 [53].

### 5.1.2 | Move Method Refactoring 

Figure 3 shows that the success rate of NetBeans is 70% while having a failure rate of 30% (15 cases). This failure rate is relatively high for a tool that is mainly designed for programing related tasks. Due to the relatively large size of the Apache Hadoop application, NetBeans failed to process the Move Method refactoring for any of the corresponding 10 test cases described in Table 3. ## 6 | Discussion 

This section discusses our empirical findings and relates them to prior work on LLM-assisted refactoring. We first analyze ChatGPT’s observed behaviors reported from Section 5, then compare our results with closely related studies. 

## 6.1 | Discussion of Empirical Results 

We conclude the results that were obtained from our experiments. We primarily emphasize the most frequently observed behaviors of ChatGPT. Therefore, addressing RQ3 by discussing the most common behaviors, limitations, and failure patterns observed in ChatGPT when performing prompt-based code refactoring. It is important to note that although prompt derivation is one of the objectives of this study, testing prompt quality and effectiveness is implicitly reflected in the results. This is because the prompt structure is kept fixed, while only the refactoring description and specifications vary according to the refactoring scenario. 



The most common observed behaviors of ChatGPT can be summarized as follows: one important issue is that ChatGPT is not able to parse the Java code into an AST, even when explicitly instructed to do so. It, instead, breaks the input tokens through a tokenizer then processes the tokens using special encoding techniques. This feature has mainly affected the results by the incorrect placement of the code of the move method, extract method, extract superclass, and pushdown refactoring. It has also affected the rename method refactoring by not being able to locate all the references of the renamed method and updating them accordingly due to its incorrect scope. 

As for the correctness of generated code, since ChatGPT statically generates the code using tokens, it does not really compile or simulate a Java compiler to ensure code correctness. Recall that the test cases of both the extract method and extract superclass have suffered from partially extracted code, incorrectly placed code outside the class or within other methods, incorrectly placed code in constructors, double extending superclasses in classes’ signatures, as well as unsuccessfully replacing the extracted code with appropriate references. As for the move method and pushdown refactoring, the ChatGPT-refactored code has suffered from incorrectly placing the moved code into its appropriate requested place and failure to update the references of the moved code. These failures are due to the fact that ChatGPT uses a tokenized textbased approach to search for the required changes, not an

## 6.2 | Discussion in the Context of Related Work 

We compare our findings with the most closely recent related work [15–18] selected from the presented work in Section 2.2. A direct quantitative comparison is, however, not always feasible. This is because existing work differs in study objectives, the refactoring scenarios considered, and the evaluation criteria employed. 

Our study provides a more comprehensive evaluation of LLMassisted refactoring by covering a broader range of refactoring scenarios and assessing prompt-only refactoring execution at the full-project level. All four prior studies that we consider in our comparison concentrate on a limited subset of refactoring scenarios, typically localized within-document method-level changes [17], such as Extract Method [18], Move Method [15, 16], and Rename operations [17]. In contrast, we evaluate five refactoring scenarios and explicitly include Extract Superclass and Pushdown Field/Method, which are rarely investigated in LLM-based refactoring literature. These two scenarios are particularly important along with basing our refactoring executions on full-project level, because they involve inheritance restructuring and class hierarchy changes, requiring systematic multifile updates, reference consistency, and inheritance-awareness, as they represent cross-cutting, dependency-heavy transformations. This expands empirical evidence beyond localized refactorings and provides deeper insights into the challenges LLMs face when refactoring object-oriented systems.

## 7 | Threats to Validity 

Various aspects may constrain the applicability and interpretation of our findings. We base our study on one refactoring tool and one GenAI tool, which may restrict the generalization of the results. Having more tools would obviously raise the confidence level of the results. We have, however, selected popular, wellknown tools in their fields to ensure a good level of reliability. It is important to note that testing the quality of the code before and after refactoring is out of the scope of this research work, as the main focus is to test the ability of GenAI to follow the prompts and perform the required refactoring method. 

Another aspect would be the currently known limitations of the GenAI tools in general and ChatGPT in particular. ChatGPT is known to be nondeterministic such that it answers the same question in different ways, and is based on the quality of the prompts. We have, therefore, paid great attention to the prompt engineering task in this study by following a framework of prompts patterns that is proposed by White et al. [23, 43] as discussed in Section 3. In this study, only a single execution has been performed for each test case to ensure fairness and consistency when comparing results with NetBeans. ChatGPT does not also give a proper explanation of its process. Because we upload the whole project to it and ask it to return the updated code as a whole project as well, there is no way to know which files are the updated files and whether it has changed unnecessary other encodings in the code files as well. These unnecessary changes have caused compilation failures in some cases. However, the main focus in this study is on the ability of ChatGPT to properly perform the refactoring. 

Two more points that might affect the generalization of results are the test cases selection and the applications selection. As for the test cases selection, we consider one example from each application to represent each defined test scenario. We have selected the examples of each scenario from different applications to reduce the 

risk that the same application might always give the same results. Therefore, such selection increases the confidence of the results’ reliability and generalization. 

Regarding the applications selection, all the applications that we base our study on are Java-based open-source applications in order to fit into the selected refactoring tool compatibility requirements. Besides, Java is one well-known leading object-oriented programingOOP language that is heavily used in many applications developments. Finally, although we base our study on only five applications, varying the sizes and domains of the applications has been taken into consideration to increase the validity of the results. 

## 8 | Conclusions and Future Work 

This article provides an empirical study on utilizing ChatGPT for refactoring object oriented software applications. The study is based on five different real applications that vary in their sizes and domains. A number of 200 test cases have been extracted from the five different applications based on different defined testing scenarios for five popular refactoring scenarios, namely, Rename Method, Move Method, Extract Method, Extract Superclass, and Pushdown Field/Method. We perform the refactoring tasks involved in the test cases using NetBeans, as a well-known coding tool, as well as ChatGPT. Prompt Engineering methods have been carefully taken into consideration when designing the prompts for ChatGPT. Results show that although ChatGPT has shown potential to simplify certain refactoring tasks, in some cases, by elevating some restrictions imposed by NetBeans, ChatGPT is not yet sufficiently reliable for fully autonomous refactoring. It appears to perform better on smaller projects with supervision, while larger real-world projects present greater challenges. 

Future work directions may include applying the same study on different GenAI models such as MS Copilot, Gemini, Claude, and Deep Seek for comparative analysis, involving more applications, refactoring scenarios, and test cases, and using different prompts’ designs. Improving the performance of LLMs, such as ChatGPT, in performing code refactoring tasks is an open area for further research. Another important direction for future research is to investigate ChatGPT’s capability to identify refactoring opportunities autonomously. This includes detecting code smells, reasoning about design deficiencies, and recommending appropriate refactoring strategies prior to execution. Evaluating the accuracy and code quality before and after applying such recommendations remains an important direction for future research. Finally, one important direction to investigate as future work is to investigate result stability through repeated trials and variance analysis for ChatGPT, rather than single-run experiments.
