- **ABSTRACT** Project-based courses increasingly rely on Git version control systems such as GitHub, yet raw commit histories are difficult for instructors to interpret in terms of collaboration and feature development. This paper investigates how large language models (LLMs) can support this analysis by generating natural-language summaries of student commits and assigning feature-level labels. Using data from a software engineering project course, we apply a four-phase LLM pipeline that (i) extracts and normalizes commit data, (ii) infers high-level feature and non-feature categories from commit messages, (iii) consolidates synonymous labels into a unified vocabulary, and (iv) classifies each commit to a specific label. We then group the resulting labels into **Compulsory Features** , **Optional Features** , and **Non-Features** (maintenance tasks such as testing, refactoring, and configuration) to distinguish teams that implemented only minimum requirements from those that pursued additional functionality. We derive analytics on feature coverage, collaboration patterns, commit-size distributions across teams, and workload balance using per-student commit shares and the Gini coefficient. Our results show that many teams collaboratively implemented core features, while optional requirements revealed substantial variation in ambition. Teams that worked on optional features tended to receive higher grades, providing initial evidence that the LLM-derived analytics align with instructor judgements. In addition, the LLM-generated summaries and labels help instructors locate where features are implemented, exercise them during grading, and provide targeted feedback. We argue that LLMs can act as practical co-analysts for educational GitHub repositories, transforming low-level version-control data into actionable insights for teaching and assessment.

### **I. INTRODUCTION** 

Project-based learning has become a cornerstone of software engineering education, emphasizing authentic teamwork, version control practices, and iterative software development [1]. As educators adopt platforms like GitHub to manage student projects [2], vast amounts of process data— such as commit histories, pull requests, and issue logs— are being generated [3]. These digital traces provide a rich opportunity to study how students collaborate, distribute work, and progress through software development cycles. However, mining and interpreting raw Git data remains challenging [4]. Commit logs, though abundant, are often

Educators therefore need visibility into how students actually collaborate, not just how much they commit. To make this actionable, analytics must separate contribution (each student’s share of work) from collaboration (the extent to which multiple students work on the same artifacts or features), enabling instructors to see whether teams integrate efforts or operate independently [5]. This distinction motivates moving beyond raw volumes toward interpretable, feature-oriented signals that reveal both who did what and who worked together.

### **II. RELATED WORK** 

### _A. MINING STUDENT GITHUB REPOSITORIES FOR_ 

### _LEARNING ANALYTICS_ 

A substantial body of work has examined how data captured during programming activities can be mined to understand student behavior and support instruction. 

Within software engineering education, GitHub has emerged as a prominent collaboration and submission platform, creating opportunities to analyze authentic teamwork traces. Qualitative investigations show that GitHub’s transparency and social features can benefit classroom collaboration while also introducing challenges (e.g., steep learning curves, limited support for education-specific workflows) [11]. Building on this, an exploratory classroom case study with 91 students reported heterogeneous team workflows and cautioned that using raw GitHub metrics as proxies for performance is _risky_ , motivating more nuanced analytics beyond simple counts [12]. 

Parallel advances in software repository analytics provide techniques relevant to education. For example, Githru demonstrates scalable visual analytics over large commit graphs—reconstructing and clustering DAGs, summarizing commit activity, and enabling comparison views—thereby illustrating how commit-level metadata can be transformed 

into interpretable overviews of development histories [13]. While evaluated on industrial datasets, these abstractions (e.g., cluster-level summaries, file-hierarchy exploration) are directly applicable to course settings seeking to interpret team trajectories over time. 

Most recently, education-focused pipelines have begun to operationalize version-control mining for real-time learning analytics. Chen et al. present _Polivr_ , an ecosystem that (i) anonymizes repositories for ethical analysis, (ii) extracts commit-level metrics via a repository processor, and (iii) renders instructor-facing visualizations—explicitly targeting behavioral insights in programming assignments [14]. This architecture concretizes the end-to-end flow from Git data to privacy-aware dashboards and enumerates metric categories obtainable from each commit, advancing prior calls for more robust, multi-dimensional analytics beyond single counts [15]. 

The literature indicates that mining student GitHub repositories can surface collaboration patterns and development behaviors at scale; however, naive use of activity metrics is insufficient for evaluation or pedagogy. Emerging systems that integrate privacy, richer commit-derived features, and interpretable visual summaries point toward a more reliable learning-analytics stack for software engineering courses— one that aligns with the educational need for validated, context-aware constructs rather than raw volume-based proxies [12], [14], [15]. 

### _B. LLMs FOR CODE-CENTRIC ANALYTICS AND EDUCATIONAL SUPPORT_ 

Recent work positions LLMs as versatile tools for analyzing software artifacts and supporting learning at scale. In computing education, LLMs can classify student queries with little or no labeled data, reaching near human-level agreement after light fine-tuning; this enables targeted, realtime feedback loops and lowers integration barriers for instructional tools [16]. 

On code understanding tasks more broadly, systematic evaluations show LLMs can automate aspects of code analysis yet face limitations—especially under obfuscation and low-readability conditions. These studies curate datasets of real-world and obfuscated programs and test stateof-the-art GPT/LLaMA families to map capabilities and failure modes, framing open questions around robustness and generality for downstream SE analytics [17]. This perspective is relevant when summarizing complex Git diffs: commit-level prose often compresses many edits, renames, or noisy churn, mirroring the readability challenges flagged in code-analysis evaluations [17]. 

For code-to-text specifically (summarization/explanation), technical surveys report mixed performance across benchmarks (e.g., CodeXGLUE, HumanEvalExplain): several open models achieve competitive BLEU score [18] on summarization datasets, yet explanation scores lag and remain uneven across languages and model sizes. This work bridges these domains by operationalizing LLM-based commit summarization within a learning-analytics framework tailored for software engineering education. By transforming raw version-control traces into interpretable narratives of contribution and collaboration, our approach advances beyond descriptive metrics toward a richer, semantically grounded understanding of student teamwork and learning processes. 

### **III. METHODOLOGY** 

This section covers the course context, data extraction and processing, and the analysis methods. Central to our approach is an LLM-based pipeline that infers high-level feature labels from commit messages and classifies commits accordingly. 

### _A. COURSE CONTEXT_ 

This study was conducted in a third-year Software Engineering course on Service-Oriented Architecture. Student teams developed a service-oriented application hosted in GitHub repositories (one repository per team). The dataset comprises 10 teams: eight teams of three students (Teams 1–8), one team of two students (Team 9), and one team of four students (Team 10). All teams used GitHub for version control and collaboration. The project spanned seven weeks, with students working on their projects both in class and independently. 

Students were encouraged to draft or refine commit messages with AI assistants integrated in their IDE 

(e.g., Copilot Chat). As a result, commit messages in this cohort were generally more descriptive than is typical in student repositories. This practice partially mitigates known limitations of message-only analytics and improves the reliability of our LLM labeling. 

The target application is a Learning Management System (LMS), an educational service platform implemented as a collection of services and Application Programming Interfaces (APIs). Students were required to implement a minimal core of compulsory features (listed in Table 1): API documentation; assessments (assignments & quizzes); authentication and authorization; content, course, and enrollment management; a REST API; notifications and email; progress tracking; and user & profile management. These compulsory items represent the minimum product-level functionality each team had to deliver. 

Teams were also expected to implement additional features. Example optional features include attendance tracking, certificate management, dashboards and analytics, discussion forums, meetings/calendar integration, real-time chat, scholarships/donations, sections scheduling, and survey management; teams chose optional features based on interest and available time (see Section IV for which teams implemented which options). 

Finally, teams performed routine engineering and maintenance tasks that are not product features but are essential to delivery: testing, refactoring, build and configuration work, dependency management, developer tooling, documentation, data seeding, and related infrastructure/configuration activities. These ‘‘non-feature’’ activities are treated separately in our analysis to distinguish product development from maintenance and quality-of-work efforts.

### 1) PHASE 1: COMMIT DATA EXTRACTION 

Prior to extracting commit metadata, repositories were cloned locally and author identities were normalized. Because the same student can appear with multiple name/email combinations (different machines, web edits, etc.), we mapped multiple identities to a single pseudo-identity per student and rewrote commit author metadata so that all commits by the same person appear under a consistent author record. This preprocessing ensured more accurate contributor attribution in downstream analysis. 

After identity normalization, we restricted analysis to the main branch only (the instructor required final submissions to reside on main). Limiting to main both aligns with course instructions and reduces the risk of double-counting commits that were previously merged from feature branches (including deleted branches or branches that were exploratory in nature). 

From each normalized main branch, we extracted commitlevel metadata recording: commit hash, parent(s), author, commit timestamps, commit message, and file-level changes (paths, additions, deletions). During extraction, merge commits were identified and distinguished from regular (non-merge) commits. Merge commits were omitted from commit-level counts and the remainder of the pipeline because they inflate contribution counts; their changes are already contained in the merged commits, and omitting them 

yields a clearer, linear view of individual contributions. The resulting dataset was serialized into structured JSON files, one per team, which served as the input for subsequent LLM processing. 

### 2) PHASE 2: LLM-BASED SUMMARIZATION AND FEATURE INFERENCE 

The second phase employed OpenAI GPT-5 to analyze commit messages and infer high-level product features and maintenance categories. We use GPT-5 in this phase because this step requires stronger reasoning across many commit messages, which helps produce more reliable summaries and candidate labels. This step aimed to generate interpretable summaries that capture the technical intent and pedagogical relevance of student work. 

For each repository, all commit messages (subject lines only) were collected into a single JSON array and submitted to the LLM in one request. Crucially, the model received _only_ the commit messages—no code diffs, file paths, or metadata beyond the commit message text. The prompt instructed the LLM to perform high-level reasoning across all messages to deduce the application’s purpose, its main functional areas, and the types of work represented in the commit history. 

The LLM was tasked with three specific objectives: 

- 1) **Overview synthesis:** Generate a one-paragraph natural-language summary describing the overall purpose and progression of the project, based solely on evidence from the commit messages.

### 3) PHASE 3: CONSOLIDATING SYNONYMOUS LABELS 

While Phase 2 yielded repository-level summaries and feature categories, the initial output contained substantial redundancy. Across the 10 repositories, the LLM produced 118 distinct _feature_ labels and 104 distinct _non-feature_ labels. To construct a unified, course-wide vocabulary suitable for commit-level classification, a three-stage consolidation process was applied: 

- 1) **Alphabetical ordering.** All labels were sorted alphabetically within their respective categories (features and non-features). This visual arrangement immediately exposed exact duplicates and near-duplicates differing only in capitalization, punctuation, or pluralization (e.g., _Authentication_ , _Auth_ , _Authentications_ ). 

- 2) **Exact deduplication.** Duplicate strings and trivially identical formulations were merged, reducing the label sets to intermediate counts. 

- 3) **Semantic consolidation.** The first author identified pairs and groups of labels with distinct surface forms but equivalent intent (e.g., _Sign-in_ → _Authentication_ ; _Enroll Course_ + _Course Registration_ → _Enrollment Management_ ) and proposed a curated alias map to unify them. 

The second author independently reviewed the consolidated sets, and the authors discussed discrepancies to reach consensus. This process yielded a final vocabulary of 20 feature labels and 11 non-feature labels. To ensure alignment with course requirements, the instructor verified that all compulsory features specified in the course rubric were present in the consolidated feature list; all were confirmed to be included.

### 4) PHASE 4: COMMIT-LEVEL CLASSIFICATION 

Using the consolidated vocabulary from Phase 3, we applied commit-level labeling to all non-merge commits across all 10 repositories. Each commit message was submitted individually to the LLM (GPT-5 mini) with a classification prompt instructing the model to assign exactly one label from the unified set of 20 features or 11 non-features. We use GPT-5 mini in this phase because each commit is assigned to one label from a fixed list, so a faster, lower-cost model is sufficient at this scale. The prompt specified a strict JSON output format requiring the model to select from the provided label list. To avoid inflating coverage through speculative assignments, the LLM was explicitly instructed to use the reserved label ‘‘NONE’’ for any commit that did not clearly correspond to an identified feature or maintenance category. 

The first author manually reviewed a random sample of 150 commits drawn across all 10 teams and across predicted label categories to validate classification accuracy. The review determined that the LLM achieved a very high accuracy rate of 98%, confirming that the commit-level labels were reliable for the subsequent analyses. 

By the end of this phase we obtain a finalized, cleaned commit dataset: for every non-merge commit we (1) know its resolved author identity after alias normalization, and (2) have a categorical label—either one of the 20 featurerelated labels (which includes compulsory and optional features), one of the 11 non-feature labels, or NONE when the LLM decided no assignment is appropriate. ### _C. ANALYSIS AND VALIDATION_ 

Using the labeled commit dataset from Phase 4, we derive several analytics to assess student contribution and collaboration patterns at the team and feature levels. Our analysis treats commits as discrete units of work under the assumption that each commit represents one development action. While we recognize that commits vary substantially in scope and complexity, this simplification is necessary for scalable, automated analysis across multiple teams and repositories. To evaluate the validity of this assumption, we examine the distribution of commit sizes (measured as total lines changed) to determine whether commits are relatively uniform or exhibit extreme variation, and when there is significant variation, to what extent it may affect our interpretations. 

### 1) COMMIT SIZE DISTRIBUTION 

For each commit, we define its size as the number of lines added and deleted as recorded in the Git diff. We compute the median and interquartile range (IQR) of commit sizes across all non-merge commits in each repository. This distribution provides context for interpreting commit counts: if most commits fall within a similar range, treating them as equal units is a reasonable approximation; if the distribution is highly skewed or includes extreme outliers, commit counts alone may be misleading. By reporting the distribution, we acknowledge the limitations of count-based metrics while demonstrating that, in our dataset, commit sizes are sufficiently concentrated to support comparative analysis. 

### 2) COMMIT COUNTS BY STUDENT AND LABEL 

For each repository, we aggregate non-merge commits by student and by feature or non-feature label. This produces a contributor–label matrix that, for each label, reports: (i) the total number of commits, (ii) the number of unique contributors, and (iii) the commit count for each individual contributor. We then group labels into three categories— **Compulsory Features** , **Optional Features** , and **Non-Features** —to provide a clearer picture of team effort across different types of work. This grouping enables us to distinguish between required product functionality (compulsory features), value-added functionality (optional features), and non-functional work such as testing, refactoring, and configuration (non-features). By analyzing commit counts within

each category, we can assess both the distribution of effort across team members and the balance of product-oriented versus maintenance-oriented work. 

### 3) MEASURING WORK INEQUALITY WITH THE GINI COEFFICIENT 

To quantify how evenly work is distributed within each team, we compute the Gini coefficient [21], a widely used measure of inequality in economics and social sciences. The Gini coefficient ranges from 0 (perfect equality) to 1 (maximum inequality), where lower values indicate more balanced distributions and higher values indicate concentration of work among fewer contributors. Unlike simple variance or standard deviation, the Gini coefficient captures relative differences in shares, making it suitable for comparing contribution distributions across teams of varying sizes. 

We compute the Gini coefficient for each team both overall (across all commits), separately for each of the three label categories (compulsory features, optional features, and nonfeatures), as well as for the NONE label. Category-specific Gini values reveal whether inequality is consistent across all types of work or whether certain categories are dominated by specific individuals. When fewer than five commits exist in a given category, we do not interpret the Gini coefficient, as small sample sizes can produce misleading values, and might also be artifacts of the LLM labeling process rather than true work patterns (particularly commits that were not part of the validation sample set). 

### 4) DISTINGUISHING CONTRIBUTION FROM COLLABORATION 

We define _contribution_ as the individual effort each student invests in the project, operationalized as their share of commits (overall or within a specific feature category). An equal distribution of commits (low Gini) indicates balanced contribution, while an unequal distribution (high Gini) suggests that some students contributed significantly more than others. In contrast, we define _collaboration_ as the extent to which multiple team members work on the same tasks, operationalized as the number of unique contributors per feature label. A feature with commits from all team members indicates likely collaborative development, whereas a feature with commits from only one student suggests individual or siloed work. By analyzing both commit shares (contribution) and multi-contributor features (collaboration), we can distinguish teams that divide work evenly but independently (high contribution equality, low collaboration) from teams that work together on shared tasks (high contribution equality, high collaboration).

### **IV. RESULTS** 

This section presents results from applying the LLM-based commit analysis pipeline to 10 student teams. We first validate commit counts as contribution proxies by examining commit size distributions, then analyze contribution and collaboration patterns, feature coverage, and work inequality. These analyses demonstrate how LLM-derived feature labels enable instructors to move beyond surface-level metrics toward interpretable, feature-oriented views of student teamwork. 

### _A. COMMIT SIZE AND FREQUENCY_ 

Before analyzing collaboration patterns, we establish baseline metrics on commit frequency and size to ensure that commit counts are valid proxies for contribution. If commits vary widely in scope, simple counts may be misleading; conversely, if most commits are relatively uniform in size, counting commits provides a reasonable approximation of effort distribution. 

We measured commit size as the total number of lines changed (additions + deletions). Across all teams, the distribution of commit sizes was highly concentrated: 85% of all commits modified fewer than 200 lines, 90% fewer than 300 lines, and 95% fewer than 500 lines. This indicates that the vast majority of commits represent relatively small, incremental changes rather than large-scale refactorings or bulk imports. The overall median commit size was 41 lines (IQR = [14, 122]), further confirming that typical commits are modest in scope. These figures are consistent with recommended version-control practices that favor frequent, focused commits over infrequent, monolithic ones, and suggest that

### **V. DISCUSSION** 

Our findings suggest that LLM-based commit summarization and feature labeling can surface meaningful patterns of 

61127 

VOLUME 14, 2026 

R. Gedeon, N. Giacaman: LLM-Driven Feature Inference for Analyzing Student Git Collaboration 



collaboration that are difficult to recover from raw Git logs alone—and critically, can do so at scale. In courses with dozens of teams and hundreds or thousands of commits, manual interpretation of commit histories is prohibitively time-consuming for instructors. By automating the aggregation of commits into **Feature** , **Optional Feature** , and **NonFeature** categories, we move beyond simple volume-based analytics towards a scalable view of _who worked on what, together_ . This automated, three-category view highlights that many teams engaged collaboratively on core product features, with several exhibiting sustained all-member participation on key capabilities. At the same time, the same representation makes it visible where feature development was dominated by one or two contributors, signalling potential imbalances that instructors might want to address. Without LLM assistance, identifying such patterns would require manually reviewing thousands of commits—a barrier that effectively prevents most instructors from engaging with repository data at this level of granularity. 

The contrast between **Feature** and **Non-Feature** categories is also pedagogically informative. In our cohort, maintenance-oriented work (e.g., refactoring, configuration, testing, and documentation) does not consistently skew toward solo activity; in some teams it is concentrated in one or two contributors, while in others it is shared among multiple members. This raises questions about role specialisation in student teams: students who predominantly work on maintenance may miss opportunities to engage with core design decisions, while those focused mainly on features may have fewer chances to practise software quality activities [23]. The LLM-derived labels offer a compact way to detect such patterns (e.g., students contributing mainly to ‘‘maintenance’’ versus mainly to ‘‘features’’) and open space for targeted formative feedback or deliberate role rotation in future iterations of the course. 

The commit-size analysis provides a complementary lens on how teams structure their work. The observed inverse relationship between commit size and number of commits suggests two distinct collaboration strategies: some teams favour fewer, larger commits, while others distribute their work into many smaller increments. From a teaching perspective, these patterns matter because very large commits can hinder code review, make rollback harder, and obscure individual contributions, whereas excessively small or noisy commits may indicate trial-and-error rather than planned changes [24]. By combining LLM-based semantic labeling with basic repository metrics, instructors gain a richer picture of not only what students built but also how they organized their workflow. 

The temporal view (Figure 5 and Figure 6) further complements these metrics by surfacing _when_ teams invested effort in particular labels and whether that effort was interleaved across multiple features. In an educational context, this can help contextualize teams’ pacing (e.g., whether work on compulsory features overlaps over time and whether non-feature activity is distributed or concentrated), while 

remaining agnostic about underlying intent. It also provides a qualitative view of implementation ordering (e.g., whether a label becomes active early or late) and the extent to which multiple contributors are active within the same label window, which can support interpretation of the aggregate collaboration patterns. 

Feature coverage results further illustrate how the pipeline can reveal differences in project scope and strategy. Across the cohort, teams reliably covered compulsory features, but Optional Features showed substantial variation: some teams implemented a few while others did not implement them at all. This variation is pedagogically useful. It suggests that optional requirements can function as a ‘‘design space’’ where teams express different priorities and levels of ambition, and that monitoring which students engage with these advanced features can help instructors identify students who are ready for more challenging tasks [25]. 

Finally, the alignment between LLM-derived coverage and instructor grading provides initial support for the validity of the approach. Teams that implemented a broader and deeper set of features—especially optional ones—tended to receive higher grades, indicating that the analytics captured aspects of project depth that the instructor valued. Beyond this aggregate agreement, the LLM-generated summaries and feature labels can directly support grading: they help instructors quickly locate where a given feature is implemented, verify whether it behaves as specified, and formulate targeted comments on missing or partially implemented behaviour. We see this not as a replacement for instructor judgement, but as an assistive layer that reduces the effort of navigating large student repositories and enriches assessment, moderation, and feedback. 

Our pipeline is designed for _end-of-semester_ use after repositories are frozen. We run all phases once to produce a retrospective, instructor-facing analysis for grading moderation and targeted feedback, which avoids mid-semester volatility, reduces surveillance concerns, and ensures a stable label vocabulary. 

Overall, the study demonstrates that LLMs can act as practical ‘‘co-analysts’’ for mining student GitHub repositories in educational settings. By turning low-level commits into feature-level, collaboration-aware views, they enable instructors to move beyond counts and timelines towards more interpretable, pedagogy-oriented analytics. We envision future iterations of this work embedding such analyses into instructor dashboards, enabling early warnings about unbalanced teams, and supporting student-facing reflection activities where teams review their own collaboration footprint. 

### _A. THREATS TO VALIDITY_ 

Several threats to validity affect the interpretation of our feature-level and temporal analyses. Our contribution and inequality metrics treat commits as discrete units of work. Although we report commit-size distributions to 

61128 

VOLUME 14, 2026 

R. Gedeon, N. Giacaman: LLM-Driven Feature Inference for Analyzing Student Git Collaboration 



contextualize this assumption, commits still vary in cognitive effort (e.g., debugging versus mechanical changes), and different teams may adopt different commit granularities. This can bias comparisons when one team batches work into fewer commits or when contributors differ in how often they commit. 

The Gantt-style spans in Figure 5 and Figure 6 are derived from the timestamps of commits, not from continuous work logs. A long span between a label’s first and last commit does not imply sustained work throughout that period; it may reflect intermittent revisits, delayed commits, or late integration. Conversely, intense development can occur off-record (e.g., local changes accumulated before a single commit). Git timestamps also reflect when changes were committed, not necessarily when they were developed. Git’s reliance on local clocks means that commits can be recorded with inaccurate timestamps if a contributor’s clock is misconfigured. 

We restrict analysis to main-branch commits and omit merge commits to avoid double counting. While appropriate for the course submission context, this choice can under-represent work that occurred on short-lived branches, in rebases, or in squash merges where intermediate commits are not preserved. As a result, some ordering and collaboration traces may be partially obscured. 

Our labels are inferred and assigned using LLMs, and Phase 4 uses commit messages (not diffs) for commitlevel classification. Ambiguous, terse, or AI-assisted commit messages may lead to misclassification, especially for maintenance work that is difficult to distinguish from feature work without code context. Although our manual validation suggests high accuracy on a random sample, errors can affect low-frequency labels, and the estimated start/end of a label’s activity window. 

We interpret collaboration primarily through multicontributor activity on the same label. This may undercount collaboration when students coordinate via pair programming, design discussions, or code review but commit under a single author, and may overcount collaboration when multiple authors contribute small, independent changes to the same label category. 

The study is based on a single course, institution, and project domain with a limited number of teams. Workflows, commit practices, and feature taxonomies may differ in other contexts (e.g., industrial repositories, different pedagogical scaffolding, or teams with different sizes), limiting generalizability. 

### **VI. CONCLUSION** 

This paper explored how LLM-based commit summarisation and feature labelling can be used to analyse collaboration and feature coverage in student software projects. By grouping commits into **Feature** , **Optional Feature** , and **Non-Feature** categories and combining this view with commit-size statistics, we moved beyond raw counts to reveal who worked on which capabilities, how work was distributed 

within teams, and how teams balanced core features with maintenance tasks. The analysis showed that teams with broader and deeper feature coverage—particularly on optional requirements—tended to receive higher grades, and that LLM-generated summaries and labels can help instructors more efficiently locate, exercise, and comment on specific features during grading. 

Several avenues for extending this work remain open. First, integrating the pipeline into real-time dashboards could enable instructors to monitor team dynamics as projects unfold, surfacing early warnings when teams exhibit extreme inequality or minimal feature coverage. Second, incorporating code review activity, issue discussions, and pull requests would provide a more complete picture of collaborative practices. Third, evaluating the approach across multiple courses and institutions would establish generalizability. Finally, student-facing analytics—where teams review their own LLM-generated summaries—could support metacognitive reflection and team self-regulation.
