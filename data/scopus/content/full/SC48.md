For each open source generative AI model, three distinct prompts are given. 

## Prompt Engineering 

In a generative AI model, prompting refers to the provision of input that contains clear instructions for guiding an LLM to produce the desired output. Different styles and structures of prompts can lead to significantly varied outputs even when the underlying task remains unchanged.<sup>12</sup> If the prompt explicitly instructs the model to prioritize security, the LLM is more likely to produce code that incorporates safeguards against these vulnerabilities. 

For the purpose of our study we use three different prompts: 

- _P1 (security-agnostic prompt)_ : Write a code in the Python language to _[Task]_ . 

- _P2 (task + role definition prompt)_ : You are a computer security professional tasked with writing nonvulnerable code. Write a code in the Python language to _[Task]_ . 

- _P3 (security-aware prompt)_ : You are a computer security professional tasked with writing nonvulnerable code. Write a code in the Python language to _[Task]_ . Make sure there are no potential vulnerabilities or security concerns with the code. Follow secure coding practices throughout.

## Code Generation 

As they are trained on a large corpus of programming languages and documentation, LLMs show great capabilities in generating functional code snippets. These models learn the syntax and semantics of the programming constructs and common software development patterns.<sup>13</sup> This makes them useful for automating code generation, solving algorithmic problems, and performing advanced software development tasks such as bug fixing and test generation. 

In this study, we use multiple LLMs and feed each model with the three carefully designed prompts to evaluate their ability to generate 

secure and functionally correct Python code. 

## Vulnerability Analysis 

To assess the security of the generated code, we conduct a comprehensive vulnerability analysis using three independent static analysis tools: Bandit, Semgrep, and Snyk. Each of these tools is based on different rule sets, detection heuristics, and threat models. To further reduce tool-specific bias, we adopt a majority voting (best-of-three) strategy to determine whether a generated code sample is vulnerable or not. This ensemble-style approach significantly lowers the likelihood that overly conservative rules from any single tool dominate the results. At the same time, it preserves sensitivity to genuine security flaws that are consistently recognized across tools. 

To further address the potential false positives generated by static analysis tools, a manual review was conducted. This review was conducted by three independent practitioners experienced in secure 

software development. A stratified random sample of 356 flagged outputs was selected among the outputs flagged by the tools. This ensured representation across models, prompts, and task categories. The reviewers verified each reported issue against its CWE definition and checked whether the vulnerable code path was practically reachable. The reviewers found that single-tool detections occasionally showed conservative pattern matching. The issues flagged by multiple tools were consistently (>90%) validated as genuine security weaknesses. These findings support the effectiveness of the majority-voting approach. This also indicates that the reported vulnerabilities are not driven by false tool outputs.

## How Do Open Source LLMs Perform While Generating Code for Different Software Engineering Tasks? 

To answer this RQ, we present the percentage of vulnerable code generated by LLMs across various software engineering tasks in Table 1. Table 1 highlights the variability in secure code generation capabilities of LLMs based on both the task type and the prompt settings. Lower percentages indicate better performance and a model’s ability to generate secure and reliable code. Among the evaluated models, CodeLlama and DeepSeek-r1 frequently generate less vulnerable code. 

Prompt design also plays a significant role. For instance, across nearly all models, prompts P1 and P2 often result in generating the worst case. This indicates that strategically serving the right prompt to the LLM can increase the security of the generated code drastically. 

## How Do Different Prompts Impact the Vulnerability Levels in the Code Generated by Different Open Source LLMs? 

Given the nondeterministic nature of LLMs, each prompt–model– category configuration was evaluated over 20 independent runs. We first applied one-sided Wilcoxon signedrank tests to compare the baseline prompting condition (P1) against the security-biased prompts (P2 and P3). Across almost all LLM–category pairs, at least one of the contrasts P1 > P2 or P1 > P3 was statistically significant. This indicates that introducing security bias into the prompt generally reduces the probability of generating vulnerable code. The few cases where neither contrast was 

significant are highlighted in Table 2. We further conducted Page’s L trend test to assess whether increasing the levels of security bias induces a monotonic improvement in secure code generation (P1 > P2 > P3). The evidence for a monotonic trend is less consistent across models and categories. Several combinations show strong results. For example, CodeLlama (app features and system operations), Codestral (emerging tech), DeepSeek-r1 (emerging tech), and CodeGemma (web and back end) have statistically significant Page test results. They also show large trend effect sizes (r ≥ 0.6). Trend effect sizes were computed as _r_ = ( _Z_ / _N_ ), where _N_ = 20. Many other combinations differ and show nonsignificant or oppositely signed trend statistics. Yet, they often have significant Friedman tests. 

To investigate further, we measure the CWE counts in the code generated by the LLMs. As can be seen in Figure 2, DeepSeek-r1 generates the lowest number of codes that contain only one vulnerability for all three prompts. But DeepSeek-r1 with prompt 1 also has the highest count for the number of codes generated that have a vulnerability count greater than three. Output code that contains a single vulnerability may indicate a localized mistake. Conversely, outputs that contain multiple concurrent vulnerabilities suggest that secure design principles were not consistently applied. This, in turn, suggests deeper weaknesses in how the model constructs security-sensitive logic. GPTOSS with prompt 1 has the highest count of the generated code that contains exactly one vulnerability among the open source LLMs. 

Taken together, these findings suggest that although security-biased prompting generally improves code 

safety relative to the baseline, further increasing the bias is not a guarantee for uniform proportional gains. 

These results support the conclusions of the previous studies<sup>10,14</sup> that LLMs generate codes prone to security vulnerabilities, as evidenced by the identified CWEs in our work. 

## Which CWEs Are Prominent in the Code Generated by Various Open Source LLMs? 

To answer this RQ, we analyze the data extracted from each model’s output using static analysis. 

First, we collect the vulnerability data from the outputs of different LLMs. The code generated by each model is assessed for the presence of software vulnerabilities. Next, we examine how frequently different CWEs appear in the code generated by each LLM. By identifying the CWEs that occur most often for each model, we gain insight into the kinds of security issues that each model introduces. To enable a fair comparison, we focus on the set of CWEs that are most common across all models. This creates a unified set of vulnerability categories and allows us to compare the models on a common ground.

# **Why Model Choice Shapes Security Outcomes** 

A central finding of this study is that selecting an LLM is not a neutral engineering choice. Even when the models receive the same prompts and programming tasks, they produce markedly different numbers of vulnerabilities. This pattern holds across domains and prompting strategies. Statistical tests show that the result remains significant after accounting 

for task difficulty and generation randomness. These differences arise because each model encodes its own notion of what constitutes acceptable code. This is mostly shaped by the LLM’s training data, underlying architectures, and alignment processes. 

Training data play a particularly important role. Models trained on heterogeneous or noisy code sources, such as legacy repositories or community forums, are more likely to reproduce insecure coding patterns. On the other hand, models trained on more curated datasets tend to default to safer coding practices. 

Architectural and capacity differences further amplify this gap. The Wilcoxon test shows that introducing security bias improves the 

results relative to P1. Still, Page’s monotonic trend test is inconsistent. This suggests that it is not necessary that structural safety shall increase proportionately by increasing the strength of security instructions. In some cases, more explicit constraints may simply increase verbosity without altering the underlying implementation patterns. For some architectures, security language may be interpreted as guidance style instead of a strict requirement.

# **Implications for Practitioners** 

## Treat the Choice of LLM as a Security-Critical Decision 

Our results show that by switching models, vulnerability rates shift by 10%–15% across domains. This shows that while keeping the prompts 

and tasks constant, organizations and developers can change vulnerability counts by large. This magnitude is comparable to changing cryptographic libraries or third-party dependencies. This is a security-critical decision that requires evaluation, documentation, and periodic reassessment rather than ad hoc adoption.

## Use Security-Aware Prompts as the Default Policy

## Integrate Human Review for Sensitive Modules 

LLM-generated code should not be merged automatically for components that require extra security clearance. For code related to authentication, authorization, file system operations, or network communication, organizations should 

require mandatory human review. There should be a human reviewer to verify that the generated code satisfies the organizational and domain security requirements. LLMs should be positioned as productivity aids rather than autonomous code authors.
