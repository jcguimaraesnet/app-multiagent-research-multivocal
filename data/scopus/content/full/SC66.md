# An Empirical Study of Exploring the Capabilities of Large Language Models in Code Learning 

Shangqing Liu , Daya Guo , Jian Zhang , Wei Ma , Yanzhou Li , and Yang Liu , _Senior Member, IEEE_ 

**_Abstract_ —Since the advent of ChatGPT, large language models (LLMs) have attracted widespread attention from academia and industry. They have also brought significant changes to software engineering. However, until now, there has been a lack of comprehensive studies comparing LLMs with previous smaller code pretrained models. To address this gap, we conduct a study in this paper to illustrate the performance of LLMs in different software engineering tasks. Specifically, we select three open-source large language models, CodeGen, LLaMA, and StarCoder, for the research targets, and our study is conducted from four aspects, including code syntax understanding, code semantic reasoning, encoding representation quality, and adaptation performance for different software engineering tasks to compare LLMs with previous code pre-trained models. Four aspects build on each other, forming important components of AI for Software Engineering. We conclude that: (1) Compared with previous smaller pretrained models like CodeBERT, LLMs exhibit distinct trends in how they learn code syntax or semantics as the number of layers increases. Additionally, mastering code semantics proves to be more challenging, with semantic information usually learned in the final layers; (2) Causal decoder architecture with left-to-right attention masking does not perform well in zero-shot tasks; (3) For classification tasks, the mean vector representation generated by LLMs over a sequence tends to outperform the last token representation in the sequence; (4) Incorporating parameterefficient fine-tuning techniques into LLMs for downstream tasks can help LLMs achieve better performance than previous code pre-trained models on code generation tasks but may not be optimal in some code understanding tasks; (5) LoRA emerges as a more effective PEFT technique for LLMs in downstream coderelated tasks. We hope these findings will better guide future researchers in designing more powerful code models.** 

**_Index Terms_ —Large language models, code pre-trained models, probing analysis, parameter-efficient fine-tuning.**

## I. INTRODUCTION 

INCE the emergence of ChatGPT [1], large language mod- **S** els (LLMs) have rapidly become a hot topic in academia and industry. Almost every day, we can find the latest news about these large language models on various social media platforms. For example, which institution has released a model with how many parameters, or in which tasks LLMs have achieved or surpassed human-level performance, etc. Undoubtedly, models like ChatGPT have brought disruptive impacts to our studies and daily lives. They have even reshaped many research areas. One representative example is the field of software engineering, where we found that LLMs are usually capable of assisting software developers in accomplishing different developing tasks such as code development [2], repairing vulnerable code [3] and generating unit tests [4]. The powerful capabilities of these models learned from massive amounts of data significantly improve the productivity of software developers and reduce their workload [5]. Although large language models offer numerous benefits for software developers, a systematic analysis of their capabilities is needed for discussion. 

Existing work has largely focused on demonstrating the effectiveness of individual LLMs on specific tasks, such as function synthesis or code summarization. Moreover, smaller models like CodeBERT [6] and UniXcoder [7], although widely used, they have not always been included in direct comparisons with LLMs, leading to fragmented insights and a lack of clarity about the true impact of model scale. As organizations and developers weigh the cost, and resource demands of deploying these models, it becomes crucial to provide a _rigorous, applesto-apples evaluation across model families_ . 

This paper presents a comprehensive empirical study that addresses this gap by comparing several large and small code models across multiple dimensions of code learning and adaptation. Unlike prior work that often centers around a specific downstream task, we adopt a broader view, organizing our study into four key dimensions: (1) syntactic understanding, (2) semantic reasoning, (3) representational quality, and (4) adaptation performance. By systematically varying the task types and evaluation granularity, we aim to identify how model size, architecture, and adaptation technique interact to shape performance. Concretely, our study is organized around the following guiding questions: 

- **Syntax Understanding:** To what extent do different models internalize the structural properties of programming languages? We design a suite of AST-based probing tasks

For semantic reasoning, we first employ two probing tasks from Troshin et al. [12] to explore whether LLMs can identify variable names and variable misuse. Both probing tasks are designed to require models to learn code semantics for correct prediction. In addition, we also build the data flow graphs for the Python dataset in CodeSearchNet [13]. Firstly, we randomly select some semantic-equivalent samples from the dataset of clone detection [14] and plot their space distribution produced by LLMs without finetuning for visual inspection. Secondly, we compare the results of LLMs with some existing baselines in the zero-shot code-to-code search task from UniXcoder [7] to evaluate the performance of code fragment embeddings for statistical analysis. For the last scenario of parameter-efficient fine-tuning for different software engineering tasks, we select **four** typical PEFT techniques (i.e., LoRA [15], Prefix Tuning [16], Prompt Tuning [17] and P-Tuning [18]) and incorporate them into different LLMs to separately validate their performance on **five** code-related tasks including code understanding tasks (i.e., clone detection and code search), code generation tasks (i.e., code summarization and code generation) and code completion. 

Our empirical study yields several key findings that offer guidance for both model analysis and downstream deployment. First, we observe that LLMs are capable of learning both

## _B. Semantic Reasoning_ 

Code semantic understanding is essential for code neural networks to achieve great performance on different code-related tasks such as vulnerability detection [26] and clone detection [27]. Many existing works have proposed to utilize different code structures [26], [28], [29], [30] to facilitate neural networks learning code semantics. With the widespread usage of code pre-trained models, advanced works [12], [20] have been conducted to explore whether code pre-trained models can automatically learn code semantics without using complex code structures in pre-training. Similarly, we analyze three semantic probing tasks to explore the capabilities of LLMs in learning code semantics. 

_1) Variable Misuse Prediction:_ The first probing task for semantic analysis is variable misuse prediction, which aims to detect the misused variable in a code snippet [31]. We introduce this task as it requires the model to learn the semantics of the entire code to make correct predictions regarding variable misuse. We use the open-source dataset from Troshin et al. [12] for analysis. Specifically, in this dataset, the wrong identifier name is randomly copied from another identifier from a code snippet to construct a misused code snippet. A correct code snippet for each wrong snippet is also provided to formulate this task as a binary classification. In total, after filtering out the tokenized code with a length greater than 512, 429,068 samples are retained. As the correct and wrong identifier may consist of multiple subtokens by tokenization, we use the mean hidden representation of these subtokens appearing at their corresponding code representation positions in a layer of LLMs to obtain the vectors for the correct and wrong identifier and further train a binary linear classifier for analysis.

## _C. Encoding Representation Quality_ 

Zero-shot encoding vector representation is essential for LLMs, especially in scenarios where labeled data is limited or unavailable. In many real-world software engineering applications such as code search or vulnerability detection in proprietary codebases, fine-tuning may be infeasible due to insufficient task-specific data, privacy concerns, or computational constraints. In such cases, LLMs must rely on their pre-trained knowledge to generate meaningful representations that generalize well across tasks. Therefore, understanding and improving the quality of zero-shot embeddings is crucial for maximizing the utility of LLMs in low-resource and out-of-distribution settings. Furthermore, some works [33], [34] have confirmed that robust vector representation can effectively defend against adversarial attacks. Lastly, most LLMs employ autoregressive Transformers with the next-token prediction (i.e., unidirectional attention) as the learning objective, which differs from BERTstyle code pre-trained models that use bi-directional attention for pre-training. It is necessary to compare the qualities of the generated embedding vectors by different models. We designed two different tasks for analysis. 

_1) Learnt Space Visualization:_ The first task aims to visualize the space distribution using the embedding vectors for inspection. Specifically, we use the clone detection (POJ-104) dataset provided by CodeXGLUE [35]. The reason for selecting this dataset is that it has semantically equivalent programs with different implementations so that we can compare their embedding vector qualities, although they may have different implementations. This dataset consists of 104 programming problems and each problem has 500 different implementations. We randomly selected five different problems<sup>1</sup> . Each problem has 100 different implementations, and we take them as the input to LLMs without finetuning to obtain their hidden representations _H ∈_ R<sup>_l×d_</sup> , where _l_ is the code length and _d_ is the dimensional length. ## _D. PEFT for Adaptation Performance_ 

For different downstream tasks, it is a common practice to finetune the code pre-trained models to achieve the best performance [35]. However, it is expensive and infeasible to conduct full-parameter fine-tuning on LLMs as these models are larger than previous code pre-trained models. For example, StarCoder [10] and LLaMA-7B [40] have 40 and 32 Transformer layers, respectively; in contrast, CodeBERT [6] and UniXcoder [7] only have 12 Transformer layers. Hence, to ensure LLMs can learn domain knowledge, some parameter-efficient fine-tuning techniques [15], [16], [17], [18], [40], [41] are proposed. In this work, we select four typical PEFT techniques and incorporate them into LLMs to conduct parameter-efficient fine-tuning on different SE tasks for comparison. 

## _1) Adapters:_ 

**LoRA** [15]: The idea behind low-rank adaption (LoRA) is to train some dense layers in LLMs indirectly by optimizing rank decomposition matrices of dense layers’ change while keeping pre-trained weights frozen. Inspired by the “intrinsic dimension” [42], for a pre-trained weight matrix _W_ 0 _∈_ R<sup>_d×k_</sup> , where _d_ and _k_ are the input/output dimension length, the weight updates can be expressed as _W_ 0 + Δ _W_ = _W_ 0 + _BA_ where _B ∈_ R<sup>_d×r_</sup> , _A ∈_ R<sup>_r×k_</sup> and the rank _r ≪ min_ ( _d, k_ ). During training, _W_ 0 is frozen and does not receive any gradient updates, while _A_ and _B_ contain the trainable parameters. For a forward pass operation in LLMs, in contrast to the typical full-rank matrix multiplication in the full-parameter fine-tuning where the output vector is _h_ = _W_ 0 _x_ , the output vector is _h_ = _W_ 0 _x_ + Δ _Wx_ = _W_ 0 _x_ + _BAx_ 

where _x_ is the input vector. As LoRA only optimizes some subsets of weight matrices in LLMs, the trainable parameters are reduced significantly. 

**Prefix Tuning** [16]: It keeps the parameters of LLMs frozen while optimizing a sequence of continuous vectors, namely prefix. Specifically, it prepends a sequence of continuous vectors to the input. LLMs can attend to the prefix at every transformer layer. These prefix vectors do not correspond to real tokens, and the prefix-tuning only optimizes the prefix vectors, which leads to a small overhead to downstream tasks. 

**Prompt Tuning** [17]: Different from prefix tuning, which prepends a sequence of vectors to the input, prompt tuning aims to prepend a series of tokens to the input text and further optimize the embeddings of these prepend tokens for update. The model parameters are frozen. As only the embeddings of the prepend tokens are optimized, the trainable parameters are reduced significantly. 

**P_tuning** [18]: It leverages a few continuous free parameters as the input to LLMs and then optimizes the continuous parameters for tuning. Specifically, to obtain the continuous parameters, p-tuning proposes to use a very light neural network such as long-short term memory networks (i.e., LSTM) or multilayer perceptron (MLP) and then update its parameters, but the parameters of LLMs are frozen. Because the extra neural network is very small, the trainable parameters are reduced significantly. 

_2) Evaluation Tasks:_ To evaluate PEFT techniques, we select some typical tasks from CodeXGLUE [35]. Following the categories defined in CodeXGLUE, they can be grouped into code understanding tasks, code generation tasks, and code completion. 

**Code Understanding Tasks** : They require the neural networks to learn the code representations and further make a prediction based on the learned representations. For example, the output is predicted to be 0 or 1. We select two different tasks, i.e., code clone detection and code search for evaluation. 

**Code Generation Tasks** : They require neural networks to learn sequence representations and further generate new sequences based on the learned representations. We select two tasks, i.e., code summarization (PL2NL) and code generation (NL2PL), for evaluation. 

**Code Completion** : Different from code understanding and code generation tasks, which first obtain the representations over the entire sequence by neural networks and then use them for prediction or generation, code completion aims to complete the code from the previous text where the text is partial code. A typical neural architecture for this task is the decoder-only Transformer. 

For these downstream SE tasks, we compare the results with the reported values in UniXcoder [7] and CodeT5 [36]. We further compare these downstream tasks with LLMs without fine-tuning to verify the importance of fine-tuning. Similar to UniXcoder, we use the datasets of different tasks provided in CodeXGLUE [35] for evaluation. ## III. EXPERIMENTAL SETUP 

We propose four research questions to explore the capabilities of LLMs in code learning, they are formulated as follows: 

- **RQ1:** What is the performance of large language models in understanding code syntax? 

- **RQ2:** What is the performance of large language models in reasoning code semantics? 

- **RQ3:** How is the quality of the embedding vectors produced by large language models? 

- **RQ4:** What is the performance of LLMs in different downstream tasks and which kind of PEFT technique is better? 

## _A. Evaluation Metrics_ 

As probing tasks are classification problems, we use accuracy as the evaluation metric. For different code-related tasks, we use the same evaluation metric with UniXcoder. Specifically, we use MAP@R for POJ-104, F1 for BigCloneBench, MRR for code search, smoothed BLEU-4 [48] for code summarization, exact match (i.e., EM) and BLEU-4 for code generation, EM and Levenshtein Edit Similarity [49] (i.e., Edit Sim) for code completion. 

## _B. Model Settings_ 

For the probing analysis of code syntax understanding and semantic reasoning, one fully connected linear layer classifier is trained on the code representations extracted from each layer of LLMs for analysis. We configure different hyper-parameters in different adapters for PEFT. Specifically, in the LoRA adapter, we set the rank _r_ (See Section II-D1) to 8 to optimize the weight matrices in the MLP layer and self-attention module in each Transformer layer of LLMs. In prefix tuning, the number of prefix vectors is set to 12, and we use a forward neural network for stable optimization, following Li et al. [16]. In prompt tuning, a series of tokens are needed to prepend to the input. We refer to the official documentation from the pubic repository [50], which has over 95k stars for designing the prompt. The prompt are also shown in our public code [19]. In P-tuning, we use a multi-layer perceptron as an extra neural network to optimize their parameters.We set the batch size to 8 for CodeGen-6.1B, LLaMA-7B, and 4 for StarCoder in tasks<sup>2</sup> . ## V. DISCUSSION 

## _A. Relations Between Research Questions_ 

This work is structured around four research questions, which progressively explore the application of LLMs in software engineering. Firstly, a program is a collection of instructions that comply with specific syntax rules. Even the most complex programs adhere to these rules, making the exploration of LLMs’ capability to learn syntax rules a starting point. 

Building on this, the semantics of a program are the higherlevel issue. Code implementations with the same semantics may appear differently, making it significant to explore the capability of LLMs to learn the semantics of code. Only when LLMs accurately learn program semantics can they perform exceptionally well across various software applications. It is the foundation issue when employing these LLMs, which is worth discussing in depth. 

After that, since deep learning models fundamentally operate with vectors, it is crucial to explore the quality of vector representations of the programs inputted into the model. Previous works [33], [34] have confirmed that robust vector representation can effectively defend against adversarial attacks for code. Furthermore, these vectors are directly utilized in some zeroshot software applications, such as code search [7]. Therefore, investigating the quality of zero-shot vectors generated by LLMs holds significant importance. 

Finally, discussing the performance of LLMs in various software engineering applications under fine-tuning is a direct research question. This is because the purpose of proposing these models is to better use them in assisting software development. Since different applications often require specific domain knowledge, fine-tuning domain-specific datasets is one of the solutions to ensure the model equips domain knowledge. However, full-parameter fine-tuning is not feasible for LLMs. Hence, we propose to adopt parameter-efficient fine-tuning for downstream tasks for exploration. 

## _B. Insights_ 

One of the most striking findings from our study is the consistent trend that **middle layers** tend to yield the most effective representations for syntax-based probing tasks across both smaller code-pretrained models (e.g., CodeBERT) and large LLMs (e.g., CodeGen, LLaMA, StarCoder). However, our results reveal a novel contrast: while smaller encoder-based models peak sharply at intermediate layers, LLMs exhibit **flatter accuracy profiles** . This suggests a shift in representational distribution. We hypothesize that the differences stem from **pretraining objectives** : bidirectional masked language 

modeling may encourage more local, structure-sensitive representations than left-to-right decoding, which prioritizes sequential dependencies. This highlights the importance of architectural alignment when transferring language models to code domains. 

In RQ3, our experiments suggest that embeddings from contrastively pre-trained models (e.g., UniXcoder, ContraBERT) form more distinct and semantically coherent clusters than embeddings from LLMs. LLMs lag behind in zero-shot settings, indicating that **next-token prediction alone may be insufficient for learning semantically discriminative embeddings** . Our results confirm that **contrastive objectives strengthen the embedding space** by encouraging proximity between semantically equivalent code snippets while separating dissimilar ones. Furthermore, we observe that **mean pooling over token representations outperforms using the last token representation** for all LLMs. In classification or search scenarios, the final token may overly emphasize recent context, leading to degraded generalization. These findings provide actionable guidance for designing embedding-based code analysis systems using LLMs. 

In RQ4, applying PEFT techniques consistently improves performance across all models. However, the magnitude of improvement varies by task. The results of code search indicate that despite the scale of LLMs, their autoregressive architecture may be fundamentally less suited to understanding-oriented tasks that require holistic analysis of the input, for which bidirectional encoding (as used in BERT-style models) is more appropriate. Thus, a promising future direction could involve hybrid architectures or dual-objective training that combines autoregressive and masked modeling for improved generality. Among all PEFT techniques evaluated, LoRA consistently yields the best results across a wide range of tasks. We hypothesize that this effectiveness arises from LoRA’s architectural design, which fine-tunes an introduced low-rank subspace of the model’s weight matrices. In contrast, prompt-based methods (Prompt Tuning, Prefix Tuning, P-Tuning) modify the input or introduce external modules, which may limit their expressiveness for complex tasks. 

## _C. Implication_ 

Through our research, we have found that compared to understanding code syntax, it is more challenging for LLMs to learn program semantics. Therefore, we recommend integrating program semantics into the pre-training phase rather than relying solely on simple natural language-based pre-training objectives like next-token prediction. A viable approach is explicitly extracting program semantics, such as using the code property graph [26], [28], and implementing semantic-aware pre-training goals by designing different training objectives. Furthermore, during the pre-training phase, we also recommend incorporating contrastive learning objectives [34] to enhance the robustness of the LLMs. However, while these potential solutions could further enhance the model, we also recognize

LLMs have significant advantages in generation tasks. This observation reveals that LLMs do not perform better in any task than smaller models. These findings also indicate that to select a proper PEFT technique, apart from the task characteristics, we also need to consider the characteristics of the used datasets. In the future, we plan to explore the automated selection strategy from these PEFT techniques. 

## _D. Threats to Validity_ 

One limitation of our probing setup lies in the design of the pair prediction task, particularly the strategy for generating negative examples. In our current formulation, negative samples are randomly selected nodes from the same program that do not share a dependency with the given head node. While this ensures class balance, such randomly sampled negatives are often structurally and semantically distant from the head node, potentially making the classification task overly simple. As a result, models may achieve high accuracy. Despite this simplicity, we emphasize that the pair prediction task still yields two meaningful and nontrivial findings. First, we consistently observe that the highest accuracy is achieved at intermediate layers across all models, confirming prior evidence that middle layers tend to encode syntactic or local structural information. Second, we observe distinct layer-wise accuracy trends between small code-pretrained models and large LLMs, suggesting architectural and pretraining differences in how code representations are distributed across layers. These trends are central to our study. In the future, we plan to incorporate more challenging “hard negatives” e.g., nodes from nearby or structurally similar contexts for evaluation. 

The second potential threat lies in the designed prompts that are used for prompt tuning in different code-related downstream tasks. To mitigate it, we refer to the official website [50] to 

design the prompts for tuning. These may not always be the most representative ones, although the website has over 95k stars. We plan to investigate the effect of other prompts in the future. 

Another threat lies in the selected three large language models, CodeGen-6.1B, LLaMA-7B, and StarCoder, for the experiments. We agree that there are many open-source large language models [56], [57], [58] and even for CodeGen and LLaMA, there are different versions such as CodeGen-16.1B, CodeGen2 [59], LLaMA 2 [9], LLaMA 3 [60]. As our work progressed, many latest models [61], [62], [63] emerged simultaneously. We cannot enumerate all open-source LLMs for experiments. To mitigate this, we select three representative ones CodeGen, StarCoder, and LLaMA. All of them have attracted significant attention from both academia and industry, underscoring their representativeness within the current LLM landscape. In particular, their respective Google Scholar citation counts have reached to 1,317 (CodeGen), 1,081 (StarCoder), and 17,391 (LLaMA), reflecting their widespread adoption and impact. Additionally, in selecting representative models, we also considered the practical constraint of computational feasibility. The chosen models strike a balance between scale and accessibility, allowing us to conduct a thorough and meaningful evaluation within our available computing resources. In addition, we want to emphasize that these LLMs share almost the same model architecture, i.e., the causal decoder for pre-training. Thus, although our selected models may not be the latest, the conclusions are consistent. 

There is also a potential threat related to the evaluated downstream code-related tasks. To mitigate it, we select different tasks, including clone detection, code search, code summarization, code generation, and code completion, from different categories i.e., code understanding and code generation, to ensure the comprehensiveness of evaluation. 

A potential threat lies in the fact that different PEFT techniques do not necessarily use the same amount of FLOPs or the same total compute spent. This is due to the inherent differences between the methods themselves, for example, LoRA updates low-rank matrices, whereas prompt tuning updates the embeddings of prepended sequences, which makes it difficult to ensure they use the same number of updated parameters or have the same compute spent. 

## VI. RELATED WORK 

## _A. Large Language Models_ 

Large language models (LLMs) usually refer to Transformer models that contain billions of parameters. Compared with previous pre-trained language models [37], [64], LLMs usually have more parameters and are trained with more data, hence they tend to exhibit stronger capacities in understanding natural language and solving complex tasks. Furthermore, when the size of LLMs exceeds a certain level, LLMs usually exhibit some surprising abilities such as emergent ability [65], incontext learning [66], step-by-step reasoning [67]. 

In terms of the model architecture for LLMs, most LLMs adopt a causal decoder for training, where a causal decoder

As representative language models of this architecture, GPT-series models [66], [68], [69] are typical causal decoder models. In addition, BLOOM [56], Gopher [70], StarCoder [10], OPT [57], PaLM [71], CodeGen [8] and LLaMA [9] are also based on this architecture. In terms of the training data used, the vast majority of LLMs adopt general-purpose pre-training data from webpages, books, and conversational text to ensure the model’s generalization abilities. In addition, some LLMs purely use domain-specific data to obtain a domain-specific model, such as StarCoder [10] and AlphaCode [74], which both use code data from GitHub for pre-training. 

## _B. Probing Analysis for Code Models_ 

The probing technique is an effective way to investigate the learnt property of deep learning models. Probing analysis aims to determine what regularities are captured within dataderived representations. The underlying principle of probing posits that if a simple classifier, such as a linear classifier, can adequately perform a task using these representations, then it must inherently possess informative features about that task. 

Several works [11], [12], [20], [21], [22], [23], [75] have been conducted to analyse the learned knowledge from code models. For example, Wan et al. [23] proposed a structural probing approach based on AST to investigate whether the syntax structure is embedded in the linear-transformed contextual word embedding of CodeBERT and GraphCodeBERT. CodeSyntax [11] constructed the probing benchmark, including the extracted relation types among different code components from Python and Java, to evaluate the performance of BERT-style models on code syntax understanding. Besides the code syntax analysis, Troshin et al. [12] proposed to design several code semantic probing tasks to evaluate several code pre-trained models, including CodeBERT, GraphCodeBERT, PLBART, CodeT5 and CodeGPT2. Compared to these works, which mainly focus on analysing code pre-trained models such as CodeBERT, we target large language models and compare them with smaller code pre-trained models in learning code syntax and semantics. 

We compare our study with the most closely related work by Ma et al. [75] in terms of scope, methodology, and key findings. Notably, our work adopts a broader perspective, encompassing syntax learning, semantic learning, embedding representation quality, and parameter-efficient fine-tuning (PEFT) for downstream software engineering (SE) tasks. In contrast, 

Ma et al. primarily focus on probing analysis. Accordingly, our methodological design diverges: we employ five probing tasks to analyze syntactic and semantic understanding, utilize both visualization and zero-shot code-to-code retrieval to assess embedding representation quality, and evaluate multiple PEFT techniques across a variety of SE tasks. By comparison, Ma et al. adopt structure-reconstruction probing tasks and attention pattern analysis to examine how models encode syntactic and semantic information, without addressing downstream task performance or fine-tuning strategies. In terms of findings, beyond the shared observation that semantic understanding is more challenging than syntactic processing, our study uniquely reveals that LLMs and code pre-trained models exhibit distinct learning patterns for syntax and semantics. Additionally, our broader evaluation enables us to derive further insights in areas—such as embedding quality and PEFT effectiveness— that are not covered in Ma et al.’s study. 

## _C. Parameter-Efficient Fine-Tuning of LLMs for SE_ 

As the scale of models continues to expand, the traditional method of full-parameter fine-tuning to ensure that models learn domain knowledge in various software engineering scenarios becomes increasingly impractical due to limited computational resources. Therefore, increasing work is starting to focus on parameter-efficient fine-tuning. PEFT enables models to learn domain knowledge by training only a few parameters while keeping most of the model’s parameters unchanged, significantly reducing the demand for computing resources. 

There are several works [76], [77], [78], [79] that have employed PEFT techniques for different software engineering applications. For example, Li et al. [76] proposed to fine-tune four pre-trained LLMs using four different PEFT methods for automated program repair. WEYSSOW et al. [77] proposed to fine-tune LLMs with LoRA, IA3, Prompt tuning and Prefix tuning for Python code generation. Liu et al. [78] proposed the use of two common PEFT techniques, Adapter Tuning and LoRA used on code pre-trained models to evaluate their performance on the task of commit message generation and JustIn-Time Defect Prediction. In addition, LLaMA-Reviewer [79] utilized prefix-tuning on LLaMA for automated code review. Compared with these works, which focus on using different PEFT techniques for various SE scenarios, in this work, following the previous works [35], we categorized typical software engineering tasks into code understanding tasks and code generation tasks. We systematically demonstrated the performance of various PEFT techniques across different types of code-related tasks, including clone detection, code search, code summarization, code generation and code completion on the popular benchmark CodeXGLUE [35]. In addition, a recent work [80] explored different PEFT techniques for code models such as CodeBERT and GraphCodeBERT. Compared with this work, which focuses on smaller code pre-trained models, we examine the performance of PEFT on LLMs. We compare the performance between different LLMs and code pre-trained models and further achieve new state-of-the-art performance on various code generation tasks with LLMs.

## VII. CONCLUSION 

In this study, we present an in-depth empirical investigation into the capabilities of LLMs for code learning. Through a series of carefully designed probing, visualization, and finetuning experiments, we arrive at the following key findings: (1) We observe that LLMs are capable of learning both syntactic and semantic aspects of code; however, they do so in ways that differ from smaller pre-trained models such as CodeBERT; (2) Models with causal decoder architectures and left-to-right attention masking underperform in zero-shot code understanding tasks, suggesting architectural limitations in capturing bidirectional dependencies; (3) For classification tasks, we find that using mean-pooled sequence representations outperforms relying on the final token, highlighting an important consideration in feature extraction from LLMs; (4) While PEFT can significantly enhance LLM performance on code generation tasks i.e., surpassing smaller code models, it does not always yield optimal results for code understanding tasks. We hope these insights will inform both practitioners seeking to apply LLMs in software development and researchers aiming to design better code models in the future.
