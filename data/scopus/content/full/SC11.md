##### **1. Introduction** 

The era of developers manually authoring software line-by-line is almost over. With the release of OpenAI’s Codex [1] and the subsequent integration of Large Language Models (LLMs) into development environments via tools like GitHub Copilot and Cursor, software engineering has undergone a fundamental paradigm shift. We have moved from a discipline of _writing_ to one of _prompting_ , editing, and curating. This transition offers unprecedented productivity gains, automating up to 46% of coding tasks by some estimates [2]. However, it has simultaneously altered the concepts of software provenance. In this new reality, the question ‘‘Who wrote this code?’’ is no longer a 

simple inquiry to find the right author. Rather, it is a complex forensic challenge with profound legal, security, and academic implications. 

The urgency of this challenge extends far beyond academic integrity to professional development practices. In the industrial sector, the opacity of AI-generated code introduces severe supply chain risks. If a proprietary LLM inadvertently reproduces snippets of copyleftlicensed code like GPL into a closed-source commercial product, the resulting license violation can lead to disastrous intellectual property disputes [3]. Furthermore, the security implications are serious as well. Recent studies suggest that AI-generated code often introduces subtle vulnerabilities or ‘‘hallucinated’’ packages that human reviewers may

overlook, necessitating a mechanism to trace bad code back to the originating model [4]. Without a reliable method to distinguish between human-crafted logic and machine-synthesized artifacts, the software supply chain becomes untraceable and potentially untrustworthy. 

Prior research on detecting AI-generated content has largely focused on natural language within the NLP domain. However, extending this viewpoint to source code is insufficient, as source code exhibits fundamentally different syntactic and semantic properties. In contrast to natural language, small lexical substitutions in code can significantly impact program correctness and execution. Source code behaves very differently from natural language. Even a small change at the token level can break syntax or functionality. In some cases, it may also change the program’s behavior in ways that are hard to predict. 

Standard text detection methods, such as perplexity-based or statistical detectors like DetectGPT [5], typically rely on signals such as ‘‘burstiness’’ or ‘‘perplexity’’. While these signals work well for natural language, they are fundamentally less suitable for source code due to its structural rigidity, low entropy, and strict functional constraints that invalidate generic text perturbations. Although code-specific adaptations (e.g., DetectCodeGPT [6]) show improvements, they remain less effective than methods explicitly modeling syntax and semantics. This makes it necessary to shift attention towards more rigorous solutions tailored to software engineering settings. 

This literature review takes a close look at the growing area of **code provenance** , which focuses on identifying the origin and development history of software artifacts in the presence of AI-generated code. Instead of treating code as just another form of text, this review approaches the problem from a Software Engineering (SE) perspective. Earlier surveys often group code with general text watermarking methods. In contrast, we focus specifically on how these ideas apply to real software systems. Through this lens, we observe a clear split in the existing literature, which can be described as a fundamental dichotomy: 

1. **Active Provenance (Watermarking):** These methods modify the generation process itself. A signal is embedded while the code is being produced, for example, by biasing token logits. The goal is to make the origin of the code verifiable later. 

2. **Passive Provenance (Authorship Attribution):** These approaches work after the code has already been generated. They rely on forensic analysis of the code, focusing on stylistic or structural patterns that may reveal its origin. 

We reviewed **31** primary studies published between 2020 and 2025 focusing on the methods that respect the functional constraints of executable software. Our analysis shows a clear trade-off in the field between the two provenance approaches. Passive methods are easier to deploy, but they often fail after standard refactoring. In contrast, active watermarking supports stronger ownership claims, yet it can introduce what is often described as ‘‘technical debt’’, making code harder to read and maintain over time. 

This study makes three primary contributions. First, we introduce a unified taxonomy that organizes existing techniques by their level of operation, ranging from raw tokens to Abstract Syntax Trees (AST). Second, we examine the so-called ‘‘access paradox’’. This highlights the mismatch between watermarking methods that rely on white-box access and the black-box nature of most commercial APIs. Third, we show how fragile many current approaches are when exposed to standard software engineering workflows. Tools that developers regularly use to improve code quality, such as linters and formatters, often end up disrupting or removing provenance signals.

Section 4 presents and synthesizes the results. Section 5 discusses the implications for real-world software development. Finally, Section 6 outlines open challenges and directions for future research. 

##### **2. Preliminaries** 

To provide a foundation for the systematic review, this section defines the core concepts of watermarking, authorship attribution, and code provenance. We delineate the theoretical frameworks and mathematical formulations that underpin the methodologies analyzed in this study. 

##### _2.1. Source code watermarking_ 

**Watermarking** is the process of embedding a unique, identifying mark into physical or digital content to assert ownership, verify authenticity, and protect intellectual properties [7]. This idea has long been used in areas such as multimedia and software protection. In the context of generative AI, watermarking takes a more specific form. Here, it describes techniques that slightly adjust the sampling behavior of a Large Language Model (LLM) during generation. These adjustments leave behind a detectable pattern in the output text or code, which can later be used to identify its origin [8]. 

**Source code watermarking** differs from natural language watermarking in one critical aspect: it must preserve **functional correctness** . For natural language, small semantic changes are often acceptable. Source code does not allow this flexibility. The generated program must remain syntactically valid and executable. Even small mistakes may lead to compilation failures or incorrect runtime behavior. As highlighted in recent studies [9,10], effective code watermarking schemes must introduce provenance signals without altering the underlying program logic or breaking the build process. 

At a high level, a watermarking scheme embeds a detectable signal during the generation process and later verifies its presence. Formally, let  denote a generative model. A watermarking scheme for  can be described as a tuple ( _,_ ), where  is the watermark generation procedure and  represents one or more verification procedures. 

##### _2.1.1. Watermark generation_ 

**Generation (**  **):** Given a prompt _𝑥_ , a generative model , and a secret watermarking key _𝑘_ , the watermarking algorithm produces a watermarked output _𝑦𝑤_ : 

##### _𝑦𝑤_ ← ( _𝑥,_  _, 𝑘_ ) 

In practice, this process slightly biases the model’s sampling distribution _𝑃_ ( _𝑦_ ∣ _𝑥_ ). Common techniques include Green–Red token partitioning [8] or entropy-based control strategies [9]. These biases are designed to be subtle. The generated code should remain natural and functionally correct, while still carrying a hidden provenance signal. In most source code settings, the watermark is a _zero-bit watermark_ . The goal is to indicate presence rather than embed an explicit message. 

##### _2.1.2. Watermark verification and attribution_ 

Generation constitutes the core mechanism of watermarking. However, to ensure practical usability and robustness across diverse scenarios, watermarking frameworks incorporate additional algorithms for watermark detection and classification. Here, we outline the key algorithms required for a comprehensive watermarking approach as mentioned in the previous study [11]: 

1. **Detection:** The detection function tests whether a given code artifact _̂𝑦_ contains a watermark associated with key _𝑘_ : 

   - _𝑑_ ← detect( _̂𝑦, 𝑘_ ) _, 𝑑_ ∈{Watermarked _,_ Unwatermarked} 

Detection is usually performed using statistical hypothesis testing. For example, the detector may compute a z-score to check whether the frequency of preferred (‘‘green’’) tokens significantly exceeds random expectation [9,12]. Detection is commonly used for large-scale auditing or misuse monitoring.

3. **Attribution:** Attribution aims to determine whether a specific code artifact was generated by a particular model or system using a known key:

Usually, the same secret key is used for generation, detection, and attribution. In other designs, these keys are separated to improve security or limit misuse. Overall, effective source code watermarking requires careful coordination between watermark embedding and verification, while maintaining the strict functional constraints imposed by executable software. 

##### _2.2. Authorship attribution and stylometry_ 

**Authorship Attribution** is the process of inferring the author of a document based on its linguistic characteristics. In Software Engineering, this is often referred to as **Code Stylometry** , which relies on the hypothesis that programmers (or generative models) exhibit unique, consistent coding habits [14,15]. Stylometry analyzes code through various feature sets including: 

- **Lexical Features:** The usage frequency of keywords, variable naming conventions (e.g., snake_case vs. camelCase), and comment styles [16]. 

- **Syntactic Features:** Patterns derived from the Abstract Syntax Tree (AST), such as the depth of nesting, specific control flow structures, or n-grams of AST nodes [17,18]. 

- **Latent Representations:** Modern approaches utilize neural embeddings like CodeBERT, UniXcoder to capture high-dimensional semantic signatures that distinguish between human-written and AI-generated code [19–21]. 

While watermarking is an active process, attribution is **passive** ; it does not modify the code but analyzes existing artifacts to calculate the likelihood _𝑃_ ( _𝐴𝑖_ | _𝐶_ ) that a specific author (or model) _𝐴𝑖_ generated code _𝐶_ . 

##### _2.3. Code provenance_ 

**Code provenance** denotes the overall history of a software artifact, including the code origin, the author, and the process by which it has evolved. In traditional software engineering, provenance is mainly tracked through version control systems, such as Git history [22]. With the introduction of Large Language Models (LLMs), this view is no longer sufficient. Now, provenance also needs to answer whether a piece of code was generated by AI. If that is the case, which specific model produced it is crucial information to investigate the authorship [23]. 

From this discussion, we can understand how code provenance works as a common objective rather than a single method. Different identification techniques approach the problem in different ways. Watermarking and authorship attribution rely on distinct technical ideas. 

Still, both aim to establish the lineage of a code artifact. Nowadays, the identical motive of both of the approaches is crucial as software development increasingly depends on automated tools and AI-assisted workflows. 

**Watermarking** supports provenance by explicitly asserting origin. A verifiable signal is embedded during the code generation process. This signal can later be checked to confirm that the code was produced by a particular model. In practice, this acts as a digital chain of custody. It allows model owners to claim ownership and track their generated code when it appears outside controlled environments [24,25]. 

In contrast, **authorship attribution** attempts to recover provenance directly from the code itself without relying on any intentionally embedded signal. It assumes that the generation process leaves behind identifiable patterns. These patterns may appear in lexical choices, syntactic structure, or other stylistic properties. By analyzing such signals, attribution methods aim to infer the likely origin of the code, even when no deliberate watermark is present [14,16]. 

##### **3. Related studies** 

The rapid spread of generative AI has increased the need for reliable methods to track provenance and detect AI-generated content. As a result, several survey papers have appeared that summarize existing identification techniques. Most of these reviews study the problem from a Natural Language Processing (NLP) perspective. They often focus on the statistical behavior of generated text rather than on executable artifacts. 

Liu et al. [26] and Zhang et al. [27] present detailed taxonomies of watermarking methods for Large Language Models (LLMs). Their surveys mainly distinguish between ‘‘soft’’ approaches that modify token logits during generation and ‘‘hard’’ approaches applied after generation. In a similar direction, Wang et al. [28] and Cao et al. [29] study intelligent identification systems across multiple modalities, including text, audio, and visual content. 

While these works provide useful classifications of watermarking strategies, their evaluation remains largely natural language-centric. Performance is often measured using metrics such as perplexity [30] or BLEU [31], primarily tailored for natural language. However, in the software engineering domain, they do not capture the constraints of software artifacts. In Software Engineering, source code must satisfy strict syntactic and semantic rules. As a result, insights from NLP-focused surveys do not directly transfer to the domain of source code. 

Besides investigating artifact provenance, a separate stream of research focuses on Intellectual Property (IP) protection for the models themselves. Boenisch [32] conducts a systematic review of model watermarking for neural networks. They investigated the techniques to embed signatures into model weights (parameters) rather than the output. While crucial for preventing model theft, these techniques do not solve the challenge of tracing the provenance of the generated content in downstream applications. 

Several studies have also explored passive analysis techniques under the umbrella of authorship attribution and stylometry. Huang et al. [33] examine different approaches for distinguishing between human-written and AI-generated text. Their analysis highlights how this task becomes harder as modern LLMs produce more fluent output. Related survey work on stylometry, such as that by Neal et al. [34], and more recent studies focused on AI-generated text by Fröhling et al. [35], list a wide range of features. These include measures like lexical richness and properties of syntactic parse trees. While informative, these reviews are almost entirely centered on natural language. 

They largely overlook the stylistic characteristics that are specific to source code. In addition, they do not study how common code transformations, such as minification or other forms of obfuscation, affect detection performance. As a result, their findings are difficult to apply directly to software artifacts.

##### **4. Methodology** 

The objective of this review is to classify and evaluate different mechanisms that are used to trace the provenance of AI-generated source code. Unlike earlier surveys that primarily focused on natural language text, this study concentrates on **code provenance** . To guide the review in a structured way, we formulate six Research Questions (RQs) that cover taxonomy, usefulness, and security-related aspects. 

##### _4.1. Research questions_ 

##### . **RQ1 (Provenance Mechanisms): What mechanisms are currently employed to trace the provenance of AI-generated source code?** 

This question focuses on identifying how existing work approaches the provenance problem. We divide prior techniques into **active methods** , where signals are embedded during code generation, and **passive methods** , where provenance is inferred after the code is produced. By making this distinction, we intend to clarify the overall landscape and understand the underlying mechanisms that are used in practice to trace AI-generated code. 

##### . **RQ2 (Granularity): At what structural level of the source code do these provenance techniques operate?** 

Here, we examine the level of code structure at which each technique operates. This classification will help us to explain how deeply each approach is integrated into the code and what kind of information it can realistically capture. 

##### . **RQ3 (Objectives): What are the primary objectives driving the adoption of code provenance techniques?** 

This question investigates the motivating factors behind adopting different code provenance techniques. We analyze the intended goals behind each approach and link them to their application domains. 

##### . **RQ4 (Performance Capabilities): What are the performance capabilities of existing techniques regarding information capacity and detection accuracy?** 

In this question, we focus on how well current techniques perform. For watermarking approaches, this includes evaluating properties such as **bit-rate or information capacity** . For authorship attribution methods, we examine metrics like **classification accuracy and F1score** . Together, these measures provide insight into how reliable each technique is in identifying code provenance. 

##### . **RQ5 (Utility and Impact): How do these techniques impact the functional correctness and natural readability of the generated code?** 

This question looks at the practical cost of applying provenance techniques. We assess whether active injection methods affect code execution or reduce readability, which could limit their real-world usage. For passive approaches, we verify that provenance analysis does not interfere with code behavior. Overall, this RQ helps us to evaluate whether provenance can be added without sacrificing the code quality. 

##### . **RQ6 (Robustness): How robust are these methods against codespecific transformations and adversarial attacks?** 

The final question addresses security and reliability. We examine whether provenance signals remain detectable after common software 

engineering operations, such as refactoring or compression. We also consider more adversarial scenarios, including obfuscation and targeted attacks, to understand how well existing methods hold up under realistic and hostile conditions. 

##### _4.2. Search strategy_ 

The quality of a Systematic Literature Review depends largely on how clear and repeatable the search process is. In this section, we explain how we searched for prior work related to AI-generated source code provenance. We describe how the search terms were defined and how the data sources were selected. The goal was to collect a broad but relevant set of primary studies that fit the scope of this review. 

##### _4.2.1. PICO(C) framework & keyword formulation_ 

To define the search terms, we used the PICO(C) framework proposed by Kitchenham and Charters [36]. We followed this structure to build the search string in a systematic way. Unlike existing studies, we placed strong emphasis on the **Context (C)** , restricting it to the Software Engineering domain. This step was necessary to exclude studies that focus only on natural language processing and are not related to source code. 

- **Population (P):** In Software Engineering, the population usually refers to the technology or system being studied [36]. For this study, the population comprises **Generative AI models** like Codex, StarCoder, and ChatGPT, specifically capable of generating source code. 

- **Intervention (I):** The intervention refers to the specific methodology or tool being investigated [37]. Here, the intervention is **watermarking** and **authorship attribution** techniques used to embed identifying signals into AI-generated artifacts. 

- **Comparison (C):** In this study, we compare different watermarking and authorship attribution approaches against each other or baselines. No specific comparison intervention is enforced in the search string to ensure broad coverage. 

- **Outcomes (O):** The outcomes of interest include metrics related to **robustness** , **functional correctness** (executability), and **imperceptibility** . However, to avoid restricting the search results excessively, outcome-related terms were not included in the search string, following the recommendation by Brereton [38]. 

- **Context (C):** To ensure the relevance to the **Software Engineering** domain, we explicitly restricted the context to **source code** and **software** , filtering out studies focused solely on other domains, such as Text, Image, Audio, or Video. 

Based on these criteria, we identified 3 distinct sets of keywords (synonyms and alternative spellings) to construct the final Boolean search string: 

- **Set 1:** Terms related to the generative models. _Keywords:_ ‘‘Large Language Model’’, ‘‘LLM’’, ‘‘Generative AI’’, ‘‘Decoder-only Models’’ 

- **Set 2:** Terms related to the identification technique. _Keywords:_ ‘‘Watermark*’’, ‘‘Fingerprint*’’, ‘‘Provenance’’, ‘‘Attribution’’ 

- **Set 3:** Terms related to the software engineering domain. _Keywords:_ ‘‘Source Code’’, ‘‘Software’’, ‘‘Artifact*’’, ‘‘AI-generated Code’’, ‘‘LLM-generated Code’’

To make sure the review was as complete as possible, we applied **backward snowball sampling** following the guidelines proposed by Wohlin [44]. This step was used after the main screening phase. ##### _4.4. Quality assessment_ 

After finalizing the set of studies, we assessed their quality and methodological soundness. For this purpose, we designed a **Quality Assessment (QA)** checklist that fits the combined context of Software Engineering and AI. The checklist is based on existing guidelines by Dybå and Dingsøyr [46], but it was adjusted to better reflect issues specific to code provenance research. The detailed QA checklist is shown in Table 3. 

Each paper was scored using a three-point scale: Yes = 1, Partly = 0.5, No = 0. This scoring approach helps weigh the overall contribution of each study. Papers that treated source code only as plain text, without checking functional correctness or execution, were penalized. To keep the review aligned with its goals, each QA criterion was mapped to one or more Research Questions. 

##### _4.4.1. Quality filtering process_ 

The total Quality Score, ranging from 0 to 5, serves as a filtering mechanism to classify the reliability of the evidence. Based on this score, we grouped the papers into three quality tiers: 

- **High Quality (Score** ≥ **4.0):** These studies represent the state-ofthe-art, rigorously verifying executability (QA2) and robustness (QA5). They serve as the primary evidence for **RQ4** , **RQ5** , and **RQ6** . 

- **Medium Quality (Score 2.5 – 3.5):** These studies often propose novel algorithms (strong on RQ1) but fail to provide sufficient evidence regarding their reliability or robustness against standard semantic transformations. To maintain the validity of our conclusions regarding utility and security, these studies were excluded. 

- **Low Quality (Score** _<_ **2.5):** These studies typically treat source code as plain text or do not compare against human-written baselines. As a result, these papers were also excluded from the review. 

##### _4.4.2. Quality assessment score analysis_ 

Quality assessment of the **49** candidate studies reveals a clear trend across evaluation criteria. Most studies perform well on usabilityrelated aspects but are weaker on security-focused checks. **86%** of the papers evaluated Functional Correctness (QA2), and **90%** reported minimal impact on readability (QA4), indicating that most techniques preserve code usability. In contrast, validation approaches are inconsistent among the studies. Only **62%** of the studies used standard, reproducible datasets (QA1) or reported detailed False Positive Rates (QA3). In addition, just **58%** evaluated robustness against semantic-preserving or adversarial transformations, which limits practical reliability. 

Based on overall quality scores, **34** studies were identified as high quality. Three pre-2020 papers were then excluded, leaving a final set of **31 primary studies** for the remainder of the analysis (see Table 4).

##### _5.2. RQ1: What mechanisms are currently employed to trace the provenance of AI-generated source code?_ 

To address this question, we analyzed the provenance approaches proposed across the primary studies. For clarity and comparison, we grouped these mechanisms into **13** categories, as listed in Table 7. At a broader level, these categories fall into two distinct provenance paradigms: **Passive Provenance** and **Active Provenance** . This distinction is important because it separates methods that infer provenance after code generation from those that embed provenance signals during generation. The following subsections describe each of the categories under the aforementioned paradigms in detail: 

##### _5.2.1. Passive provenance approaches_ 

Passive provenance methods aim to infer the origin of a code artifact without changing the generation process. Instead, they analyze traces 

that are assumed to appear naturally in the generated code, such as stylistic or statistical patterns. In our taxonomy, categories **C01** to **C07** belong to this paradigm. 

The first category is based on **training-data provenance via membership inference (C01)** . These methods examine model confidence or loss signals to decide whether a code sample was part of the training set. While effective in controlled settings, they often require white-box or semi-white-box access and are sensitive to model updates, which limits real-world use. 

A large portion of passive work focuses on **stylometric attribution** . Traditional **feature-engineered methods (C04)** use hand-crafted features such as naming patterns, formatting choices, or AST n-grams with standard classifiers. More recent work adopts **representation learning (C02, C05)** , using fine-tuned encoders or contrastive learning to learn attribution features automatically. These approaches are usually more robust to simple transformations but are harder to interpret and often sensitive to domain shifts. 

Another emerging direction uses **LLMs as attribution judges (C03)** through prompting or in-context learning. These methods are easy to deploy but raise concerns about reproducibility and bias, especially when the judge model is closely related to the generator.

##### _5.2.2. Active provenance approaches_ 

Active provenance approaches **embed identifiable signals during code generation** , which allows the origin of the code to be verified later. In our taxonomy, categories **C07** to **C13** fall under this paradigm and are commonly referred to as **code watermarking** . 

The most common active methods are **token-level statistical watermarks (C07)** . These approaches bias token selection during generation using techniques such as green–red token sets, entropy thresholds, 

or multi-bit encoding. Studies like P24, P25, P26 add grammar- or AST-based constraints to reduce syntactical errors. While these methods offer strong detectability guarantees, they often involve trade-offs between watermark strength, code quality, and naturalness. 

**Syntax-aware watermarking (C08)** builds on this idea by avoiding syntax-critical tokens. This improves compilability and stealth, but requires language-specific parsers and adds system complexity. 

**Transformation-based watermarking (C09)** embeds provenance by applying semantic-preserving refactoring or structural rewrites during code generation. These methods are comparatively robust to reformatting and paraphrasing tasks. However, many valid transformations are limited in number, and some overlap with the optimizations already performed by compilers or IDEs. 

There are some lightweight approaches, including **identifier-based watermarking (C10)** and **synonym-substitution methods (C11)** . These preserve readability but are easily broken by automated renaming or normalization tools. 

Finally, **learned watermarking with cryptographic verification (C12)** aims to provide strong ownership guarantees using cryptographic techniques, but remains complex and lightly evaluated. **Trigger or backdoor-style watermarking (C13)** embeds signals through specific triggers or prompts, making the watermark hard to observe directly. These methods are stealthy but sensitive to prompt changes and

##### _5.2.3. Comparative perspective_ 

The literature shows a clear trade-off between ease of deployment and reliability. Passive provenance methods are widely used because they are easy to apply, but they tend to be fragile under transformation or attack. Active methods offer stronger and more reliable attribution, but they face practical issues related to standardization and integration into existing toolchains. 

This gap points towards the need for **hybrid provenance approaches** that combine passive detection with active watermarking. Such combinations may offer more practical and trustworthy traceability of AI-generated code, which motivates several of the open challenges discussed in Section 6. 

##### _5.3. RQ2: At what structural level of the source code do these techniques operate?_ 

To answer RQ2, we examined the **structural granularity** at which provenance approaches operate, as reported in the primary studies. Structural level refers to the abstraction layer of source code where provenance signals are either inferred or embedded. Based on the analysis of the extracted data, the existing approaches can be organized into **5** structural levels. Each level exhibits distinct dominance patterns across passive and active provenance paradigms. In the following subsections, we explain each of these levels in detail: 

##### _5.3.1. Level 1: Token level granularity_ 

At the lowest level, token-based approaches work directly on lexical tokens or their probability distributions. This level is most common in **active provenance** , especially in statistical watermarking that biases token selection during generation. Many studies modify logits or sampling behavior to embed provenance signals in a language-agnostic way. Papers P5, P8, P9, P11, P16, P24, and P25 show that token-level watermarking is highly detectable under clean conditions. Token-level signals are also used in **passive detection** , primarily through perplexity or perturbation-based methods, as seen in P19 and P23. These approaches measure how prediction scores change when the code is slightly perturbed. 

##### _5.3.2. Level 2: Syntax-aware/AST level granularity_ 

At a higher level, syntax-aware or AST-based techniques work on the syntactical pattern of programs rather than raw tokens. This level is important for both passive and active provenance. Many featurebased stylometric methods rely on AST features such as node sequences, subtrees, or syntactic paths. Papers P10, P18, P29, P30, and P31 show that these features are more stable than token-level signals, since they are less affected by simple formatting changes. For active provenance, syntax-aware watermarking avoids embedding signals in syntactically critical locations. This helps to reduce compilation errors and makes the watermark harder to notice, as shown in P6. Syntactical information is also used in representation-learning approaches. Papers P13 and P21 use encoders that capture syntactic structure, which improves attribution performance. Overall, the AST level offers a practical balance. It is more robust than token-level methods, while still being feasible to apply across different settings.

##### _5.3.3. Level 3: Identifier and lexical-semantic granularity_ 

Identifier-level approaches work on variable names, function names, and similar lexical elements that do not affect program behavior. These methods appear less often in the literature, but they form a distinct structural category. P20 shows how provenance can be embedded through context-aware identifier renaming, while still keeping the code natural. P28 follows a similar idea using synonym substitutions on semantically equivalent constructs. These approaches are lightweight and easy for humans to read. However, they are also fragile to common refactoring tools, naming conventions, or normalization steps that can quickly remove or overwrite identifier-level signals. Therefore, such methods offer limited robustness and are usually used as supporting techniques rather than standalone solutions. 

##### _5.3.4. Level 4: Representation/embedding-level granularity_ 

Representation-level approaches operate on latent embeddings learned by neural models and are exclusively associated with passive provenance. These methods infer provenance by comparing learned representations of code samples, rather than analyzing explicit syntactic or lexical features. P2, P3, and P27 employ fine-tuned encoders and classifiers to capture authorship or generation patterns. P21 and P22 use contrastive learning and benchmarking frameworks to evaluate detector generalization. P4 further explores large language models as judges through prompting-based attribution. Representation-level techniques demonstrate strong empirical performance and cross-task generalization, making them dominant within passive provenance research. However, they suffer from limited interpretability and weak localization of provenance signals, as embeddings do not provide clear explanations at the code level. Their effectiveness is also tightly coupled to training data and model architecture. 

##### _5.3.5. Level 5: Semantic/behavioral granularity_ 

At the highest level, semantic approaches embed or detect provenance through meaning-preserving transformations or behavioral equivalence. This level is almost entirely tied to active provenance and is designed to remain robust against rewriting and paraphrasing. Papers P7, P12, and P26 use refactoring patterns or idempotent rewrites so that the watermark survives common code edits. P15 builds on this idea by combining learned watermarking with cryptographic verification to provide stronger guarantees. These methods show the highest robustness among all structural levels. At the same time, they are the most complex to implement. They often depend on language-specific tools, formal reasoning, or detailed program analysis, which makes them harder to scale and deploy.

##### _5.4. RQ3: What are the primary objectives driving the adoption of code provenance techniques?_ 

This research question examines the motivations and practical use cases behind code provenance techniques. ##### _5.4.1. Copyright protection and model ownership verification_ 

The most common motivation behind **active provenance** methods is copyright protection and model ownership verification. Studies in this category aim to show that a given piece of code was produced by a specific AI model. This is mainly used to support intellectual property claims or legal disputes. Many works rely on token-level statistical watermarking, as seen in papers P5, P8, P9, P11, P16, P24, and P25. These approaches embed a detectable signature into generated code that can later be verified with high confidence. Some studies go further by using transformation-based or semantic watermarking. A second key objective relates to academic integrity, especially the detection of AI-generated code in educational settings. This objective mainly relies on **passive provenance** methods. These methods do not change the code generation process and can be applied after submission, which fits typical classroom scenarios. Most studies in this group use stylometric and feature-based techniques. Papers P10, P18, P29, P30, and P31 focus on coding style, syntax, and structural patterns to separate human-written code from AI-generated code. Some works, such as P2, P3, and P27, use representation-learning-based detectors and report high accuracy on benchmark datasets. These approaches operate directly on the generated code. ##### _5.4.3. Digital forensics and post-hoc attribution_ 

Digital forensics aims to identify the source of code after it has been generated and possibly modified, often without cooperation from the model provider. Papers P1, P4, P13, and P21 address this goal using approaches such as membership inference, LLM-as-judge prompting, and representation-level attribution. **Passive provenance** methods are again dominant, mainly because they work in adversarial or uncontrolled settings. However, the literature reports clear limitations, including higher false positive rates and sensitivity to changes in underlying models, which reduce reliability in real-world forensic use.

##### **6. Discussions** 

While the preceding analysis answers the research questions at a technical level, a purely algorithmic view does not fully capture how code provenance techniques interact with real software development practices. This section synthesizes our findings from a software engineering perspective, focusing on where academic assumptions, such as white-box access or static code settings, conflict with practical requirements, including CI/CD integration, long-term maintainability etc. Rather than cataloging prior work, we highlight the key tradeoffs and open challenges that must be resolved for code provenance methods to become viable in industrial software systems. 

##### _6.1. Redefining robustness: The conflict between provenance and maintenance workflows_ 

A key observation from the reviewed studies is a fundamental misalignment in how ‘‘robustness’’ is defined versus how software is actually maintained. Many watermarking approaches treat code transformations or refactoring as adversarial attacks that tend to remove provenance signals. However, in practice, developers regularly apply refactoring, linting, and formatting to maintain code quality and readability. As a result, robustness against such transformations should be treated as a basic requirement rather than a defense against malicious actors. The following discussion illustrates how watermarking at different granularities holds up in practice. 

**Fragility of Token-Level Provenance:** The review suggests that ‘‘token-level’’ watermarking is fundamentally incompatible with standard development workflows. These methods rely on preserving exact, statistically biased lexical sequences. However, standard maintenance tools such as formatters and linters are explicitly designed to normalize syntax, strip non-essential whitespace, and enforce uniform conventions. Consequently, what watermarking literature often frames as an ‘‘adversarial obfuscation attack’’ is frequently just standard code refactoring practice. Because token-level provenance is anchored entirely in surface-level text rather than program logic, it is inherently susceptible to being inadvertently erased by automated quality-assurance pipelines. 

**Resilience of Structural and Semantic Embedding:** Approaches that operate at the AST-level or Semantic-level demonstrate a higher degree of compatibility with standard maintenance workflows. By embedding signals into the logical structure of the code rather than its syntax, these methods persist through formatting normalization. ACW (AI Code Watermarking), proposed in P7, utilizes a set of idempotent, semantic-preserving transformations, such as specific loop structures or Boolean simplifications. As these transformations align with valid optimization rules, the watermark remained detectable (AUROC _>_ 97%) even after the code was processed by optimization tools. Furthermore, recent frameworks have moved towards ‘‘model-driven’’ adaptation to ensure survival through refactoring; for example, CodeTracer (P5) leverages reinforcement learning to identify watermark positions that are syntactically rigid and less likely to be altered during maintenance. 

**Implications for Long-Lived Software Systems:** This dichotomy highlights a ‘‘Generalizability Gap’’ in current research. A watermarking technique that treats refactoring as a malicious attack is non-viable for long-lived code bases where code evolution is inevitable. The literature suggests that for provenance traceability to be adopted in industry, future techniques must decouple the watermark signal from surfacelevel tokens. Instead, they must anchor provenance in the invariant semantic properties of the software, properties that persist whether the code is formatted by a linter, optimized by a compiler, or refactored by a human developer. Without bridging this gap, active provenance injection risks being erased by the very tools designed to maintain code quality.

##### _6.2. Code naturalness and the hidden cost of technical debt_ 

While functional correctness (e.g., Pass@k) is the primary constraint for code watermarking, the literature reveals a secondary, often underprioritized friction: the degradation of code ‘‘naturalness’’. We define **Technical Debt** in this context as the long-term maintenance cost incurred by embedding provenance signals that satisfy a compiler but violate human coding conventions. In this way, code may remain executable yet become less idiomatic, less readable, and harder to refactor for future maintainers. 

**Unidiomatic Flow and Formatting Tool Conflicts:** A significant portion of technical debt emerges as a high Pass@k value can boost the long-term costs associated with maintaining ‘‘unidiomatic’’ code. By the term ‘‘unidiomatic‘‘, we refer to functionally correct code that violates a language’s natural syntactic conventions and standards, analogous to how non-standard code triggers linter warnings [73]. Selective watermarking schemes like CodeTracer (P5) and SWEET (P25) frequently disrupt token statistics, generating such unidiomatic ‘‘code smells’’. This scenario also peaks in structural approaches like ACW (P7), CodeMark-LLM (P12), and SrcMarker (P26), where provenance signals are embedded in transformation or formatting choices that can be systematically erased by standard code formatters like _Black_ , _Prettier_ , or _Ruff_ [74]. Ultimately, the tension between preserving watermarks and writing maintainable code creates an unintended debt that undermines the practical integrity of watermarks in real development scenarios. 

**Semantic Drifting and Maintainability:** One big problem with these trade-offs is ‘‘semantic drifting’’. This happens when a tool swaps one piece of code for a synonym that technically still compiles but changes the subtle behavior of the program. For example, ToSyn (P28) shows that swapping something like np.max for the standard max function in Python can cause these small functional drifts. These changes might not break a simple test, but they can cause failures in edge cases or just confuse developers when they are reviewing the code. Another study, P24, looked at how natural this code looks by measuring ‘‘Perplexity’’. They found that the grammar-based approach, CodeIP (P24), can really distort the code. In their Java benchmarks, the Perplexity score jumped from 2.4 to 7.3. This basically means the generated code starts looking very different from what a human would normally write. 

**Implications for Software Engineering Practice:** This deviation constitutes a form of technical debt. If AI-generated code is ‘‘watermarked’’ by forcing the selection of obscure variable names or unidiomatic syntactic structures, it increases the cognitive load for human reviewers. The watermarking signal, intended to protect intellectual property, inadvertently acts as a pollutant that lowers code readability. Therefore, high Pass@1 scores are insufficient for claiming utility. Future provenance techniques must minimize Technical Debt by optimizing for CodeBLEU alongside execution rates to ensure the resulting code remains maintainable. 

##### _6.3. From snippets to systems: Scalability and integration bottlenecks_ 

Current literature exhibits snippet-level bias, as primary benchmarks like ‘‘HumanEval’’ [1] and ‘‘MBPP’’ [75] focus exclusively on isolated functions. These controlled settings fail to reflect the complexity of real-world software systems, which are built as multi-file repositories with shared dependencies and maintained through CI/CD pipelines. While P10 demonstrates the feasibility of repository-level analysis by evaluating over 76,000 files, such efforts remain a notable exception in the field. Most techniques work only within a single function, as noted in studies such as P12. These studies also highlight module-level and cross-file watermarking as an important research gap. 

**Integration and Latency Constraints:** Apart from granularity issues, inference latency creates another serious barrier to real-world use. To make the provenance techniques compatible with development 

workflows, they need to be integrated and matched seamlessly without having any latency. This is especially important for Git hooks, IDE feedback, and CI pipelines, where delays interrupt normal development. In practice, developers expect near-instant results. However, several studies report much higher processing times. P23 show that perturbation-based perplexity detectors are not suitable for just-in-time detection. Depending on the model, inference can take anywhere from 10 s to more than 150 s for a single sample. Similar concerns appear in P9. Their method, **MCGMark** , introduces significant overhead. On DeepSeek-Coder, detection requires around 100 s per instance. Such delays do not scale with high-throughput development environments and limit the practicality of these approaches. 

**Economic Scalability and Cross-Project Provenance:** The computational and economic costs further compound scalability issues. P12 estimates that training-based watermarking incurs costs of approximately $0.0123 per sample, while their training-free alternative reduces this to $0.0020 per sample. Despite this improvement, reliance on commercial LLM APIs still imposes a high cost when applied to large repositories. In addition, current research largely overlooks **crossproject provenance persistence** . While P29 addresses multi-author attribution within a single file, there is limited evidence that provenance signals survive when code is migrated across files or integrated into new modules. This lack of cross-context robustness limits the applicability of existing techniques for tracing provenance across the full life-cycle of large-scale software systems. 

##### _6.4. Model architectures and provenance trade-offs: From encoders to generative models_ 

An architectural analysis of the reviewed studies reveals that code provenance research spans a wide spectrum of model families, ranging from lightweight encoder-based architectures to large-scale decoderonly generative models. Table 11 summarizes the classification of models employed across the primary studies, organized by architectural type and concrete instantiations. The following discussion interprets these architectural patterns through the lens of scalability and operational sustainability in real-world software systems. 

**Accuracy and Resource Trade-offs in Detection:** Passive provenance approaches commonly adopt encoder-based architectures such as UniXcoder, CodeBERT, and CodeT5 due to their relatively low inference cost and suitability for high-throughput software engineering environments. Frameworks like CodeGPTSensor (P21) use the encoder-only mode of UniXcoder with contrastive learning to achieve effective attribution, while CodeGuard (P17) leverages CodeT5 to compute attention scores for identifying watermark embedding positions. These models are attractive for integration into CI/CD pipelines and large-scale audits. 

This efficiency, however, comes with a clear limitation. Encoderbased detectors often struggle to capture deeper semantic patterns. Models such as CodeT5+ and GPTSniffer can reach high accuracy in controlled experiments. But their performance drops when they are tested on code generated by unseen language models. This suggests limited generalization beyond the training setup. 

On the other hand, large generative models behave differently. Models like GPT-4 or Llama-3 can capture richer semantic information. They are better at understanding structure and intent in code. The problem is cost. Inference latency is high, and the economic overhead is significant. This makes them unsuitable for continuous scanning of large repositories or frequent use in development pipelines. 

To address this trade-off, some studies explore alternative model designs. For example, P20 uses graph neural networks (GNN) to model variable-level context. Other works rely on Siamese architectures [76] with recurrent components to support multi-author attribution. These approaches try to preserve structural awareness while avoiding the high cost of billion-parameter models.

**Stochasticity and Determinism in Active Watermarking:** Decoder-only language models introduce another layer of difficulty for active watermarking. These models generate code in a stochastic way. As a result, the same prompt can produce different outputs across runs. This behavior creates a problem for provenance verification. Watermark detection works best when signals are stable and reproducible. However, non-deterministic sampling in models such as GPT-4 or DeepSeek can change token sequences in subtle ways. These variations may not affect code correctness, but they can weaken watermark extraction. CodeMark-LLM (P12) reports similar observations. Their study shows that probabilistic generation can lead to small formatting changes, which in turn reduce detection reliability. 

Some approaches try to control this instability during generation. CodeIP (P24) introduces an LSTM-based type predictor to guide token selection. This method is applied to models such as Code Llama 

and StarCoder. Adding structural constraints, it reduces syntactic errors while keeping the watermark signal intact. In contrast, fully unconstrained statistical methods behave less reliably. Standard N-gram watermarking is highly sensitive to temperature settings and also vulnerable to semantic-preserving refactoring.

##### **7. Future research directions** 

Based on the limitations and drawbacks identified in the surveyed studies, the following future research directions are proposed to advance the field of code provenance and watermarking: 

- **Scaling from Function-Level to Repository:** Most of the current studies evaluate watermarking on small code units, which are either single functions or short snippets. Benchmarks such as HumanEval and MBPP contain a collection of such code examples. While useful, this setting does not reflect real software development. Practical projects consist of many files and shared modules. Code is often moved, split, or merged over time. In many cases, both human developers and AI tools contribute to the same file. Future work should focus on watermarking at the repository level. This would include tracing the provenance of multiple files of a project that are being written or modified in a collaborative way. The goal should be to preserve provenance even when code crosses file boundaries or is modified collaboratively. 

- **Watermarking Under Black-Box Constraints:** Many watermarking techniques assume access to model internals or the decoding process. However, in practice, most of the developers prefer using 

- API-only or inference-only models like Gemini, ChatGPT, Claude, etc. [77]. This creates a clear gap between the current studies and the real-world deployment setup. Therefore, we should emphasize more on black-box watermarking approaches. These approaches enable code auditing or authorship tracing without requiring access to the internal model. 

- **Reducing Cost for Practical Deployment:** Existing watermarking methods often suffer from high latency and cost, making them impractical for real-world deployment. Large code bases or projects introduce noticeable delay during inference. This makes them difficult to use in real development pipelines. Future research should aim for simpler solutions. Leveraging code stylistic patterns, proper code chunking, and designing efficient prompt strategies may offer a good balance. The focus should be on speed and reliability rather than model or project size. 

- **Towards Language-Independent Techniques:** Many current tools rely on language-specific heuristics (e.g., Python-specific variable renaming). These ideas do not transfer well to other languages. Maintaining separate rules for each language is costly as well. Therefore, future frameworks should aim for languageagnostic architectures, such as utilizing intermediate representations or multilingual pre-trained encoders. These designs would make watermarking suitable for different programming ecosystems. 

- **Universal Attack Benchmarks:** Current studies lack a standardized ‘‘Robustness Score’’. Each study uses its own attack approaches. This makes comparison both difficult and biased. Future work should define a unified attack suite covering both benign and adversarial transformations. That will allow a headto-head comparison of new provenance techniques. 

- **Tracing the Development Process, Not Just Code:** Sometimes source code alone is not enough to prove the provenance of generated artifacts. This is especially true in education and professional settings. The way code is produced, besides the author’s identity, also matters. Commit history, IDE activity, build logs, etc., carry useful signals of the code generation process. These traces can reveal how code evolved. Thus, future research should explore process-based provenance, facilitating stronger and more reliable attribution than artifact-only analysis.

The use of large language models for code generation is increasing rapidly, which raises new questions about authorship, responsibility, and ownership. This review analyzed **31** high-quality studies published
