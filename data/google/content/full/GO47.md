An empirical comparison of AI assisted software refactoring tools

## Abstract

Recent advancements in Artificial Intelligence (AI) have led to the development of powerful assistant tools for coding, such as ChatGPT, Gemini, and Codeium. Existing research has explored the application of ChatGPT in different areas of software engineering, including code summarization, new code generation, and error detection, whereas its potential in code refactoring has been sparsely examined. Investigating the ability of ChatGPT to improve code refactoring could streamline the processes and increase productivity of developers. The literature also reveals a significant gap regarding specific tools like Codeium and Gemini for code refactoring assistance. This study has conducted a detailed evaluation of ChatGPT, Gemini, and Codeium from a code refactoring point of view while focusing on maintaining functionality and ensuring correctness. To generate the reports for ChatGPT, Gemini, and Codeium, both original and refactored code, the tool SonarQube is utilized. The three AI models exhibited distinct performance trends across datasets. For smaller code dataset, Gemini and Codeium attain higher average scores such as 82% and 83%, respectively as compared to ChatGPT having 59%, while for larger codebases dataset, ChatGPT improved substantially to 77.2%, surpassing Codeium in several refactoring attributes. The results showed that these AI-supported tools still possess certain limitations, but Gemini stands as so far best tool for meeting the most quality attributes of refactoring, ChatGPT & Codeium outcomes vary from short to lengthy code. This study helps software developers to use suitable AI-supported code refactoring tools as per their requirements. It also helps policy makers to design policy guidelines to improve software code refactoring through AI-supported tools.

Specifically, the study aims to:

- Assess the accuracy of the refactoring suggestions provided by ChatGPT, Gemini, and Codeium by comparing the refactored code with industry-standard coding practices and guidelines.

- Identify the specific strengths and limitations of each tool in handling different types of refactoring tasks, such as performance optimization, code modularization, and code improvements.

The findings from this study may offer valuable insights for developers, researchers, and other individuals involved in coding-related tasks contributing to the discourse on the integration of the trendy technologies of AI in software engineering practices. ## Methodology

The methodology that has been used for the implementation of this research is adapted from a study conducted regarding the code refactoring capabilities of ChatGPT[16](https://www.nature.com/articles/s41598-026-49590-0#ref-CR16 "DePalma, K., Miminoshvili, I., Henselder, C., Moss, K. & AlOmar, E. (2024).                    https://doi.org/10.1016/j.eswa.2024.123602                                    ") as shown in Fig. [1](https://www.nature.com/articles/s41598-026-49590-0#Fig1). It particularly integrates the strengths of LLMs and static code analysis tools to the process of refactoring code. By leveraging these tools, the approach optimizes existing codebases for enhanced efficiency, readability, and maintainability that are considered crucial aspects of modern software development. This methodology is structured in such a way to evaluate the capabilities in terms of code refactoring of three prominent tools that include ChatGPT, Gemini, and Codeium. These tools offer diverse perspectives on improving the existing code. Following is the step-by-step procedure of our followed methodology:

1. A.

Prompts are basically instructions that have been given to an LLM to enforce rules, automation of processes and ensure specific qualities or quantities of the result to be generated. Prompts are also a kind of programming that can customize the results and interactions with an LLM[18](https://www.nature.com/articles/s41598-026-49590-0#ref-CR18 "Giray, L. Prompt Engineering with ChatGPT: A Guide for Academic Writers, Ann. Biomed. Eng., vol. 51, no. 12, pp. 2629–2633, (2023).                    https://doi.org/10.1007/s10439-023-03272-4                                    "). Although various types of prompts have been explored in prior work, this research employs task-specific prompts, which are tailored to the precise requirements of each coding task and have demonstrated significant improvements in performance. For example, GPT-4 with task-specific prompts performed well as fine-tuned models in code generation, as mentioned in[13](https://www.nature.com/articles/s41598-026-49590-0#ref-CR13 "Mohammed Salih, H. 1. C.

Code quality check is done in this stage to highlight which refactored code produces best quality results in terms of refactoring attributes. 1. D.
Code Refactoring Evaluation Criteria.


In this step, each refactoring solution is evaluated on eight quality attributes of code refactoring, including:

1. i
Reusability


This shows that either the refactored code contains classes, functions, and modules that can be used across multiple projects or modules or not. Reusability in the refactored code is ensured by breaking down of code into smaller, modular, loosely coupled components, extracting functions and adhering to DRY (Don’t Repeat Yourself) principles[21](https://www.nature.com/articles/s41598-026-49590-0#ref-CR21 "Zhang, Q. et al. An Example from ChatGPT and Automated Program Repair. ArXiv 2310 (08879), 1–12 (2024)."). In SonarQube this metric is calculated by the following equation:

$$Reusability{\\text{ }}\\left( f \\right){\\text{ }} = Number~of~Calls~to~f{\\text{ }}/{\\text{ }}Number~of~Lines~in~f$$

Here:

- Number of Calls to f: Total times the function is invoked.

- Number of Lines in f: The total number of lines in the function. This could be further calculated as End Line−Start Line.


A higher value indicates that a function is frequently reused while maintaining compact implementation which actually reflects better reusability.

1. ii
Coupling


This identifies that either the refactored code contains inter-dependency between various modules ensuring that the modifications in one part of the system would result in minimal and not major impact on others. Coupling in the refactored code is ensured by low coupling is preferred, where within the program, various modules, methods, functions or classes should have minimal knowledge of each other and dependency on each other’s[4](https://www.nature.com/articles/s41598-026-49590-0#ref-CR4 "Fowler, M. Refactoring: Improving the design of existing code (Addison-Wesley Professionals, 2018)."). The coupling in SonarQube score is calculated as:

$$Coupling{\\text{ }} = {\\text{ }}Number~of~Lines{\\text{ }}/{\\text{ }}Number~of~Calls$$

A lower value indicates fewer dependencies per unit of code structure helps in suggesting lower coupling and better modularity.

1. iii.
Maintainability


Maintainability highlights the degree of ease in which the code can be managed, modified to correct errors, updated, enhance performance, or flexible to the changing requirements. Refactoring enhances maintainability by organizing code into manageable and well-structured components. This involves reduction of duplication, modularization functionality, and adhering to the Don’t Repeat Yourself (DRY) principle and Single Responsibility principles (SRP)[4](https://www.nature.com/articles/s41598-026-49590-0#ref-CR4 "Fowler, M. Refactoring: Improving the design of existing code (Addison-Wesley Professionals, 2018)."). We calculated maintainability via SonarQube. The maintainability rating in SonarQube is derived from the Technical Debt Ratio, which basically compares the cost to fix issues (technical debt) against the cost to develop the software. Here, a lower ratio indicates better maintainability. It can be calculated as follows:

$$Technical~Debt~Ratio = {\\text{ }}\\left( {Remediation~Cost{\\text{ }}/{\\text{ }}Development~Cost} \\right){\\text{ }} \\times 100$$

The Maintainability Rating is then assigned based on the Technical Debt Ratio:

$${\\text{A }}\\left( { < = {\\text{5}}\\% } \\right),{\\text{ B }}\\left( {{\\text{6}} - {\\text{1}}0\\% } \\right),{\\text{ C }}\\left( {{\\text{11}} - {\\text{2}}0\\% } \\right),{\\text{ D }}\\left( {{\\text{21}} - {\\text{5}}0\\% } \\right),{\\text{ E }}\\left( { > {\\text{5}}0\\% } \\right)$$

1. iv.
    Performance


Performance is another important attribute of code refactoring in which enhancement of the efficiency of the code can be achieved in terms of speed, utilization of memory, or other system resources but without changing its actual functionality. This study considers performance as the refactored code by eliminating redundant code, simplifying loops and modules, and reducing computational complexity or removing unnecessary computations, and catching results. In this regard, the complexity of the code in terms of time as well as space is calculated and shown. Here time complexity highlights how much time refactored program is taking for execution whereas, space complexity highlights how much space the refactored program is taking to store[4](https://www.nature.com/articles/s41598-026-49590-0#ref-CR4 "Fowler, M. Refactoring: Improving the design of existing code (Addison-Wesley Professionals, 2018)."). Space complexity and time complexity are combined here to evaluate the overall performance of an algorithm or program. It is generally assessed based upon how efficiently an algorithm uses both time and space. Here all other attributes are calculated directly while performance is calculated by the combination of space complexity and time complexity in the following manner by the given code of the data set A:

Performance = Time Complexity + Space Complexity.

$$T(n) = \\left\\{ {\\begin{array}{\*{20}l} {O(n^{{{\\text{max}}\\\_{\\text{loop}}\\\_{\\text{nesting}}}} \\times 2^{n} ),} \\hfill & {{\\text{if recursion exists}}} \\hfill \\\ {O(n^{{{\\text{max}}\\\_{\\text{loop}}\\\_{\\text{nesting}}}} ),} \\hfill & {{\\text{if loops exist}}} \\hfill \\\ {O(1),} \\hfill & {{\\text{otherwise}}} \\hfill \\\ \\end{array} } \\right.$$

$$S(n) = \\left\\{ {\\begin{array}{\*{20}l} {O(2^{n} ),} \\hfill & {{\\text{if recursion exists}}} \\hfill \\\ {O(n^{{{\\text{max}}\\\_{\\text{loop}}\\\_{\\text{nesting}}}} \\times n),} \\hfill & {{\\text{if function space is used}}} \\hfill \\\ {O(1),} \\hfill & {{\\text{otherwise}}} \\hfill \\\ \\end{array} } \\right.$$

To enable quantitative comparison across tools, asymptotic complexity classes are mapped to normalized percentage scores using a pre-defined scale as follows:

$${\\text{O}}\\left( {\\text{1}} \\right){\\text{ }} = {\\text{ 1}}00\\% ,~{\\text{O}}\\left( {{\\text{log n}}} \\right){\\text{ }} = {\\text{ 9}}0\\% ,{\\text{ O}}\\left( {\\text{n}} \\right){\\text{ }} = {\\text{ 8}}0\\% ,{\\text{ O}}\\left( {{\\text{n log n}}} \\right){\\text{ }} = {\\text{ 7}}0\\% ,{\\text{ O}}\\left( {{\\text{n}}} \\right){\\text{ }} = {\\text{ 5}}0\\% ,{\\text{ O}}\\left( {{\\text{2}}{}} \\right){\\text{ }} = {\\text{ 2}}0\\%$$

This mapping is based on standard algorithmic efficiency hierarchies, where lower complexity corresponds to better performance. The normalization enables consistent comparison across various refactored outputs.

    Time Complexity Calculation


We need to find:

- Loops and their nesting level.

- Recursion, if any exists.

- In our refactored code we have:

- No recursion found → First case does not apply.

- Maximum loop nesting = 2 (in filter\_duplicate\_instance()).

- Thus, using the formula: _T(n) = O(n_ _2_ _)_.


1. b)
Space Complexity Calculation


- No recursion exists → First case does not apply.

- Max loop nesting = 2 (nested for loops in filter\_duplicate\_instance()).

- Function space usage includes lists and MinHash structures → O (Nm).


Thus, apply the formula: _S (n) = O (n_ _2_ _)_.

Now Performance Calculation: P (n) = O (N2) + O (N2).

Since both terms are of the same order, the sum remains: _P (n) = O(N_ _2_ _)_ Then to assign percentages to the performance we considered the standard notations table.

$$Performance~Score = \\alpha \\cdot Time~Complexity + \\beta \\cdot Space~Complexity$$

Here:

- Time Complexity (T (n)): Measures how execution time grows with respect to input size.

- Space Complexity (S (n)): Measures how memory usage grows with respect to input size.

- α and β: Weighting factors based on the importance of time or space for a specific application. Both possess equal weight = 0.5.

1. vi.
Security


Security in code refactoring highlights the process of enhancement of the code’s ability to resist vulnerabilities while maintaining its functionality. It helps by eliminating insecure coding patterns, enforcing access controls, and improving data validation. A well-refactored code here follows secure practices of coding, such as using parameterized queries, validating user inputs and avoiding provision of hard-coded credentials[4](https://www.nature.com/articles/s41598-026-49590-0#ref-CR4 "Fowler, M. Refactoring: Improving the design of existing code (Addison-Wesley Professionals, 2018)."). SonarQube detects potential security vulnerabilities based on predefined rules and patterns. It relies on static code analysis to identify issues like SQL injection or cross-site scripting vulnerabilities. The rules in SonarQube are categorized primarily into two types:

1. 1.
_(1)_ Security-Injection Rules: These rules focus on identifying vulnerabilities arising from invalidated user inputs that can lead to injection attacks. Common vulnerabilities detected include SQL Injection (CWE-89), Cross-site Scripting (XSS) (CWE-79), and Code Injection (CWE-94).

2. 2.
_(2)_ Security-Configuration Rules: These rules address security issues that arise due to improper configurations or using insecure parameters. They basically ensure that the sensitive functions are called by using secure parameters and that necessary security checks are performed correctly. It includes Weak Cryptographic Algorithms and Insecure Protocol Versions.

1. F.
Evaluation Limitations.


1. i.
    Lack of Human Validation


SonarQube, one of the state-of-the-art tools provides consistent and objective metrics for evaluating quality of the code[15](https://www.nature.com/articles/s41598-026-49590-0#ref-CR15 "Deshapriya, L., Licorish, S. A. & Woodford, B. J. Understanding warnings generated by PMD and SonarQube, their rules and compliance to established coding standards. Sci. Comput. Program. 252, 103470. (2026).                    https://doi.org/10.1016/j.scico.2026.103470                                    "). This study does not incorporate manual expert validation of the refactored code. As a result, our assessment mainly focuses on structural and maintainability improvements rather than direct verification of functional equivalence or developer perceived readability. To mitigate this limitation, basic correctness was ensured by verifying that the refactored code remained free of compilation errors and major static analysis violations. Apart from that, SonarQube metrics such as code smells, duplication, and maintainability index were utilized as standardized indicators of code quality improvement[20](https://www.nature.com/articles/s41598-026-49590-0#ref-CR20 "SonarQube SonarQube, Sonar Source. Accessed: Oct. 04, 2025. [Online].                     https://docs.sonarsource.com/                                    ").

1. ii.
Interpretability of AI Models


Due to black-box nature of LLMs, this study adopts an output-based evaluation approach, mainly focusing on measurable improvements in code quality using SonarQube metrics. While this enables objective and consistent comparison across models, it does not provide direct insight into the internal decision-making processes of the AI tools. Large language models such as ChatGPT, Gemini, and Codeium operate as black-box systems in which the underlying reasoning behind specific refactoring choices is not fully observable. To partially address this, a qualitative inspection of the output generated was conducted. It was observed that different models tend to favor distinct refactoring strategies such as, some models emphasized structural simplification while others focus on modularization and readability improvements specifically in larger codebases.

1. iii.
Experimental Scope and Generalizability


Additionally, this study focuses on evaluating AI-based refactoring tools using Python code snippets of relatively small to moderate size. While this enables controlled experimentation and consistent comparison across models, it limits the generalizability of the findings to large-scale, real-world software systems and other programming languages. The selection of smaller code snippets was intentional to ensure clarity in evaluating refactoring changes and to maintain consistency with prior studies in this domain. Moreover, the inclusion of both shorter as well as relatively longer code samples provides a balanced perspective on model performance across varying levels of code complexity.

## Conclusion

In conclusion, this study presents a comparative analysis of ChatGPT, Gemini and Codeium, based on key refactoring attributes such as code simplification, readability, complexity reduction, and functional preservation. The findings provide insights into the relative strengths and limitations of these tools when applied to Python-based code refactoring tasks.

The experimental results indicate that performance varies across datasets and code sizes. For Dataset A, Codeium achieved the highest overall score (83%), whereas Gemini outperformed other tools on Dataset B with a score of 82%. When considering both datasets collectively, Gemini demonstrated the most consistent performance, achieving an average score of 82.54%. Therefore, it can be stated that Gemini generally performs more consistently across varying scenarios although no single tool dominates in all cases as presented in Table [3](https://www.nature.com/articles/s41598-026-49590-0#Tab3).

It was also observed that the effectiveness of these tools may decrease with increasing code size and complexity. In the experimental setup of our research, all tools showed limitations when handling larger lines of code segments such as beyond 500 lines of code, suggesting that current AI-assisted refactoring tools may be more suitable for small to medium-sized codebases rather than full-scale projects.

However, this study has several limitations. First of all the evaluation is restricted to Python programming language only, which may limit the generalizability of the findings to other languages. Secondly, the datasets utilized are limited in size and diversity, which may affect the robustness of the results. Thirdly, the study relies primarily on automated evaluation metrics without incorporating too much human expert assessment of code quality. Additionally, the outcomes are influenced by prompt design and interaction strategies, such as similar prompt of refactoring along with changed code from dataset each time was given to refactor the code.

Future research can extend this work by including multiple programming languages and more diverse datasets in order to enhance generalizability. Incorporating human evaluation alongside automated testing would provide a more comprehensive assessment of code readability and usability. Furthermore, advanced prompt engineering techniques could be used and repeated trials may improve reliability. Another future direction is the integration of AI-assisted refactoring tools into CI/CD pipelines for evaluating their performance in real-world development environments. Finally, future studies should also consider other quality attributes of code refactoring and can also consider ethical, security, and privacy implications associated with the use of AI tools in software development.

Overall, this research contributes to the growing body of knowledge on AI-assisted software engineering and highlights the potential of these tools to support developers in code refactoring tasks, while also emphasizing the need for further improvements and careful integration into development workflows.
